## 一、语言模型已经很通用了，但世界不是只有文字

到目前为止，这门课**只讲了语言模型**。而语言模型其实已经相当通用了 ——
它给你的能力是「从任意一段文本，到任意一段文本」：

$$
\text{文本} \;\Longrightarrow\; \text{文本}
$$

这里的「文本」可以是任何语言、可以是代码、可以是诗，Percy 甚至举了 **DNA 序列**当例子 ——
只要你能把它写成字符串，语言模型就能在上面工作。

**但世界是多模态的。** 除了文字，还有图像、音频、视频。

<Figure src="/courses/cs336/l17-multimodality.png" caption="四种模态：文本、图像、音频、视频。语言模型目前只覆盖左上角那一格。来源：CS336 课件" />

### 北极星：omni 模型

那么理想的终点是什么？人们管它叫 **omni 模型（omni model，全模态模型）**：

<Theorem type="definition" name="omni 模型">

- **输入**任意模态的任意组合（这一侧叫**理解**，understanding）
- **输出**任意模态的任意组合（这一侧叫**生成**，generation）

</Theorem>

Percy 举了几个具体的用法：

- 我给你**一张图和一段视频**，然后问一个同时关于这张图和这段视频的问题。
- 我让你**生成图像**。
- 我让你**把一段音频转换成一张图像**。

有了 omni 模型，你能想象出各种各样的应用。

不过他也说清楚了这一讲的范围：**这门课不会真的搭出一个完整的 omni 模型**，
而是给出其中的一些零件 —— 至少让你能**把图像喂进去**。今天绝大部分篇幅是关于图像的。

## 二、为什么一切都必须先变成 token

那么，这样的东西该怎么建？Percy 说，接下来这个动机听上去有点怪，但现实就是这样：

<Theorem type="proposition" name="从 Transformer 倒推出来的约束">

1. **Transformer 的效果实在太好了。** 尽管很多人努力尝试过别的东西，
   但**跨所有这些模态**、**在大规模上**，Transformer 仍然是我们手上最好的东西。
   所以我们只能想办法用它。
2. **而 Transformer 是为文本设计的，它说的是 token。** 它吃进去一堆 token，吐出来一堆 token。
3. **所以我们必须把一切都转换成 token。**

</Theorem>

这里的「token」要比第 1 讲里的含义更宽一点。Percy 特意做了扩展：

<Callout type="note" title="补充：离散 token 和连续 token">
第 1 讲里的 token 是**离散**的：一个整数下标，配一张词表，模型内部再用嵌入矩阵
把这个下标查成一个向量。

这一讲里的 token 还可以是**连续**的：跳过「整数下标」这一步，
直接给模型一个向量。你可以把它理解成「一个 token 的嵌入，只不过这个嵌入不是从词表查出来的，
而是某个编码器现算出来的」。

**Transformer 本身不在乎这个区别** —— 它的输入本来就是一串向量。
离散 token 只是「先查表再变成向量」，连续 token 是「直接就是向量」。
后面会看到，视觉语言模型（LLaVA、Qwen-VL）走的是连续这条路，
而 Chameleon 走的是离散那条路。
</Callout>

关键在于下面这条性质，它决定了后面所有设计：

<Theorem type="definition" name="token 应该是什么">
**一个 token 应该代表某种「语义单元」的信息。**

- 在自然语言里，token 是**子词（subword）**，这些东西多少是有意义的。
- 而**一个像素本身，肯定不是有意义的**。
</Theorem>

所以，我们必须想办法把音频、图像这些东西，变成**离散或连续的 token**。

### 这正是第 1 讲那个伏笔

注意，这件事**我们对文本已经做过一遍了**。第 1 讲讲的分词就是这个。
Percy 对那次经历的评价是：还行 —— 我们有 BPE 分词器，大家自己实现了一遍，
它不完美，你当然可以希望有个更好的分词器、甚至希望根本不要分词器，
但**它不算什么天大的坏事**。

**换成非文本模态，就得多挠好几下头了**：什么东西才是「图像版的 BPE 分词器」，
能把一张图变成 Transformer 消化得了的东西？

<Callout type="insight" title="回头看第 1 讲留下的两条">
第 1 讲末尾，Percy 说过：即使将来分词器被干掉，**任何替代方案也必须满足两条性质**。

1. **模型必须在序列的某种「块」或抽象上操作，而不是在最原始的单元上。**
   他当时特意补了一句：这一点在文本上还不太明显，
   **换成视频或 DNA 就非常明显 —— 单个字节的信噪比极低。**
2. **这些块必须是可变的**，也就是要有自适应计算 —— 不是所有输入都该被同等对待。

这一讲就是那句话的兑现。第一条对应的是「一个 token 应该是一个语义单元，
而一个像素不是」；第二条对应的是后面会反复出现的**动态分辨率** ——
一张高清文档该切成上万个 token，一张小小的公式条只值 8 个。
</Callout>

### 两个问题

于是整个多模态问题被拆成两半：

1. **怎么把非文本数据「输入」进 Transformer？** 比如理解一张图像。
2. **怎么「输出」非文本数据？** 比如生成一段音频、一张图像。

**这一讲主要回答第一个问题。** 第二个问题会在最后讲 Chameleon 时碰一下，
以及在总结里提一句扩散模型。

## 三、CLIP：用图文对学出图像的语义

要理解今天的视觉语言模型，得先把时钟拨回 **2021 年**。
**CLIP** 是现代视觉语言模型（VLM）的地基之一，所以值得讲细一点。

CLIP = **Contrastive Language-Image Pre-training（对比式语言-图像预训练）**，
出自 OpenAI（[论文](https://arxiv.org/abs/2103.00020)）。

### 3.1 当时的处境

那个时间点上：

- **语言这一侧**：GPT-2、GPT-3 都已经出来了，语言模型已经进入了「基础模型」时代。
- **视觉这一侧**：还停留在传统路线上 —— 依赖**大规模人工标注数据集**（比如 ImageNet），
  在上面训练 **ResNet** 这类模型，再配合数据增强等一堆手段，把性能刷得很好。

于是 OpenAI 的研究者们想的问题是：

> **能不能利用互联网上数量大得多的「图像 + 文字说明（caption）」？**

这个想法直接来自语言那边的经验：你去爬互联网，能拿到海量文本，它非常嘈杂，
但只要模型足够大，你就能从这堆噪声里学出有用的东西。**那么图像的对应物是什么？**

CLIP 就是这个问题的答案。

### 3.2 方法：一批图文对，把对角线推上去

CLIP 的想法相当简单直白。

<Figure src="/courses/cs336/l17-clip.png" wide caption="CLIP 的三张图：(1) 对比式预训练，把 n×n 相似度矩阵的对角线推高；(2) 用类别名造出一批文本；(3) 零样本预测 —— 图像编码后和这些文本比点积，取最高的那个。来源：CLIP 论文" />

具体步骤：

1. **拿一批（图像, 文本）样本。** 课件上的数是 **32768** 个（Percy 课上口头说的是 3 万 2 左右）。
2. **把每张图像编码成一个向量**，得到 $I_1, \ldots, I_n$；
   **把对应的每段文本也编码成一个向量**，得到 $T_1, \ldots, T_n$。
3. **对每张图像，让它偏好自己配对的那段文本，胜过批次里其他所有文本。**
   也就是说，要让 $I_1 \cdot T_1$ 远大于 $I_1 \cdot T_2, I_1 \cdot T_3, \ldots$
4. **反过来对每段文本也一样**：要让 $T_1 \cdot I_1$ 大于 $T_1$ 和其他所有图像的点积。

这就是 CLIP 的目标函数。你可以把它看成 $2n$ 个 softmax 多分类问题：
$n$ 个「给定图像挑文本」，$n$ 个「给定文本挑图像」。

<Figure src="/courses/cs336/l17-clip-code.png" caption="CLIP 论文里的伪代码。来源：CLIP 论文" />

把这段伪代码逐行拆开：

```python
# image_encoder  - ResNet 或 Vision Transformer
# text_encoder   - CBOW 或 文本 Transformer
# I[n, h, w, c]  - 一个 minibatch 的对齐图像：n 张图，每张 h×w 像素、c 个通道
# T[n, l]        - 一个 minibatch 的对齐文本：n 段文本，每段 l 个 token
# W_i[d_i, d_e]  - 把图像特征投影到联合嵌入空间的矩阵
# W_t[d_t, d_e]  - 把文本特征投影到联合嵌入空间的矩阵
# t              - 可学习的温度参数

# 第一步：各自过自己的编码器，抽出特征
I_f = image_encoder(I)   # 形状 [n, d_i]
T_f = text_encoder(T)    # 形状 [n, d_t]

# 第二步：投影到同一个空间，并做 L2 归一化（归一化之后点积就是余弦相似度）
I_e = l2_normalize(np.dot(I_f, W_i), axis=1)   # [n, d_e]
T_e = l2_normalize(np.dot(T_f, W_t), axis=1)   # [n, d_e]

# 第三步：两两算相似度，得到一个 n×n 的矩阵，再乘上温度
#         第 (i, j) 格 = 第 i 张图 和 第 j 段文本 的相似度
logits = np.dot(I_e, T_e.T) * np.exp(t)        # [n, n]

# 第四步：对称的损失。正确答案就是对角线 —— 第 i 行的答案是 i，第 j 列的答案也是 j
labels = np.arange(n)
loss_i = cross_entropy_loss(logits, labels, axis=0)   # 按图找文
loss_t = cross_entropy_loss(logits, labels, axis=1)   # 按文找图
loss   = (loss_i + loss_t) / 2
```

用 Percy 的话说：**这基本上就是一个多分类问题，只不过样本被组织成了一个 n×n 的矩阵形式。**

<Callout type="tip" title="动手：把对角线推上去">
下面这个演示就是上面那张 n×n 矩阵。每个格子上面那行是**相似度**（两个向量的点积），
下面那行是**经过目标函数处理之后的概率**。

几件值得试的事：

- 把「训练进度」从 0 拖到 100%：一开始整张矩阵是一团噪声（编码器还没学到东西，
  图文点积和配不配对没关系），随着训练，**对角线被推亮，其余格子被压暗**。
  这就是「对比学习」四个字的全部含义。
- 把「温度 τ」调小：softmax 变得更尖锐，损失对「排第二的那个负例」更敏感。
- **切到 SigLIP**：同一张矩阵，但每个格子变成一道独立的是非题。注意看下面那行概率的变化 ——
  CLIP 的每一行加起来等于 1（它们互相竞争），SigLIP 的每个格子各管各的。
  这一点等到 3.9 节和第四节再回来看。
- 把批大小 n 从 4 拖到 8：CLIP 模式下每个正例要压过的负例变多了。真实的 CLIP 用的是
  32768，也就是每个正例对着 32767 个负例。

</Callout>

### 3.3 数据从哪来

论文里关于数据的细节不多。大致是：

- **搜集了 50 万条查询词**，对每条查询去网上搜，挖出**约 2 万个（图像, 文本）对**。
- 最终训练用了 **4 亿个图文对**。
- **数据集没有公开。**

后来有人做了复现，就是 **OpenCLIP**（[论文](https://arxiv.org/abs/2212.07143)）。
它用的是公开的 **LAION-5B** 数据集 —— 50 亿张带文字描述的图像。

<Callout type="note" title="补充：这里有个自举（bootstrapping）">
LAION-5B 在构建时，**用 CLIP 本身来做数据过滤** —— 拿 CLIP 判断一张图和它的说明文字
是不是真的匹配，不匹配的扔掉。然后人们再用这个数据集训练 OpenCLIP。

Percy 特意点出了这里的循环：**用 CLIP 筛数据，再用筛出来的数据训 CLIP 的复现版。**
好处是，至少这个模型你能指着说「它就是在这个数据集上、用这份代码训出来的」，
而原始 CLIP 你指不了。
</Callout>

### 3.4 数据预处理：为什么非要裁成正方形

（[CLIP 的处理代码](https://github.com/openai/CLIP/blob/main/clip/clip.py#L79)）

**互联网上的图像什么分辨率都有**：可能又长又窄，也可能又高又瘦，尺寸完全任意。

而 Percy 说，学神经网络时你会学到的一件事是：**神经网络不喜欢动态的东西，它们要固定尺寸。**
所以 CLIP 做了一套相当启发式的处理：

1. **双三次插值（bicubic interpolation）缩放**，让**短边**变成 336 像素
   （目标尺寸也可以是 224）。
2. **中心裁剪（center crop）**，切掉边缘，得到一张 336×336 的正方形图。

这显然是为了方便。Percy 补充说：**后面会看到有比这好得多的做法**，
这里纯粹是图省事；而且在当时，CLIP 的作者们心里想的是 **ImageNet 和图像分类** ——
分类任务里目标物体通常在正中间，你切掉的只是一些背景，影响不大。

### 3.5 视觉编码器：Vision Transformer

CLIP 试过好几种视觉编码器，包括 **ResNet-50** 和刚出不久的
**Vision Transformer（ViT）**（[论文](https://arxiv.org/pdf/2010.11929)），
结果发现 **ViT 效果最好**。所以人们说 CLIP 时，一般指的是 ViT 那个版本。

<Figure src="/courses/cs336/l17-vit.png" wide caption="Vision Transformer：图像切成 patch，展平后过一个线性投影，加上位置嵌入，再送进标准 Transformer 编码器。来源：Dosovitskiy et al., 2020" />

ViT 做的事情是：

1. **把图像切成一个个 patch（图块）。** 原始 ViT 论文用的是 16×16，CLIP 用的是 14×14。
2. 每个 patch 展平成一个向量，过一次线性投影 —— **于是每个 patch 在某种意义上就是这个
   Vision Transformer 的一个 token。**
3. **加上位置嵌入**，跟你训练语言模型时做的一模一样。
4. 然后**直接扔进一个标准的 Transformer**。

<Theorem type="definition" name="把「切 patch」算成具体的数">
拿 CLIP 最好的那个配置 **ViT-L/14@336px** 算一遍：

- 图像是 $336 \times 336$ 像素，patch 是 $14 \times 14$。
- 横着能切 $336 / 14 = 24$ 块，竖着也是 24 块，
  所以一共 $24 \times 24 = 576$ 个 patch。
- 每个 patch 原本是 $14 \times 14 \times 3 = 588$ 个数
  （14 行 × 14 列像素，每个像素 3 个颜色通道 RGB）。
- 这 588 个数过一次线性投影，变成一个 $d$ 维向量 —— 这就是一个 token。

**所以一张 336×336 的图 = 576 个 token 的序列。** 作为对照，
第 1 讲里那句 `Hello, 🌍! 你好!` 只有 8 个 token。
</Theorem>

下面这个演示把上面这套算术做成了可以拖的。默认就是 CLIP 的 336×336 / patch 14。

几件值得试的事：

- 拖动图像的宽和高，看 token 数怎么涨。**token 数是随面积增长的** ——
  边长翻倍，token 数变四倍。
- 把 patch 边长从 14 换到 32：token 数掉到约五分之一，但每个 token 要吞下
  $32\times32\times3 = 3072$ 个数 —— 细节被压得更狠。
- 点「Qwen2-VL 图 2（公式条）」这个预设：一条 224×28 的窄公式条只值 **8 个 token**。
  再点「Qwen2-VL 图 3」：一张 1260×700 的风景照要 **1125 个 token**。
  这两个数就是 6.2 节那张架构图上标的真实数字 —— 到那里再回来看。
- 勾上「2×2 合并」，token 数直接掉到四分之一。这是 Qwen2-VL 压缩上下文的手段，6.2 节会讲。

</Callout>

#### 怎么从一串向量得到一个向量

Transformer 编码器吐出来的是**一串向量**（每个 patch 一个）。
但 CLIP 要的是**一个**向量来代表整张图。怎么办？

- **最简单的办法**：把这些向量**平均**一下，得到一个向量。
- **CLIP 实际用的**：**注意力池化（attention pooling）**，效果更好。

<Callout type="note" title="补充：注意力池化是什么">
Percy 课上是这样描述的：**先取所有激活值的全局平均**，
把这个平均向量当作**查询（query）**，然后**再做一轮注意力** ——
用这个 query 去和每个位置的键（key）、值（value）做注意力。

得到的输出向量比「直接取平均」更有信息量：直接取平均是无差别地一视同仁，
而这一轮注意力让模型可以**给不同位置不同的权重**，
比如更看重前景物体所在的那些 patch。
</Callout>

#### 命名规则

CLIP 最好的模型叫 **ViT-L/14@336px**，把这个名字拆开：

- **L** = large（大号）。Percy 说他记得大概是 24 层左右，但不太确定。
- **14** = 14×14 的 patch。
- 每个 patch 是 **RGB**，所以有 3 个通道。
- **@336px** = 在 336×336 分辨率的图像上训练。

关于最后这条，Percy 提到一个训练上的细节：**他们其实是先在更低分辨率上训**，
**训练的后半段才换成高分辨率** —— 因为高分辨率图像算起来慢，这样能省时间。

<Callout type="note" title="课堂问答：位置嵌入能不能做得更聪明">
有同学问：文本只有一个维度，但图像有两个维度（横和纵），
位置嵌入是不是应该比文本复杂一些？直接编成 0 到 N 的线性下标是不是太粗糙了？

Percy 的回答：**CLIP 论文里确实试过 2D 版本的位置嵌入，但他们发现和 1D 相比差别不大。**

不过他紧接着提醒：**这类结论要放在「他们心里想的是分类任务」这个背景下看。**
对分类来说也许无所谓。**后面会看到更讲究的位置嵌入，它们是真的把空间结构考虑进去的**
—— 指的就是 6.2 节的 MRoPE。
</Callout>

### 3.6 文本编码器

文本编码器就是一个标准的 Transformer，**GPT-2 风格**（因为 CLIP 出自开发 GPT-2 的同一批人），
**6300 万参数、12 层**。

怎么从一串 token 得到一个向量？做法是：

- 在序列前面加一个 `[BOS]`（序列开始），后面加一个 `[EOS]`（序列结束）；
- 然后取**最高层上 `[EOS]` 那个位置的激活值**，作为整段文本的表示。

到这里两个编码器都齐了：挑一批样本 → 把所有文本编码 → 把所有图像编码 →
组成那两个方向的交叉熵损失 → 反向传播。

### 3.7 头条结果：零样本打赢了监督训练的 ResNet

2021 年让很多人为 CLIP 兴奋起来的结果是：

> **在 ImageNet 上，零样本的 CLIP 打败了在 120 万张 ImageNet 图像上训练出来的 ResNet。**

Percy 强调了这个对比的分量：**那 120 万张 ImageNet 图像，是无数小时的
Amazon Mechanical Turk 众包工时标出来的。** 而 CLIP 是在更「有机」的网络数据上训练的 ——
当然那些图文说明也是网上一大批人的劳动成果，但**只要你能把已经存在的数据利用起来，
你就能直接零样本上阵**。

<Theorem type="definition" name="零样本分类具体怎么做">
CLIP 本身不是分类器，它是怎么做分类的？

1. 拿到一张待分类的图像，用图像编码器编成向量 $I_1$。
2. 把每个候选类别名填进一个模板，比如 `A photo of a {object}.`，
   得到 `A photo of a plane.`、`A photo of a car.`、`A photo of a dog.` ……
   把每一句都用文本编码器编成向量 $T_1, T_2, \ldots$
3. **算 $I_1$ 和每个 $T_j$ 的点积，取最高的那个**，它对应的类别就是预测结果。

注意整个过程**不需要任何该数据集的训练样本** —— 这就是「零样本」的意思。
（这就是 3.2 节那张图的第 (2)(3) 部分。）
</Theorem>

<Callout type="note" title="课堂问答：如果很多图片的说明里都有「狗」怎么办">
有同学问：这种自监督的做法有个隐患 —— 假如批次里有一张狗的图，
但别人的说明文字里也提到了狗，那模型不就被搞糊涂了吗？

Percy 的回答分两层：

1. **这个过程本来就是嘈杂的，但平均下来没问题。** 就算这一批里碰巧还有另一只狗，
   也不可能每一批里都有狗 —— 有时是苹果，有时是猫。
2. **更根本的是，这些数据本身就极其嘈杂。** 这些图像是从网上爬的，
   配的是旁边的文字或者 `alt` 属性里的文字。有研究专门看过这件事：
   **图片的说明文字并不会逐字复述图里有什么** ——
   如果一张图是一只狗，你恰恰不会去写「一只狗」。

所以 Percy 说，**这件事居然能成，其实有点出人意料，但也正因如此才有意思**。
当然，前提是**做了大量的数据过滤** —— 如果你直接拿网上任意的图文对来训，
噪声大概率会大到根本训不出东西。
</Callout>

<Callout type="note" title="课堂问答：为什么要用图文对，不能只用图像吗">
有同学问：为什么非得在（图像, 文本）对上训练，不能只用图像？

Percy 说，确实有一条工作线是**只靠数据增强**来学图像表示的，比如 **SimCLR** 的思路：
拿一张图，**裁剪它、随机扰动它、旋转它**，然后要求这些变体的嵌入应该彼此相似
（因为它们本来就是同一张图）。

**但问题在于：这种做法只能学到低层次的细节。**
Percy 的原话很形象 —— **你没法靠数据增强，把一种狗「增强」成另一种狗。**

**用文本就不一样了：它给你的是更高层次的语义表示。**
后面会看到这些语义表示在什么样的任务上派上用场。
</Callout>

### 3.8 消融实验：为什么不直接从图像预测文本

这一点 Percy 说他会在最后再回来讲，所以值得留意。

CLIP 论文还试了一个**替代方案**：不做这种排序 / 对比，
而是设一个目标 —— **直接从图像预测文本**。预测的形式可以是**词袋（bag of words）**，
也可以是完整的语言模型（逐个 token 预测）。

<Callout type="note" title="补充：词袋（bag of words）是什么">
把一段文字看成「里面出现了哪些词的一个集合」，**完全不管顺序**。
比如 `a dog runs fast` 和 `fast runs a dog` 在词袋视角下是同一个东西。

「从图像预测词袋」就是：给定图像，预测它的说明文字里会出现哪些词，
但不去管这些词以什么顺序、什么句式组织起来。
</Callout>

<Figure src="/courses/cs336/l17-clip-efficiency.png" caption="三种目标函数的零样本 ImageNet 准确率随训练算力的变化：最慢的是 Transformer 语言模型，其次是词袋预测，最快的是 CLIP 的对比目标。来源：CLIP 论文" />

结果**有点反直觉**：**用更强的模型（完整的 Transformer 语言模型）反而更差**，
或者至少**效率更低** —— 比预测词袋还慢，而 CLIP 那种对比式排序又比词袋更快。

Percy 对这件事的解读是：

> **对于你想学的这种表示来说**（注意，这里的指标是 ImageNet 准确率，也就是分类），
> **精确地建模说明文字里的那个 token 序列并没有那么重要** ——
> 你要的只是图像的一个粗略的表示。

（这个观察会在最后一节的总结里再出现一次：**理解和生成要的东西不一样。**）

### 3.9 CLIP 这一节的小结

<Theorem type="proposition" name="CLIP 小结">

1. **CLIP 编码出来的图像表示，捕捉的是文本所给出的语义** ——
   因为它是和文字配对训练的，而文字一般谈的就是语义。
2. **它的设计决策是围绕图像分类做的，所以粒度不细。**
   但即便如此，它仍然是后面一切工作的一个**稳健的起点**。
3. **技术上的一个明显缺点：它要求非常大的批大小**（比如 3 万）。
   批大小如果是 1，显然完全没法学；哪怕是 10 也不行。

</Theorem>

关于第 3 点，还有一层更麻烦的：**softmax 是跨整个批次做的，所以这个损失不好分解。**

对比一下普通的语言模型训练：一批样本里的各个序列是**互相独立**的，
每张卡各算各的，最后在梯度上做一次聚合就行。而 CLIP 的每一行 softmax
都需要**这一批里所有的文本嵌入**，样本之间是耦合的。

**这个问题由下一节的 SigLIP 来解决。**

## 四、SigLIP：把多分类换成一堆是非题

**SigLIP** = **Sigmoid Loss for Language Image Pre-training（用于语言-图像预训练的 sigmoid 损失）**，
来自 Google（[论文](https://arxiv.org/abs/2303.15343)）。可以把它理解成 **CLIP 的改进版**。

### 4.1 目标函数：唯一的区别

<Theorem type="definition" name="CLIP 和 SigLIP 的区别">

- **CLIP 做的是多分类**：我有这一对对齐的（文本, 图像），它是正例；
  然后**跟所有其他候选图像比**，或者**跟所有其他候选文本比**。
- **SigLIP 做的是二分类**：**对任意一个（图像, 文本）对，问一句 —— 它们对齐了没有？**

</Theorem>

对着 3.2 节那张 n×n 的矩阵看：**对角线上是正例，非对角线上是负例，就这么简单。**

<Figure src="/courses/cs336/l17-siglip-code.png" caption="SigLIP 论文里的 sigmoid 损失伪代码。来源：SigLIP 论文" />

逐行拆开：

```python
# img_emb    : 图像模型的嵌入，形状 [n, dim]
# txt_emb    : 文本模型的嵌入，形状 [n, dim]
# t_prime, b : 可学习的温度和偏置
# n          : minibatch 大小

t = exp(t_prime)                  # 温度，取指数保证为正
zimg = l2_normalize(img_emb)      # 归一化，和 CLIP 一样
ztxt = l2_normalize(txt_emb)

# n×n 的相似度矩阵，乘温度再加一个偏置 b（这个 b 是 CLIP 没有的）
logits = dot(zimg, ztxt.T) * t + b

# 标签矩阵：对角线是 +1（是一对），其余全是 -1（不是一对）
labels = 2 * eye(n) - ones(n)

# 每个格子独立地做一次 sigmoid 二分类，加起来
l = -sum(log_sigmoid(labels * logits)) / n
```

**关键差别在最后两行。** CLIP 那边是 `cross_entropy_loss(logits, labels, axis=0)`，
需要沿着一整行做 softmax；SigLIP 这边 `log_sigmoid` 是**逐格子**作用的，
每个格子只看自己的那个数。

<Callout type="insight" title="那个偏置 b 是干什么的">
注意 SigLIP 多了一个可学习的偏置 `b`，CLIP 没有。原因是**正负例极度不平衡**：
一个 $n \times n$ 的矩阵里，只有 $n$ 个格子是正例，$n^2 - n$ 个是负例。
$n = 32768$ 时，正负比是 1 : 32767。

如果不加偏置，模型一上来会被负例淹没。`b` 初始化成一个较大的负数，
相当于先告诉模型「默认答案是『不是一对』」，让它从一个合理的先验出发。

（可以回到 3.2 那个演示，切到 SigLIP 模式，拖一拖那个偏置滑块看它的效果。）
</Callout>

### 4.2 数据：WebLI

用的是另一篇论文（2022 年一个图像语言模型的工作）构建的 **WebLI** 数据集
（[论文](https://arxiv.org/pdf/2209.06794)）：

- **十亿量级**的（图像, 文本）对；
- **从互联网上爬的**；
- 对**图里本身带文字的图片，用自动 OCR 把文字抽出来** —— 这是另一种造图文对的办法；
- 做了过滤，**只保留质量最高的 10%**；
- **支持 100 种语言**（多语言）。

### 4.3 效率：这篇论文的主要卖点

这篇论文的核心主张是**训练起来比 CLIP 高效得多**：

| | 硬件 | 时间 |
| --- | --- | --- |
| CLIP | 256 张 TPU v3 | 10 天 |
| SigLIP | 32 张 TPU v4 | 5 天 |

<Callout type="warning" title="别被「v4 比 v3 新」骗了">
你可能会想：v4 比 v3 新，那单卡肯定更快吧？**按每秒浮点运算数算，其实不是。**

Percy 说 TPU v4 在这个口径下**大概比 v3 慢 60% 左右**。
v4 更好的地方在于**一个 pod 里能塞进更多芯片、互联更好**，
但在这个规模上，单卡算力反而不占优。

所以「32 张 v4、5 天」对上「256 张 v3、10 天」，是一个**非常大**的效率差距。

他也补了一句公道话：**CLIP 那边大概根本没有为吞吐量优化过代码**，
当时的心态就是「先把这东西跑通」。
</Callout>

那 SigLIP 是怎么快起来的？关键在于**这个损失可以漂亮地并行化**。

<Figure src="/courses/cs336/l17-siglip-parallelism.png" wide caption="SigLIP 的分块并行：每张卡先算自己手上那一小块的损失，然后把文本嵌入沿环轮转一圈，逐块补齐所有非对角线的块。来源：SigLIP 论文" />

用第七讲那套系统语言来说，可以把它想成**数据并行（DDP）**：

1. **每台设备存一部分（图像, 文本）对。**
2. **第一轮**：每台设备用手上本地的那些图文对，算出它们之间所有的损失
   （也就是对角线上那些块）。
3. **但样本之间有交互** —— 这一点和语言模型训练不一样，语言模型那边每条序列都能完全因子化。
   所以接下来要**把文本嵌入传出去**：设备 1 拿到 $T_5 \ldots T_8$，
   算出这一块的负例；然后再拿到 $T_9 \ldots T_{12}$，再算一块。
4. **就这样沿着环轮转一圈，直到所有非对角线的块都被覆盖。**

这套做法之所以成立，正是因为 **sigmoid 损失是逐格子的** ——
你可以一块一块地算、算完就加，不需要一次性拿到一整行。
CLIP 的 softmax 做不到这一点：**归一化因子需要整行的数据都到齐。**

### 4.4 批大小被解耦出来了

还有一个好处：**批大小和损失函数解耦了。**

在 CLIP 里，**损失是和批大小绑死的** —— 你换一个批大小，
你优化的就是**另一个损失函数**（因为 softmax 的候选集变了）。

SigLIP 不是。每个格子的二分类和批里有多少样本无关，批大小只影响你**一次算多少个格子**。

于是他们能做这些实验：

- **在小于 1.6 万的批大小下，SigLIP 明显好于 CLIP** ——
  因为 CLIP 的损失函数在批太小的时候本身就退化了；
  而 SigLIP 在小批下只是**方差更大**，**期望上仍然是同一个损失**。
- **他们也往上试到了 100 万的批大小，但那并没有什么帮助。**
  用第 1 单元讲过的说法：**临界批大小（critical batch size）是有上限的**，
  而 **3.2 万基本上就是他们的临界批大小**。

<Callout type="note" title="课堂问答：需要精心设计的负样本采样策略吗">
有同学问：这种对比学习方法一般不是需要一些讲究的采样策略吗？
比如要平衡正负例的比例，或者要挖「难负例（hard negative）」，免得学出偏差？

Percy 的回答：**至少在最初这篇论文里，没有。**
他们就是直接在那张矩阵上操作 —— 同一批里其他人的文本自然就是负例，没有额外的挑选。
</Callout>

## 五、把图像编码注进语言模型：VLM 的标准模板

到这里我们手上有 CLIP 和 SigLIP 了。它们的作用是：

> **吃进一张图（在这个版本里是固定尺寸，比如 336×336），吐出一个向量，
> 这个向量里装着这张图的语义。**

接下来开始搭**视觉语言模型（VLM, vision language model）**。
Percy 讲了两个家族：**LLaVA** 和 **Qwen-VL**。
两者的**大框架非常像**，细节上有差别。

基本想法就一句话：**把图像的嵌入拿过来，注射进一个语言模型。**

<Callout type="insight" title="注意这是「拼装」，不是「从头训」">
这套做法的味道更像**中期训练（mid-training）或后训练（post-training）**，
而不是预训练：

- 拿一个**现成的图像编码器**（CLIP / SigLIP）；
- 拿一个**现成的大语言模型**；
- **把它们缝在一起**，而不是从零训练一个新东西。

这也是为什么这些论文里绝大部分的篇幅是在讲**数据**而不是架构。
</Callout>

### 5.1 LLaVA：最简单的那个版本

**LLaVA** = **Large Language and Vision Assistant（大型语言与视觉助手）**，
2023 年（[论文](https://arxiv.org/abs/2304.08485)）。

它当时让人兴奋的原因是：那个时候闭源模型（GPT-4）已经能做**视觉推理**了，
而 LLaVA 表明**开放模型也能做一些视觉推理**。当然效果不如 GPT-4，
**但它是开放的，大家终于能看到引擎盖底下是什么。**

#### 零件清单

- **视觉编码器**：CLIP —— 具体用的是性能最好的 **ViT-L/14**（14×14 的 patch）。
- **文本解码器**：**Vicuna** —— 就是第一代 LLaMA 在
  [ShareGPT](https://www.lmsys.org/blog/2023-03-30-vicuna/) 的对话上微调出来的模型。

<Callout type="note" title="补充：ShareGPT 是什么">
那是一个网站，人们把自己和 ChatGPT 的对话贴上去分享。
于是这些对话就变成了一份现成的指令微调数据。Percy 说这个网站现在应该已经关掉了。
</Callout>

#### 数据：让 GPT-4 造出来

LLaVA 的数据是**合成**的，来源是 **MS COCO**：

- MS COCO 是一个**人工标注**的数据集，众包工人给每张图标了**边界框**
  和**描述图里有什么的说明文字**。
- LLaVA 的做法是：**把这些说明文字、或者检测到的物体列表，喂给 GPT-4，
  让 GPT-4 生成问题或对话。**
- 然后**把 GPT-4 的生成结果，和原来那张图配成一对。**
- 一共 **15.8 万条样本**。

<Figure src="/courses/cs336/l17-llava-gen.png" wide caption="LLaVA 的数据合成：拿 MS COCO 里人工标注的 caption 和边界框喂给 GPT-4，让它生成对话、详细描述、复杂推理三类样本。来源：LLaVA 论文" />

Percy 逐格讲了这张图：这是 MS COCO 里的一张图，这是人标的说明文字，这也是人标的（边界框）。
然后你让 GPT-4「给我生成一段对话」，你会得到一问一答；
你让它「给我生成一段详细描述」，它生成的东西大概和原 caption 差不多；
你让它「生成复杂推理」，它就生成那样的东西。

#### 模型：一个矩阵 W 就够了

<Figure src="/courses/cs336/l17-llava-architecture.png" wide caption="LLaVA 的结构：图像过视觉编码器得到 Z_v，再过一个投影矩阵 W 变成 H_v；文本走普通嵌入变成 H_q；两串向量拼在一起送进语言模型。来源：LLaVA 论文" />

一步步看：

1. 图像送进视觉编码器（CLIP），**得到一个向量**。
2. **但这个向量并不在文本所在的那个空间里。** 于是**乘上一个矩阵 $W$**，
   得到另一个向量 —— 这个向量就和**文本嵌入在同一个空间**了。
3. 文本这边就是标准做法：查嵌入表，得到向量。
4. **于是文本被编码成一串向量，图像也被编码成一串向量，
   这一整串向量直接送进一个标准的 Transformer，产生输出。**

<Callout type="insight" title="这一步在干什么">
用 Percy 的话说：**我们在某种意义上把图像转换成了「文本 token」** ——
这样就能直接**利用已经预训练好的语言模型**，一行架构都不用改。

注意这里的投影 $W$ 是**最简单的形式：一个线性投影**。
更复杂的做法是存在的（Flamingo、Q-former），但 LLaVA 就用了个矩阵。
后面会看到这个「适配器（adapter）」一路变复杂：
线性投影 → 两层 MLP → 交叉注意力 → DeepStack。
</Callout>

#### 训练：分两阶段，各冻结什么

<Theorem type="proposition" name="LLaVA 的两阶段训练">

- **阶段一（对齐 alignment）**：**冻结视觉编码器，冻结语言模型，只训练 $W$。**
- **阶段二（微调 fine-tuning）**：**仍然冻结视觉编码器，训练 $W$ 和语言模型。**

</Theorem>

**为什么第一阶段只训 $W$？** Percy 解释得很具体：
如果 $W$ 是随机的，那么图像编码乘出来的那些向量，**根本不长得像任何自然语言 token 的嵌入**。
**训练 $W$ 的目标，就是让它们看起来像自然语言 token 的嵌入。**
这就是「对齐」二字的意思。

第二阶段是在他们自己合成的数据上**微调语言模型** ——
数据的形式就是「这里有一张图，这里有一段对话 / 描述 / 复杂推理」。

<Callout type="warning" title="注意：这一整套都是「图 + 文 ⇒ 文」">
输入可以是图像加文本，**但输出永远是文本**。
这些模型**不生成图像，也不生成视频**。
（这一点要到第七节讲 Chameleon 时才会被打破。）
</Callout>

<Figure src="/courses/cs336/l17-llava-example.png" wide caption="LLaVA 论文里的例子：问「这张图有什么不寻常的地方」，模型答出「一般人不会在小货车后面熨衣服」。来源：LLaVA 论文" />

Percy 说这个例子挺有意思的。而且论文还强调了一点：
**即使用户的提问并没有在问「哪里不寻常」，模型仍然会主动谈到这一点。**
GPT-4 当然能做到这个，但**当时其他的开放模型做不到**。

### 5.2 LLaVA-OneVision：多图、视频，和「保住分辨率」

跳到 2024 年的 **LLaVA-OneVision**（[论文](https://arxiv.org/pdf/2408.03326)）。
这是 LLaVA 系列的最新版 —— 中间还有 LLaVA 1.5、LLaVA-Next，
Percy 说这些改进他会一并放在这里讲。

**它的主要变化不是配方，而是野心**：他们想覆盖更多类型的多模态应用，
所以要能处理**多张图像**和**视频**。

<Figure src="/courses/cs336/l17-llava-onevision.png" wide caption="LLaVA-OneVision：同一套结构现在要吃单图、多图和视频三类输入（视频就是抽出来的一串帧）。来源：LLaVA-OneVision 论文" />

**零件全都升级了一遍**，但整体还是同一个系统：

| 部件 | LLaVA（2023） | LLaVA-OneVision（2024） |
| --- | --- | --- |
| 视觉编码器 | CLIP ViT-L/14 | **SigLIP** |
| 文本解码器 | Vicuna | **Qwen-2 72B** |
| 适配器 | 线性投影 | **两层 MLP** |

Percy 对视觉编码器补了一个细节：他们用的是
**最后一个 Transformer 层「之前」和「之后」的网格特征（grid features）**。
Qwen-2 是当时最好的开放语言模型之一。

用他的话说：**这就像你有一套音响，把里面的零件挨个换成更好的，但系统还是那个系统。**

#### AnyRes：不要再把文档裁成 336×336 了

数据处理这一块有一个真正重要的改进。动机是 **OCR**（光学字符识别）。

<Callout type="warning" title="为什么 OCR 逼着他们改分辨率">
OCR 需要保住**非常精细**的信息。用 Percy 的话说：
**否则一个 J 看起来就像一个 I，那就完蛋了。**

回想 3.4 节：**CLIP 会把图缩放并裁剪成 336×336。**
如果你手上是一份文档，裁成 336×336 之后，**你根本读不出来上面写了什么。**
</Callout>

解决方案叫 **AnyRes**，最早出现在 LLaVA 1.5
（[论文](https://static.hliu.cc/files/llava/improved_llava.pdf)）。想法很直接：

<Theorem type="definition" name="AnyRes">

1. **把图像切成 $a \times b$ 块。**
2. **每一块的尺寸，正好等于视觉编码器能吃的分辨率**（比如 336×336）。
3. **每一块分别编码**，得到各自的一串向量。
4. **把这些向量拼接起来。**

如果原图分辨率实在太高，切出来的 token 太多，
就**用双线性插值（bilinear interpolation）把 token 数降下来**。

</Theorem>

Percy 的总结是：**你注意到视觉编码器处理不了高分辨率，那么与其把图降采样，
不如把它裁开、分别去看图的不同部分。**

<Figure src="/courses/cs336/l17-llava-onevision-anyres.png" wide caption="AnyRes 的两条路：上面一条把论文首页切成小块分别编码（必要时再用双线性插值压 token 数），下面一条把整页缩小成一张缩略图编码。两条的向量拼起来送进 LLM。来源：LLaVA-OneVision 论文" />

看这张图上的实际做法：假设你手上是一张论文的截图。

- **一条路是降采样**：得到整张论文的一个**缩略编码** —— 这条保住了全局布局。
- **另一条路是切块**：每一块正好是视觉编码器期望的分辨率，各自编码。
- **最后把所有这些拼起来**；如果块太多，就用插值把数量降下来。

<Callout type="insight" title="AnyRes 是「自适应」，这正好接上第 1 讲的第二条">
Percy 特意点出了这一层：**AnyRes 的好处在于它是自适应的。**

而 **Transformer 本来就是自适应的** —— 语言里句子可以是任意长度，
Transformer 早就处理得很好了。**结果发现图像也可以是任意分辨率。**
所以这只是把 Transformer 已有的那份「动态能力」重新捡起来用。

对照第 1 讲留的那两条性质：**第二条要求「块必须是可变的」**。
AnyRes 就是这条性质在图像上的具体形态 ——
一张高清文档配上更多的块，一张小图配少一点。
</Callout>

#### 三种输入，被有意凑成差不多长

他们要同时处理**单图、多图、视频**三类输入。
技术上这些都可以归约成图像，但他们**故意在秤上压了一下**：

<Theorem type="proposition" name="目标：让所有模态产出大致相同的长度">

- **单张图**：用**更高的分辨率**（切成多块，看得最细）。
- **多张图**：每张用**基础分辨率**。
- **视频**：每一帧用**更低的分辨率**（更少的 token）。

</Theorem>

Percy 的说法很形象：**如果只有一张图，我可以仔细端详它；
如果给我一堆图，我就只能远远地看个大概。**

为什么要这么做？因为**视频可能非常长**，
他们不希望数据集被一堆重复的视频帧主导。
而且视频**很快就会撞上上下文长度的问题**（LLaVA-OneVision 最多用 32 帧）。

> 后面会看到：**能不能处理多模态，很大程度上就是能不能处理长上下文。**

<Figure src="/courses/cs336/l17-llava-onevision-modalities.png" wide caption="LLaVA-OneVision 的 token 预算表：单图 (1+N)×729、多图 N×729、视频 N×196，上限分别是 7290 / 8748 / 6272 个 token —— 三者被刻意调到同一个量级。来源：LLaVA-OneVision 论文" />

<Callout type="tip" title="动手：三种模态各值多少 token">
这张表上的数字是可以自己拨的。三条柱子的满档分别是 **7,290 / 8,748 / 6,272** ——
**它们被有意调到了同一个量级**，这就是上面那条「让所有模态产出大致相同长度」的具体落实。

几件值得试的事：

- 把「单图的 crop 数」从 9 拖到 0：单图只剩一张缩略全图，729 个 token，
  这时候你就读不出文档上的字了 —— 这正是 AnyRes 要解决的问题。
- 把视频帧数拖满，再看最后那根黄色的柱子（一段一千 token 的纯文本）。
  **一段几秒钟的视频，比一千个 token 的文字还贵好几倍**，
  但它承载的信息远没有那么多。这就是「信息密度」这个说法的具体含义。

</Callout>

#### 数据：质量优先，但也非常「应试」

<Figure src="/courses/cs336/l17-llava-onevision-data-1.png" wide caption="LLaVA-OneVision 的单图数据构成：按任务类别（通用问答、文档 / 图表、OCR、数学推理等）拼出来的一大堆数据集。来源：LLaVA-OneVision 论文" />

<Figure src="/courses/cs336/l17-llava-onevision-data-2.png" wide caption="多图与视频部分的数据构成。可以看到很多任务是高度特定的，比如「给两张图找不同」。来源：LLaVA-OneVision 论文" />

论文自称的哲学是**质量优先于数量**。Percy 给了另一种读法：

> **这些数据是非常「有针对性」的。** 你去看里面很多条目，
> 都是很任务化的 —— 视觉问答、回答关于表格的问题，等等。

所以这**确确实实是后训练的地盘**：你想让模型会做这些任务，
于是你就造一批这些任务的数据。

他还点了一句不太好看的事实：**这项工作毫不掩饰地在蒸馏 GPT-4** ——
用 GPT-4 生成数据来训自己的模型，好把性能刷上去。
Percy 说这**大概算不上理想**，但**如果你没有标注预算，你就只能这么做**。

数据里有单图、多图、视频，种类非常多，有些非常具体 ——
比如**给你两张图，让你找不同**。

#### 训练：从易到难，三个阶段

<Figure src="/courses/cs336/l17-llava-onevision-training.png" wide caption="LLaVA-OneVision 的训练阶段表：先只训投影器做对齐，然后是偏知识的高质量数据，最后是贴近下游任务的数据。来源：LLaVA-OneVision 论文" />

哲学是**由易到难**。和 LLaVA 的两阶段相比，现在是三个阶段：

1. **第一阶段还是对齐**：**只训练投影器（适配器）。**
2. **第二阶段**：高质量数据，**更侧重知识**。
3. **第三阶段**：**侧重那些长得像下游任务的样本。**

Percy 坦白说：**他不确定这里有什么特别原则性的理由**要分成三段，
区别就在于第二段偏知识、第三段偏下游任务。

#### 一个有意思的发现：模态之间会迁移

这篇论文里 Percy 觉得比较有意思的一点是：**能力会在模态之间迁移。**

<Figure src="/courses/cs336/l17-llava-onevision-transfer-s1.png" wide caption="迁移之一：训练时只有单图的图表 / 图示数据，测试时却能对「一张表格 + 一张图表」两张图做对话。来源：LLaVA-OneVision 论文" />

**例子一**：训练数据里关于图示和图表的部分**只有单图**，
但这个能力**泛化到了多图**。也就是说，**训练时它从没见过
「一张表 + 一张图表，然后问你关于它们的问题」这样的样本**，
但测试时你给它两张图，它就能围绕它们对话。

<Figure src="/courses/cs336/l17-llava-onevision-transfer-s2.png" wide caption="迁移之二：OCR 能力来自单图数据，关系推理能力来自多图数据，两者合起来能驱动 GUI agent —— 给它一串界面截图，它能描述发生了什么。来源：LLaVA-OneVision 论文" />

**例子二**：训练时 **OCR 只在单图数据上有**，
**关系推理只在多图数据上有**，
但这两者能合起来泛化到新场景 —— 比如**驱动 GUI agent**：
你给它一串界面截图（这是多图），它能生成对这些截图的描述。

<Figure src="/courses/cs336/l17-llava-onevision-transfer-s8.png" wide caption="迁移之三：视觉提示（用一个圈标出要关注的部分）只在单图数据里出现过，却能泛化到视频 —— 「描述视频里被圈出来的那个球员」。来源：LLaVA-OneVision 论文" />

**例子三：视觉提示（visual prompting）**。
所谓视觉提示是指：图上画了一个圈，这个圈本身是图像的一部分，
但它的用意是**标出你要模型关注的那个部位**。

这类数据**只在单图里存在**，但它能**泛化到视频**：
用户说「描述视频里被圈出来的那个球员」，而这个球员横跨很多帧。

<Callout type="insight" title="Percy 对这件事的评价">
他说，第一次看到这些数据时他的反应是：
**「哎呀，你这基本上是在对着每一个任务各配一批数据，这就是监督学习啊。」**

**但如果你的任务数量足够多，这些模型似乎真的能做出一些迁移** ——
他说这一点还是**让人放心的**。
</Callout>

#### LLaVA 这条线的小结

<Theorem type="proposition" name="LLaVA 小结">

1. **标准的 VLM 模板：视觉编码器 + 投影器 + 语言模型。**
2. **大部分工作量花在数据整理上**，而且非常倚重**合成的、任务特定的数据**。
3. **它是开源的** —— LLaVA 系列是少数**不只放模型权重、还放数据**的工作之一。
   所以你真的能复现它、研究它。

</Theorem>

## 六、Qwen-VL 系列：同一个模板，三代演进

现在看另一个家族。Qwen 也是从 **2023 年**开始训多模态模型的。
Percy 说这一段他会讲得快一些，因为**你应该已经能看出套路了**。

### 6.1 Qwen-VL（2023）

（[论文](https://arxiv.org/abs/2308.12966)）

**架构**：

- **视觉编码器**：**OpenCLIP 的 ViT-bigG**（14×14 的 patch）——
  回忆一下，OpenCLIP 是 CLIP 的开放复现，所以这基本上就是一个 CLIP 编码器。
- **适配器**：**一层交叉注意力（cross-attention）**，
  里面**融入了 2D 位置编码**，并把输出**映射到固定长度 256**。
- **特殊 token**：`<img>`、`<box>`、`<ref>` —— 分别用于标记图像、边界框和指代描述。

<Callout type="insight" title="「固定长度 256」是个信号">
Percy 对这一点的评价是：**这显然不怎么动态。**

但他也说，在这个时间点上**视觉编码器本身也不动态**（还是固定分辨率输入），
所以适配器固定输出 256 个 token 是配套的。
**这一点后面会被改掉** —— 6.2 节的动态分辨率就是冲着它去的。
</Callout>

<Callout type="note" title="一个小插曲">
课上讲到特殊 token 那一行时，Percy 发现幻灯片上的 `<img>` 之类**被吃掉了**，
他说了一句：**「这就是你把幻灯片写成 HTML 的下场。」**

原因是尖括号在 HTML 里会被当成标签解析掉。
（写这份讲义时也遇到了同一个问题 —— 所以上面那三个 token 是包在代码格式里的。）
</Callout>

**训练**：和 LLaVA 一样是**三个阶段**。

<Figure src="/courses/cs336/l17-qwen-vl-stages.png" wide caption="Qwen-VL 的三阶段训练。雪花表示冻结、火苗表示训练：阶段一冻语言模型、训 ViT 和交叉注意力（低分辨率图文对）；阶段二全部解冻（高分辨率、多任务与交错数据）；阶段三冻 ViT、训适配器和语言模型（对话式交错数据）。来源：Qwen-VL 论文" />

- **阶段一**：论文管它叫「预训练」，但 Percy 提醒 —— **这不是从零预训练**。
  用的是**大规模低质量数据**（可以看到有 14 亿量级的样本）。
  **冻结语言模型，训练视觉编码器 + 适配器。**

  <Callout type="warning" title="这一点和 LLaVA 不一样">
  LLaVA 是**冻结视觉编码器**，Qwen-VL 反过来 —— **它训练视觉编码器**。
  </Callout>

  <Figure src="/courses/cs336/l17-qwen-vl-stage1.png" caption="阶段一用到的数据集：以大规模、低质量的网络图文对为主。来源：Qwen-VL 论文" />

- **阶段二**：换成**更高质量的、任务特定的数据**，同时**提高分辨率**，
  **训练全部参数**。数据里能看到那些常见面孔：VQA 数据集、图表问答等等。

  <Figure src="/courses/cs336/l17-qwen-vl-stage2.png" caption="阶段二的数据集：视觉问答、图表问答等任务特定数据。来源：Qwen-VL 论文" />

- **阶段三**：**指令微调数据**。**冻结视觉编码器，训练适配器 + 语言模型。**

<Figure src="/courses/cs336/l17-qwen-vl-examples.png" wide caption="Qwen-VL 的能力示例：中文问答、代码、输出边界框（「蜘蛛侠和绿巨人在哪里」）、OCR。来源：Qwen-VL 论文" />

Percy 逐个点了几件事：

- **这是 Qwen，所以有些例子是中文的。**
- **他们的意图是让这个模型「包住」一个语言模型** ——
  语言模型会写代码，所以他们也希望这个视觉模型的代码能力不差。
- 有一个例子是给它一张图，问「蜘蛛侠和绿巨人在哪里」，
  它能**输出边界框**。注意：**它并不输出图像，它输出的只是一个边界框（几个数字）。**
- 还能做 **OCR**。

### 6.2 Qwen2-VL（2024）：动态分辨率和 MRoPE

（[论文](https://arxiv.org/abs/2409.12191)）

**视觉编码器换成更大的 ViT（6.75 亿参数）。**

<Figure src="/courses/cs336/l17-qwen2-vl-architecture.png" wide caption="Qwen2-VL 的架构图。注意下方标出的实际数字：一张 1092×8204 的长截图值 11427 个 token，一条 224×28 的公式只值 8 个，一张 1260×700 的风景照值 1125 个，一段 16 秒的 644×336 视频值 2208 个。来源：Qwen2-VL 论文" />

#### 关键改动：动态分辨率

Percy 说这是最主要的变化。回想从 LLaVA 到 LLaVA-OneVision 的路径：
**他们意识到必须处理不同尺寸的图像；而一旦你要做视频，就更明显必须有某种动态分辨率。**

看架构图上标的那几个真实数字：

| 输入 | 尺寸 | token 数 |
| --- | --- | --- |
| 图 1（一整页长截图） | 1092 × 8204 | **11,427** |
| 图 2（一条公式） | 224 × 28 | **8** |
| 图 3（风景照） | 1260 × 700 | **1,125** |
| 视频 1 | 644 × 336，16 秒 | **2,208** |

**一张巨大的长截图值一万一千个 token，而那条小小的公式只值 8 个。**

#### 这些数是怎么算出来的

做法和 LLaVA 的 AnyRes 是同一个思路：

1. **每个 224×224 的块用 ViT/14 编码。**
   $224 / 14 = 16$，所以一个 224×224 的块被切成 $16 \times 16 = 256$ 个 patch。
2. **为了压缩上下文，把每 2×2 个 patch 合并成一个。**
   $256 / 4 = 64$，于是一个 224×224 的块最终产出 **64 个 token**。

<Callout type="warning" title="课件上这里写的是 66">
课件上这一行写的是「每 2×2 压缩后 ⇒ 66 tokens」。
按上面这套算术应该是 64。这两个数差 2，课上没有解释这个差别，**这里就照实记下来**。

不过**架构图上标的那几个数字倒是可以严丝合缝地对上**，用的正是「一个 token 覆盖 28×28 像素」
（也就是 14 的 patch 再合并 2×2）这个口径：

$$
\left\lfloor \frac{224}{28} \right\rfloor \times \left\lfloor \frac{28}{28} \right\rfloor = 8 \times 1 = 8
$$

$$
\left\lfloor \frac{1260}{28} \right\rfloor \times \left\lfloor \frac{700}{28} \right\rfloor = 45 \times 25 = 1125
$$

$$
\left\lfloor \frac{1092}{28} \right\rfloor \times \left\lfloor \frac{8204}{28} \right\rfloor = 39 \times 293 = 11427
$$

三个数全中。视频那 2208 也是 $8 \times 276$，其中
$\lfloor 644/28 \rfloor \times \lfloor 336/28 \rfloor = 23 \times 12 = 276$ 正是一帧的 token 数。

**（3.5 节那个切 patch 的演示里，四个预设就是这四张图，可以回去拨一拨。）**
</Callout>

Percy 补了一句：**他们起手用的仍然是 OpenCLIP 那个 ViT，只不过在这里会被继续微调。**

**视频**：**每秒抽 2 帧**，但**最多 16384 个 token**封顶。

#### MRoPE：把 RoPE 从 1 维推广到 3 维

这里回应了 3.5 节那个关于位置编码的提问 —— **Qwen2-VL 确实做了不一样的东西。**

<Figure src="/courses/cs336/l17-qwen2-vl-mrope.png" wide caption="MRoPE：视频里每个 patch 的位置不再是一个整数，而是一个 (时间, 高, 宽) 三元组；文本 token 的三个坐标则取成相同的值。来源：Qwen2-VL 论文" />

<Theorem type="definition" name="多模态旋转位置编码（MRoPE）">
回忆一下 RoPE：它的设计目标是让两个向量的内积**只依赖于它们之间的距离**。
在文本里，「距离」就是**相隔多少个 token**，是一维的。

**多维版本本质上是同一件事，只是现在有三个维度**：

- **高度（height）**
- **宽度（width）**
- **时间（time）**

于是每个位置、每个 patch 的位置**不再是一个整数，而是一个三元组坐标**
$(t, h, w)$。

**计算 MRoPE 的做法是：对每一个维度分别计算 RoPE，然后把结果拼接起来。**
</Theorem>

Percy 的评价：**这是个相当直白的想法。** 不过他也预告了：
**到 Qwen3 会看到这个做法其实有点次优。**

- **语言模型用 Qwen2 初始化**（这是 Qwen 的模型，当然用 Qwen 初始化）。
- **视觉编码器用 DFN 初始化**（[论文](https://arxiv.org/abs/2309.17425)）。

训练还是三个阶段，和前面很像：

1. **阶段一**：只训练视觉编码器。
2. **阶段二**：训练全部参数。
3. **阶段三**：在指令跟随数据集上训练语言模型。

### 6.3 Qwen3-VL：四个小改动，但可能都很重要

（[论文](https://arxiv.org/abs/2511.21631)）

<Figure src="/courses/cs336/l17-qwen3-vl.png" wide caption="Qwen3-VL 的总体结构图，大框架和 Qwen2-VL 基本一致，改动集中在位置编码、适配器和损失加权上。来源：Qwen3-VL 论文" />

Percy 说这些**算不上结构性的大改动，但它们大概确实影响模型质量**。

#### 语言模型侧

- 换成 **Qwen3 系列**：既有稠密模型也有 MoE 模型，**最大到 235B-A22B**
  （2350 亿总参数、激活 220 亿）。Percy 说 Qwen3 模型本身**非常强**，
  **这对最终模型的质量帮助很大**。
- **长上下文理解：上下文长度可达 256K。**
  这一点**对长视频特别重要**。

#### 视觉编码器：SigLIP-2

用 **SigLIP-2**（[论文](https://arxiv.org/pdf/2502.14786)）—— 基本上是 SigLIP 的改进版，
**架构完全相同**，是**特意设计成向后兼容的**。

#### 改动一：交错的 MRoPE

问题出在 Qwen2-VL 那个「分块然后拼接」的做法上。

<Theorem type="proposition" name="为什么分块拼接不好">
Qwen2-VL 的做法是：**把前一段维度分给时间，接下来一段分给宽度，再下一段分给高度。**

但**回想 RoPE 的性质：每个分量对应一个不同的频率。**
于是这样切的后果是 —— **所有时间维度可能都落在低频段，
所有高度维度都落在高频段。**

Qwen3-VL 的修正是把它们**交错（interleave）**：

$$
[\,t\ w\ h\ t\ w\ h\ t\ w\ h\ t\ w\ h\, \ldots\,]
\quad\text{而不是}\quad
[\,t\ t\ t\ t\ w\ w\ w\ w\ h\ h\ h\ h\,]
$$

这样**每一个轴都同时暴露在低频段和高频段上。**
</Theorem>

#### 改动二：显式的视频时间戳

之前，**时间是隐含在位置编码里的**：视频里每一帧天然处在不同的位置，
所以「时间」这个概念是从位置中隐含派生出来的。

**Qwen3-VL 把时间变成显式的**：`0 秒` 现在**本身就是一个 token**，
序列里真的有一个 token 代表这个时间。

他们发现这样有帮助。Percy 推测原因是：**这样你就有了一个可以直接指代的东西** ——
比如「两秒之后发生了什么」。

#### 改动三：按平方根归一化的逐 token 损失

<Theorem type="definition" name="为什么要重新加权">
问题：**有些样本（视频）非常长，有些样本（单图）非常短。**

而通常的做法是**每个 token 一视同仁** —— 那结果就是
**视频样本会主导整个损失**。

**Qwen3-VL 的做法是：把每个样本按它长度的平方根做归一化**，
以此**压低超长样本的影响**。

（Percy 说论文里的细节讲得不太清楚，这是他的理解。）
</Theorem>

#### 改动四：DeepStack 适配器

先回顾一下适配器这条演化线：

| 模型 | 适配器 |
| --- | --- |
| LLaVA | 线性投影（一个矩阵 $W$） |
| LLaVA-OneVision | 两层 MLP |
| Qwen-VL | 一层交叉注意力 |
| Qwen3-VL | **DeepStack** |

**DeepStack**（[论文](https://arxiv.org/abs/2406.04334)）做的是**跨层融合**：

> 视觉编码器本来就会算出**一整叠**视觉嵌入（每一层一份）。
> DeepStack 把这些嵌入**直接加进语言模型的残差流（residual stream）里**，
> 而且是**加到多个层上**，不只是加在输入端。

<Callout type="insight" title="这是「深度融合」">
Percy 的说法是：这比之前的做法更像一种**深度融合（deep fusion）** ——
之前的适配器把视觉编码器当成一个**黑盒**，它只负责吐出一串向量，
然后这串向量从语言模型的最底下进去。

DeepStack 则是把视觉信息**注入到语言模型的多个层次**。
</Callout>

#### 训练：四个阶段 + 三个阶段

<Figure src="/courses/cs336/l17-qwen3-vl-pretraining.png" wide caption="Qwen3-VL 的预训练四阶段：S0 只训 merger（适配器），预算 670 亿 token、序列长 8192；S1 起训练全部参数，序列长度依次是 8192、32768、262144，token 预算约 1T / 1T / 1000 亿 —— 大部分 token 确实花在中间两个阶段。来源：Qwen3-VL 论文" />

Percy 说到这里流程已经**相当复杂**了：

- **预训练有 4 个阶段**：
  1. 先**只训适配器**（和前面所有模型一样）；
  2. 然后是三段**逐步加长序列**的训练：**8K → 32K → 256K**。
  **大部分 token 花在第二和第三阶段。**
- **后训练有 3 个阶段**：
  1. 在**长思维链（long CoT）数据**上做 SFT；
  2. **知识蒸馏**；
  3. **强化学习**。

他的评价是：**到这个地步，这已经是一篇系统论文了。**
他只把核心的新想法挑出来讲，但论文里还有大量不同的细节。

<Figure src="/courses/cs336/l17-qwen3-vl-results.png" wide caption="Qwen3-VL 和闭源模型（Gemini、GPT-5、Opus 4.1）的对照。加粗的是每一行的最好成绩 —— Qwen 拿下了相当多的加粗。来源：Qwen3-VL 论文" />

结论：**这是一个相当好的模型。** 把这些 benchmark 和闭源模型
（Gemini、GPT-5、Opus 4.1）比，Qwen 系列**确实相当强**。

### 6.4 课堂问答集中收一收

<Callout type="note" title="课堂问答：这些模型能生成视频吗？它怎么知道该输出视频还是文本">
Percy 的回答很干脆：

**首先，这些模型不生成视频，也不生成图像。所有的多模态都在输入侧，你输出的永远是文本。**

其次，除了 RL 那个阶段以外，**这些阶段都是在监督每一个 token** ——
没有什么「语言模型当裁判」或者对描述质量的打分，
**训练信号完完全全就是你数据集里写的那些东西**。到了 RL 阶段，你才能在奖励上做文章。
</Callout>

<Callout type="note" title="课堂问答：从系统角度看，训多模态模型比训纯语言模型更难吗">
Percy 说：**肯定不会更容易。**

难点主要在**数据加载**：**视频数据光是把它读进来就可能是瓶颈。**
他指出，讲语言模型时**我们基本没操心过数据加载**，因为那太便宜了；
但在多模态里这必须被认真对待，
**第 2 单元讲过的那些原则 —— 保证数据加载和实际计算异步进行 —— 现在必须用上了。**

提问的同学还提到「图像和视频的 token 更多」。Percy 说这确实是
**为什么会出现前面那种归一化**，好让视频不至于占太大权重。
不过他补充了一句：**你也总是可以直接降权** —— 两周前讲过**数据配比（data mixture）**，
你随时可以按你认为合理的方式给各部分加权。

最后一点他澄清了一个可能的误解：**外面的文本 token 也非常多。**
这些模型是在几十万亿 token 上训的，
**他不会说多模态 token 的数量远远压过文本 token。**
</Callout>

<Callout type="note" title="课堂问答：对齐阶段里，模型怎么知道「语言」是什么">
有同学问：做对齐的时候，模型怎么知道语言长什么样？

Percy 的回答：**语言模型一定是预训练过的** ——
否则「对齐到它」这件事根本没有意义。
在对齐阶段，**语言模型是冻结的**，你只是在训练那个适配器，
把给定的视觉编码器和给定的语言模型接起来。

**怎么训？** 你定一个 **token 预算**（比如 670 亿 token），然后就训。
**这里没有什么自适应的停止阈值。**

（6.3 节那张预训练阶段表里，第一阶段 S0 的 token 预算写的正好就是 67B。）
</Callout>

<Callout type="note" title="课堂问答：视觉编码器的参数量为什么比语言模型小那么多">
Percy 说：**一般来说是的，小很多。** 以 LLaVA-OneVision 为例：

- **语言模型：720 亿参数**
- **投影器：7200 万参数**
- **ViT：一般不到 10 亿参数**

**为什么？** 因为**视觉编码器在做的是一件非常局部的事** ——
它看的是一个 patch，而 **patch 非常小，它只是在理解这些 patch。
那里面没有多少知识，也不需要推理。**

**所以模型的绝大部分能力仍然在语言模型里。**
</Callout>

## 七、Chameleon：把一切都变成离散 token

现在跳到一个**很不一样**的东西：Meta 在 2024 年的 **Chameleon**
（[论文](https://arxiv.org/pdf/2405.09818)）。

### 7.1 前面那条路的死穴

**到目前为止，VLM 都是把图像编码成向量，再注射进语言模型。
而因为它就是一个语言模型，所以它只能生成文本 —— 它生成不了图像。**

修补这件事有很多办法。比如你可以拿一个 VLM，**给它接一个扩散头（diffusion head）**，
这样就能生成图像了。

<Callout type="note" title="补充：扩散模型是什么（Percy 这次没展开讲）">
扩散模型（diffusion model）是今天生成图像的主流做法，和自回归完全是两条路。

**基本想法**：训练时，往一张真实图像上一步步加高斯噪声，直到它变成纯噪声；
然后训练一个网络**学会反向的那一步** —— 给定一张带噪声的图，预测该去掉哪些噪声。
生成时就从纯噪声出发，反复调用这个网络去噪，几十步之后就长出一张图。

**为什么它特别适合生成**：它不需要把图像离散化成 token，
整个过程都在连续的像素（或连续的隐空间）上进行，
而且它可以**反复地在整张图上打磨细节** —— 后面 8.2 节会看到，
这正是它在**高频细节**上很强的原因。

Percy 这一讲没有讲扩散模型，但在总结里多次提到它，所以这里补一段最粗的轮廓。
</Callout>

**但 Chameleon 想的是另一件事：**

> **如果我们把一切都映射成离散 token，会怎么样？**

Percy 说这个想法在美学上很吸引他（他自嘲说「这大概说明我是个搞语言的」）：
**因为这样一来，你就能用完全同一种方式来分析和生成图像 —— 一切都是离散 token。**

<Figure src="/courses/cs336/l17-chameleon.png" wide caption="Chameleon：图像先过一个「图像分词器」变成离散 token，和文本 token 混在一起送进同一个自回归语言模型；生成时反过来，把生成出来的图像 token 交给「图像反分词器」还原成图像。来源：Chameleon 论文" />

想生成图像？那就给一个文本 prompt，然后**接着往下生成 token 就行了** ——
生成出来的那些 token 只不过恰好是图像 token。你也可以把文本和图像的顺序反过来。

<Figure src="/courses/cs336/l17-chameleon-example.png" wide caption="Chameleon 论文里的交错生成例子：用户说「我很无聊，能给我看几只鸟吗」，模型输出的是文字、图像、更多文字、更多图像交错在一起的序列。来源：Chameleon 论文" />

<Callout type="insight" title="omni 模型的另一种实现路线">
Percy 的总结是：**omni 模型的愿景是文本和图像真正活在同一个空间里，
而 Chameleon 实现这一点的方式是「把所有东西都弄得看起来像文本」。**

这只是实现方式之一 —— 第八节会看到，前沿模型今天大概走的是另一条路。
</Callout>

### 7.2 关键零件：VQ-VAE，把连续向量变成离散码

（Chameleon 用的图像分词器来自[这篇论文](https://arxiv.org/pdf/2203.13131)）

**和前面所有模型最关键的区别是：这个编码器必须映射到离散 token，
这样我们才能「生成」它们。**

<Callout type="insight" title="为什么生成非要离散不可">
自回归语言模型生成一个 token，靠的是**在词表上输出一个概率分布，然后采样**。
「在 5 万个候选里挑一个」这件事是有定义的。

但如果 token 是一个连续的 512 维向量，**「采样一个向量」该怎么做？**
没有词表可以做 softmax。所以想让 Transformer 把图像**生成出来**，
就得先把图像变成有限个离散符号。
</Callout>

用的技术是 **VQ-VAE**（**Vector Quantized Variational Autoencoder，向量量化变分自编码器**），
出自 2017 年的[一篇相当老的论文](https://arxiv.org/pdf/1711.00937)。

<Figure src="/courses/cs336/l17-vq-vae.png" wide caption="VQ-VAE：编码器把图像变成一格格连续的隐向量，量化这一步把每个隐向量替换成码本里离它最近的那条码字（于是变成一个整数下标），解码器再从这些下标重建图像。训练目标是最小化重建损失。来源：CS336 课件" />

<Theorem type="definition" name="VQ-VAE 的三步">

1. **编码**：把图像送进一个编码器，得到一格格的**连续向量**。
2. **量化（vector quantization）**：准备一个**码本（codebook）**，
   里面有比如说 8000 条**码字（code）** —— 这些是「原型向量」，
   大致对应各种典型的图块。
   **把每个连续向量，替换成码本里离它最近的那一条。**
   于是这个向量就变成了**一个整数下标**。
3. **解码 + 重建**：解码器拿着这些码，**试图把原图重建出来**。
   **训练 VQ-VAE 的方式就是最小化重建损失（reconstruction loss）。**

</Theorem>

用 Percy 的话说：**编码器先经过一个连续的阶段，然后被「四舍五入」到最近的那条码。**

<Callout type="note" title="补充：为什么这东西不好训（Percy 提到但没展开）">
Percy 说「因为这个过程不可导，所以还要加一些别的损失项，这次没时间展开」。补一句是什么意思：

**「取最近的那条码」这个操作是不可导的** —— 它是一个 argmin，
你把输入挪动一点点，输出的下标要么完全不变，要么突然跳到另一条码上。
梯度到这里就断了，反向传播传不回编码器。

标准的补救办法有两块：

- **直通估计（straight-through estimator）**：前向传播照常量化，
  反向传播时**假装量化那一步不存在**，把解码器的梯度原样复制给编码器。
- **额外的损失项**：一项把**码字拉向它负责的那些编码器输出**（更新码本），
  一项把**编码器输出拉向它选中的码字**（这一项叫 commitment loss，
  防止编码器输出到处乱跑、和码本对不上）。
</Callout>

<Callout type="tip" title="动手：量化这一步到底丢了什么">
下面这个演示把 VQ-VAE 中间那一步单独拎出来了。
平面上的每个小点是「编码器输出的一个连续向量」（这里画成二维好看），
菱形是码本里的码字，连线表示每个向量被替换成了哪一条码。

几件值得试的事：

- **把码本大小 K 从 2 拖到 128**：菱形变多，连线变短，**平均量化误差掉下去**。
  同时每个 token 的信息量从 1 bit 涨到 7 bit。
- **这就是第 1 讲那个取舍换了个皮**：词表越大，压缩得越好、丢的信息越少，
  但每个符号能分到的训练数据越少。
- 注意那些连线 —— **它们的长度就是被永久丢掉的信息。**
  这一点会直接解释 7.5 节说的「Chameleon 做不好 OCR」。

</Callout>

<L17Codebook />

### 7.3 Chameleon 的具体数字

<Theorem type="proposition" name="Chameleon 的图像分词器">

- **一张 512×512 的图像 → 1024 个 token。**
- **每个 token 来自一个大小为 8192 的词表。**

把这两个数拆开看：$1024 = 32 \times 32$，
所以图像被切成 32×32 的网格，**每个 token 覆盖 $512/32 = 16 \times 16$ 个像素**。

而 $8192 = 2^{13}$，所以**每个 token 值 13 个比特**。
整张图压完是 $1024 \times 13 = 13312$ 比特 = **1664 字节**，
而原图是 $512 \times 512 \times 3 = 786432$ 字节。
**压缩了大约 473 倍。**

</Theorem>

**接着他们训练了一个新的分词器。** 为什么？因为**数据现在长得和纯自然语言不一样了** ——
序列里混着文本 token 和图像 token，需要一个针对这种数据重新训的 BPE 分词器。

### 7.4 训练：这一段反而最简单

Percy 说到这里语气一转：**训练现在其实非常直白 —— 这就是普普通通的语言模型训练。**

**没有适配器，没有视觉编码器，没有阶段性冻结 —— 就只有一个语言模型。**
这正是这个方案吸引人的地方。

不过还是分两个阶段（其实即使是纯语言模型训练也通常有两段或更多）：

- **阶段一（占 80%）**：大规模、无监督。
  - **2.9 万亿**个纯文本 token
  - **1.5 万亿**个文本 / 图像 token
  - **4000 亿**个文本与图像**交错**的 token
- **阶段二（占 20%）**：**50% 用阶段一的数据，50% 用高质量数据。**

### 7.5 麻烦：训练不稳定

这套方案很优雅，但有几个问题。第一个是**训练不稳定**。

<Theorem type="proposition" name="不稳定是怎么来的">
**文本和图像虽然占据同一个空间，但它们的行为非常不一样。**
用 Percy 的话说：**把它们都叫做「离散 token」，并不能掩盖那底下住着一张图像的事实。**

具体地：

- **文本 token 的熵相对较低。** 你想想预测下一个词这件事 —— **大多数词是相当可预测的。**
- **图像 token 的熵非常高。** Percy 的原话：
  **「我根本不知道这个 token 具体是哪一种蓝色。」**

**结果**：这种混合训练导致**参数的范数不断增长**，进而**产生 logit 漂移和损失不稳定**。

**修补办法**：**QK norm** 和 **z-loss 正则化** —— 这两样都是用来控制范数增长的。
</Theorem>

<Callout type="note" title="补充：熵、QK norm 和 z-loss">
**熵（entropy）** 衡量的是「有多不确定」。
如果下一个 token 几乎可以确定，熵接近 0；如果 8192 个候选完全等可能，
熵就是 $\log_2 8192 = 13$ 比特（最大值）。
说「图像 token 熵高」的意思就是：模型对下一个图像 token 是什么，**预测得很差** ——
它可能是任何一种蓝，损失一直居高不下，梯度一直很大。

**QK norm**：在计算注意力分数之前，先对查询（Q）和键（K）做一次归一化。
这样点积的量级被限制住，注意力的 logit 不会随着训练越涨越大。
（第 3 讲讲过。）

**z-loss**：在损失里加一项，惩罚 softmax 的**归一化因子**
$\log \sum_i \exp(z_i)$ 偏离 0。
效果是把输出层的 logit 整体拉回一个合理的量级，防止它们无限漂移。

这两样合起来就是「控制范数增长」的具体手段。
</Callout>

### 7.6 Chameleon 的小结

Percy 说他甚至不打算展示结果，但他想强调这项工作，
**因为这里有一种优雅：它就是一个模型，用完全一样的方式对待所有模态。**

<Theorem type="proposition" name="Chameleon 小结">

1. **优雅** —— 只是对离散 token 做自回归建模，没有别的。
2. **但性能不够好。** **离散化确实会丢信息** ——
   想想 OCR：**如果你把很小的字离散化掉，你就再也读不出它了。**
3. **多模态混合训练本身很棘手。** Qwen 那边已经能看到一点苗头
   （要玩权重），**在这里被放大了。**

</Theorem>

<Callout type="insight" title="VQ-VAE 这条路为什么现在冷下去了">
Percy 补了一段历史：**VQ-VAE 曾经很流行**，很多人用它做**图像生成**。

**主要原因就是「Transformer 只能生成离散的东西」** ——
你要从 Transformer 生成图像，就只能先把数据全部搞成离散形式。

**但后来扩散模型出来了**，并且在生成上变得流行且可用。
**于是这一路做法就比过去冷门了。**
</Callout>

## 八、总结

### 8.1 Percy 的五条

<Theorem type="proposition" name="这一讲的五条结论">

1. **前沿模型现在被期待是多模态的** —— 甚至更强的说法是**原生多模态（natively multimodal）**、
   **omni 模型**。
2. **根本挑战是：怎么编码非文本模态？**
3. **理解和生成要的东西可能不一样**（语义 vs 更细的细节）。
4. **要平衡图像 / 视频（信息密度更低）和文本，否则训练会不稳。**
5. **目前的组合是：连续编码器 + Transformer + 扩散模型（负责生成）。**

</Theorem>

下面把每条展开成 Percy 课上说的话。

#### 关于第 1 条：前沿模型都说自己是原生多模态的

Gemini 出来、GPT 出来，都被宣传成**原生多模态**、能处理所有这些模态 ——
**而它们确实做到了。**

**但是关于它们是怎么建的，没有任何细节。**（这一点从第 1 讲开始就是这门课的老问题。）

Percy 的**推测**是：大概是**连续编码器**（因为你不想丢信息）
**加上扩散模型来做生成**。他明确说了这是他的猜测。

#### 关于第 2 条：根本挑战始终是「怎么编码非文本模态」

这一条其实就是整讲的骨架，不需要额外展开：Percy 说，
**当你在处理多模态时，最根本的挑战就是「你拿非文本模态怎么办」。**

这一讲给出的三种答案分别是：**CLIP / SigLIP 的对比学习**（用文字来教什么叫语义）、
**AnyRes / 动态分辨率**（用可变数量的块来保住细节），以及
**VQ-VAE 的码本量化**（干脆把它变成离散符号）。

#### 关于第 3 条：没有一个万能的编码器

这一条 Percy 讲得最细，也是这一讲两条主线的交汇点：

<Callout type="insight" title="理解和生成之间的不对称">
**理解那一侧**：看 CLIP —— 它**只关心分类**，所以只需要捕捉**高层语义**。
因此**那些向量可以相当小**，装着高层语义就够了。

**生成那一侧、以及精细理解那一侧**（比如 OCR）：**你需要非常细的细节。**

**这就是为什么扩散模型这么好用** —— 它能真正地在**高频信息**上反复打磨。

所以**不存在一个通用的编码器**：你为分类挑的编码器，和你为生成挑的编码器，
本来就该是不同的东西。
</Callout>

回过头看，这条线索在这一讲里出现过三次：

- **3.8 节**：CLIP 发现「直接预测说明文字的每个 token」反而更低效 ——
  因为对**分类**来说，精确建模文本序列并不重要。
- **5.2 节**：LLaVA-OneVision 为了 **OCR** 不得不放弃「缩放 + 中心裁剪」，
  发明 AnyRes 来保住分辨率。
- **7.5 节**：Chameleon 的离散化在 **OCR** 上丢信息 —— 小字被量化掉就读不出来了。

**同一件事的三次现身。**

#### 关于第 4 条：模态之间要称重

**处理多模态时，你必须仔细考虑怎么给它们加权。**
比如**视频的信息密度肯定低于文本**，所以**你不希望视频压过你的文本**。

（这正是 5.2 节那张 token 预算表、和 6.3 节那个平方根归一化在做的事。）

#### 关于第 5 条：连续编码器仍然是当前最好的选择

**至少目前看来，最好的做法是用连续编码器。**

Percy 说了一句挺有分量的话：

> **看起来即使是 CLIP —— 哪怕它已经五年了 —— 或者类似的想法，
> 仍然是捕捉图像语义的首选办法。**

**Transformer 还在那儿。而扩散模型（这一讲没讲）非常适合做生成。**

### 8.2 全课的收尾

讲到这里 Percy 就停了。他说：**这一讲没有配套作业**，
但如果你有兴趣，**他鼓励你自己去动手训一训这些模型**。

<Callout type="insight" title="这一讲兑现了第 1 讲留下的那张欠条（这段是本讲义的编排，不是老师原话）">
把这一讲和第 1 讲对着读，会看到一件挺整齐的事。

第 1 讲讲分词讲到最后，Percy 说他希望有一天不用再教分词器了，
但**任何替代方案都必须满足两条**：

1. **模型必须在序列的「块 / 抽象」上操作，而不是在最原始的单元上。**
   他当时补的那句是：**换成视频或 DNA 就非常明显 —— 单个字节的信噪比极低。**
2. **这些块必须是可变的**，要有自适应计算。

这一讲从头到尾就是这两条在图像上的实现：

- **第一条**：一个像素不是语义单元，所以得切 patch（3.5 节）、
  得用 CLIP 让文字来教它什么叫语义（3.7 节）、
  或者干脆学一个码本把它量化成离散符号（7.2 节）。
  **「块」在图像上的名字叫 patch 或者码字。**
- **第二条**：固定的 336×336 中心裁剪读不了文档（5.2 节），
  于是有了 AnyRes、有了动态分辨率、有了「一条公式 8 个 token、
  一整页长截图 11427 个 token」（6.2 节）。
  **这就是「块必须可变」。**

第 1 讲说这门课不讲多模态。最后一讲还是讲了 ——
而讲下来会发现，**它并不是另起一个话题，它是第一讲那个问题换了一种模态再问一遍**：

> **模型操作的「原子」应该是什么？**

从 BPE 到 patch，从词表大小到码本大小，从压缩率到 token 预算 ——
**换的是模态，不换的是那个取舍。**
</Callout>
