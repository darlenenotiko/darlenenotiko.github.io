## 一、这门课为什么存在

Percy 说，两年前开这门课时看到的问题是：**研究者正在和底层技术脱节。**

这个脱节有一条清晰的时间线：

- **2016 年前后**：AI 研究者自己实现、自己训练模型。
- **2018 年前后**：研究者下载别人预训练好的模型（比如 BERT），然后在自己的任务上微调。
- **今天**：研究者直接调 API 模型（GPT / Claude / Gemini），写 prompt 就能把事情做了。

他强调，**用 prompt 没有任何不对**，用它能做出很了不起的东西，往上走抽象层次总体上是好事。
问题在于两点：

1. **这些抽象是漏的（leaky）。** 这里的意思是：一个好的抽象应该让你不用关心底层就能把事做对 ——
   编程语言和操作系统基本做到了这一点。但 prompt 这层抽象做不到。你肯定遇到过这种情况：
   你想让模型做某件事，它就是做不到，而你**没有任何补救手段** —— 你能改的只有 prompt，
   底下发生了什么、为什么不行，你看不见也够不着。
2. **还有大量基础研究要做，而做这些必须把整个技术栈拆开。** 如果你只会 prompt 一个模型，
   你能考虑的设计空间被极大地限制住了。

所以结论是：**要做基础研究，就必须完整理解这项技术是怎么运作的。**
而这门课获得理解的方式是**动手建造** —— 这就是所谓「understanding via building」。

但这里有一个麻烦。

## 二、麻烦：语言模型已经工业化了

<Figure src="/courses/cs336/industrialisation.jpg" caption="课件里用的配图。图片来源：Wikimedia Commons" />

**前沿模型非常贵。**

- 2023 年，训练 GPT-4 据说花了 1 亿美元。而那已经是三年前的事了；
  Percy 说现在的成本量级大概是 10 亿美元，不过这是推测。
- 各大实验室在建的 GPU 集群规模极大。2025 年 xAI 为训练 Grok 建了一个 23 万张 GPU 的集群。

**而且这些模型是怎么建的，没有公开细节。** 早在 2023 年，GPT-4 的技术报告里就明确写了：
出于竞争格局和安全方面的考虑，不会公开模型是如何构建的。

<Figure src="/courses/cs336/gpt4-no-details.png" caption="GPT-4 技术报告里那段明说「不公开细节」的话。" />

所以前沿模型对我们来说是够不着的。那退一步：**我们可以建小模型**（比如参数量在 10 亿以内），
课上也确实会建。但要记住一件事 —— **小模型未必能代表真正的前沿模型**。
Percy 举了两个具体例子说明为什么。

### 例子一：算力花在哪里，会随规模变化

这个例子来自 2021 年前后。后面的课会花不少时间讲 FLOPs 计数（数浮点运算次数）、
看算力到底花在了哪儿。这里先看结论：

- 在**小规模**下，MLP 层消耗的 FLOPs 占比大约是 **44%**（也就是说另外一大半花在注意力等地方）。
- 规模放大到 **1750 亿参数**时，MLP 层的占比涨到了 **80%**。

<Figure src="/courses/cs336/roller-flops.png" caption="MLP 层 FLOPs 占比随模型规模的变化。来源：Stephen Roller" />

这意味着：**大规模下值得优化的东西，和小规模下值得优化的东西不是同一批。**
如果你在小规模上围绕注意力做了一堆优化，到大规模未必还有同样的收益 ——
因为那时注意力已经不是花钱的大头了。

### 例子二：有些能力要到一定规模才出现

第二个例子是**能力随规模涌现（emergence）**。同样是比较早的工作，但现象很清楚：
在零样本或少样本设定下测各种任务，小模型基本上**什么都不会** ——
曲线贴着地面，看上去毫无起色；只有当规模跨过某个临界点之后，性能才突然大幅提升。

<Figure src="/courses/cs336/wei-emergence-plot.png" caption="多个任务上的能力随规模涌现。来源：Wei et al., 2022" />

所以如果你只在小规模上工作，有些现象你**根本观察不到**。

## 三、那这门课能学到什么是能迁移的？

上面两个例子听起来有点让人泄气，但没关系。关键问题是：**哪些知识是能从小规模迁移到大规模的？**
Percy 把知识分成三类：

<Theorem type="definition" name="三类知识">

1. **机制（mechanics）**：东西是怎么运作的 —— Transformer 是什么、模型并行怎么工作。
2. **心态（mindset）**：怎么去着手建一个语言模型 —— 把硬件榨到极致、认真对待 scaling。
3. **直觉（intuitions）**：哪些数据决策和建模决策能带来好的准确率。

</Theorem>

这门课的定位是：

- **机制和心态能教好，而且它们确实能迁移到大规模。** 课上会反复强调：任何东西都要做性能分析（profile）
  和基准测试（benchmark），围绕效率来优化。这些习惯在什么规模下都成立。
- **直觉只能部分地教。** 哪些建模改动、哪些数据决策管用，这类知识**不一定跨规模迁移**。
  想获得这部分，你得去一个真的能做大规模实验的地方。

### 有些设计决策目前就是没法解释

顺着「直觉」这个话题，Percy 特别指出一件事：**有些设计决策就是没有道理可讲，纯粹来自实验。**

对比一下：机制那一类，你可以从构造上直接看出来并行和 kernel 为什么能加速 ——
它是可以推导的。但「哪个建模改动有效」这类直觉，你只能跑实验。

他举的例子是 Noam Shazeer 那篇提出 **SwiGLU** 激活函数的论文（后面的课会讲 SwiGLU）。
这篇论文的结论部分，作者非常诚实地写了这么一句：

> 我们无法给出解释。我们把这些架构的成功归因于神明的恩赐（divine benevolence）。

<Figure src="/courses/cs336/divine-benevolence.png" caption="SwiGLU 论文结论里那句话。" />

这句话是字面意义上的「我们不知道为什么它有效」。这类东西只能靠经验积累。

## 四、苦涩的教训，以及一个公式

最后一点铺垫是关于「苦涩的教训」（the bitter lesson）。这个说法流传很广，
但 Percy 认为存在一个常见的误读：

- **错误的解读**：规模就是一切，算法不重要。
- **正确的解读**：**能够 scale 的算法才是重要的算法。**

可以用一个很简单的式子来理解这件事：

$$
\text{准确率} \;=\; \text{效率} \times \text{资源}
$$

这里**效率是「产出 ÷ 投入」，资源是「投入」**。两者相乘得到最终能达到的准确率。

<Callout type="insight" title="为什么规模越大，效率越要紧">
在小规模实验里，如果你的程序慢一倍，你无非就是多等一倍时间，晚点回来看结果。

但在大规模下，同样的浪费可能意味着上亿美元。这时候哪怕只提升 5%，也是很大的一件事。
所以**规模越大，效率越是关键**，因为你根本浪费不起。
</Callout>

而且经验上，效率的提升是实打实的：OpenAI 2020 年的一篇论文显示，
在 ImageNet 上，2012 年到 2019 年之间**算法效率提升了 44 倍**。
硬件当然也在变好，但这 44 倍是算法层面的改进。两者相乘，才有了效率和准确率的巨大跃升。

于是整门课的问题可以这样表述：

<Theorem type="proposition" name="本课的核心问题">
给定一定的**数据预算和算力预算**，你能建出的最好的模型是什么？

换句话说：**最大化效率。**
</Theorem>

一点补充：在预训练场景下，我们主要谈**算力预算**，因为通常假设数据比算力多得多 ——
你是被算力卡住的。但如果你处在数据受限的场景，或者你恰好囤了非常多的 B200，
那你可能反过来是被数据卡住的。这个主题会贯穿整门课。

## 五、语言模型简史

在进技术细节之前，先把语言模型放在历史里看一遍。

### 前神经网络时代（2010 年代以前）

- **Shannon（1950）** 用语言模型来测量英语的熵。这是最早的语言模型工作之一。
- **N-gram 语言模型**（如 Brants et al., 2007）长期被用在**机器翻译和语音识别系统**里。
  注意它们不是整个系统，而是系统的一个重要部件 —— 负责保证生成出来的文本是**流畅**的。

### 神经网络的零件（2010 年代）

现代语言模型的血统来自神经网络架构。这十年里攒下了一堆关键零件：

- **LSTM**（1997）：长短期记忆网络，出现在 90 年代。
- **第一个神经语言模型**（Bengio et al., 2003）：注意这个**不是 LSTM**，
  它就是一个前馈网络，只看一个很小的上下文窗口。
- **序列到序列建模（seq2seq）**（2014）：为机器翻译提出，
  它相当大胆地主张 —— 可以把**一整个句子压缩成一个向量**。
- **Adam 优化器**（2014）。
- **注意力机制（attention）**（Bahdanau et al., 2015）：也是为机器翻译开发的。
- **Transformer 架构**（2017）：建立在注意力之上，同样最初是为机器翻译开发的。
- **混合专家（MoE）**（2017）。
- **模型并行**（GPipe 2018、ZeRO 2019、Megatron-LM 2019）。

可以看到，架构、系统、优化器三方面的想法都是在 2010 年代成型的。

### 早期基础模型（2010 年代末）

到 2010 年代末事情开始变得有意思：

- **ELMo**（2018）：用 LSTM 做预训练，再微调，下游任务显著变好。
- **BERT**（2018）：换成用 Transformer 做预训练，再微调。
  这两个的范式都是「拿一个预训练模型 → 在下游任务（比如问答）上微调」，提升非常大。
- **Google 的 T5**（2019，110 亿参数）：把所有任务都表述成「文本进、文本出」。
  Percy 说这篇其实**预示了后来「prompt 进、response 出」的那种视角**。

### 拥抱 scaling

真正把闸门打开的是 OpenAI：

- **GPT**（2018 左右）→ **GPT-2**（2019，15 亿参数）：能生成流畅文本，出现零样本能力的最初迹象。
- **Scaling laws**（Kaplan et al., 2020）：他们真正想清楚并接受了 scaling law 这套东西 ——
  这给了 scaling 一个可预测性，也是后面能放心训大模型的前提。
- **GPT-3**（2020，1750 亿参数）：比当时最大的模型还大 10 倍以上，
  展示出了**上下文学习（in-context learning）**这种涌现行为。

  <Callout type="note" title="补充：上下文学习是什么">
  指**不改动模型任何参数**，只在 prompt 里放几个示例，模型就能照着做新任务。
  比如你在 prompt 里写「英文：cat → 法文：chat；英文：dog → 法文：chien；英文：bird →」，
  模型接着输出 `oiseau`。

  它之所以被叫做涌现行为，是因为小模型做不到这件事 ——
  你给它示例，它不会照着模式走。到了 GPT-3 这个规模，这个能力才出现。
  </Callout>
- **Google 的 PaLM**（2022，5400 亿参数）：规模极大，但事后看是**训练不足的**（undertrained）——
  参数堆得多，喂的数据不够。
- **DeepMind 的 Chinchilla**（2022，700 亿参数）：搞清楚了**计算最优（compute-optimal）**的 scaling law。
  Percy 提到一个细节：当时 DeepMind 还没有和 Google 整合在一起。

### 开放模型

GPT-3 出来之后，对很多人来说是一次警钟，于是有了一批复现尝试：

**早期尝试（试图复现 GPT-3）**

- **EleutherAI**：一个草根组织，做了开放数据集 **The Pile** 和模型 **GPT-J**。
  模型不大，因为他们没有多少算力。
- **Meta 的 OPT**（2022，1750 亿参数）：Percy 说一看参数量就知道是冲着复现 GPT-3 去的，
  但模型并不好，而且他们遇到了大量硬件问题。
- **Hugging Face / BigScience 的 BLOOM**（2022，1760 亿参数）：重点放在数据来源上。

这批模型总体上都不算强。但最近三年，开放模型的生态变化很大。

**可信的开放权重模型（放出权重 + 论文）**

- Meta 的 **Llama** 系列（Llama、Llama 2、Llama 3）—— 带的头。
- Mistral 的模型。
- 一大批中国模型：DeepSeek 系列、阿里的 Qwen 系列、月之暗面的 Kimi 系列、
  Z.ai 的 GLM 系列、MiniMax、小米的 MiMo。
  Percy 说这里肯定有遗漏 —— 字节跳动有，腾讯大概也有 —— 全部跟上太难了，
  但大家都听说过的 DeepSeek 和 Qwen 是在的。

这些模型现在**已经在逼近闭源模型**（GPT、Claude、Gemini 等）。
取决于你问谁、怎么测，它们可能稍微落后，也可能相当。但它们确实是非常可信、
在工业界被广泛使用的模型。

**开源模型（权重 + 论文 + 代码 + 数据）**

所以复现不了。但比什么都没有强太多。
</Callout>

### 「语言模型」这个词的含义一直在变

- **2018 年（BERT）**：语言模型是一个**你拿去微调**的东西。
- **2020 年（GPT-3）**：是一个**你拿去 prompt** 的东西。
- **2022 年（ChatGPT）**：是一个**你可以跟它对话**的东西。
- **2026 年（agent）**：是一个**能自主行动**的东西。

讲到最后一条时 Percy 想点开一个 agent 轨迹的例子，结果发现没网。他描述了那个例子：
点进去是一条极长的 agent 轨迹 —— 你给它一页文字，它就去完成一个相当复杂的编程任务。
他说这些模型的强度到今天仍然让他觉得不可思议，**我们今天对语言模型的要求，
大概超出了十年前任何人的想象**。

话虽如此，**基本面变化不大**：

- 仍然建立在 GPU 和 kernel 之上；

- 仍然是 Transformer 加注意力。

**变的是规格（specs）**：现在要求更长的上下文，这意味着**推理效率变得更加重要**。
所以对课程来说是好消息 —— 不用大改，只有 Tatsu 讲最新中国架构的那部分要更新。

### 6.1 基础（Assignment 1）

**目标**：能够训练出一个基本的语言模型。
**组成部分**：分词、模型架构、训练。这大致是前两周的内容。

#### 分词

分词要回答的问题是：**模型操作的「原子」是什么？**

形式上，分词器在**原始输入（字节）**和**整数序列（token）**之间做转换。
概念上，它是对文本的一次**切分**。

<Figure src="/courses/cs336/tokenized-example.png" caption="一段文本被切成 token 的样子。" />

课上会讲的是 **BPE（Byte-Pair Encoding，字节对编码）**，
它的直观做法是**把输入切成频繁出现的块**。

从效率的角度看，分词有两个好处：

1. **缩短序列长度**：原始字节流很长，分词把它压成少得多的 token（比如 1000 字节 → 约 250 个 token）。
2. **自适应计算（adaptive computation）** —— 这一点更微妙但可能更重要：
   有些地方虽然占了很多字节，但其实应该被压成一个 token；
   而输入里更罕见、更值得关注的部分，应该保留成多个 token，
   这样模型在那里就有更多的建模容量。

Percy 说他每年都希望自己不用再教分词了 —— **理想是有一种端到端、直接在字节上工作的架构**。
这方向有不少工作（ByT5、MegaByte、BLT、T-Free、H-Net 等），看起来有希望，
但**到目前为止都还没有被放大到前沿规模**。既然前沿模型仍然在用分词器，那就还是得教。

#### 模型架构

起点是原始的 Transformer（2017）。

<Figure src="/courses/cs336/transformer-architecture.png" caption="原始 Transformer 架构。来源：Vaswani et al., 2017" />

从那之后有很多改进和细化，Tatsu 后面会详细讲。这里先列出需要考虑的几类东西：

- **激活函数**：ReLU → SwiGLU。
- **位置编码**：正弦位置编码 → RoPE（旋转位置编码）。
- **归一化**：LayerNorm、RMSNorm、QK norm，以及前置归一化（pre-norm）与后置归一化（post-norm）之争。
- **注意力**：完整注意力、稀疏/局部注意力、分组查询注意力（GQA）、多头潜在注意力（MLA）。
  改这些的原因很直接：**注意力的计算量是序列长度 $n$ 的平方**，$n$ 一大就非常贵。
- **循环 / 状态空间模型 / 线性注意力**：Mamba、Gated DeltaNet 等。
  这几年更流行了，而且**通常是这类模型和注意力的某种混合效果比较好**。
- **MLP 部分**：原始 Transformer 里是稠密 MLP，现在**混合专家（MoE）已经成为构建
  计算高效 Transformer 的主流范式**。而且 MoE 不只是换个架构，训练方式也需要相应改变。
- **形状**：多少层、多少个注意力头、隐藏维度多大、多少个专家。

最后这条看起来最平淡 —— 不就是几个超参数吗？但 Percy 强调：
**在 scaling 和语言模型的语境下，这些数怎么设有非常大的影响。** 讲 scaling law 时会再回到这里。

#### 训练

模型架构定好之后，怎么把参数训出来？涉及这些决策：

- **损失函数**：默认是下一个 token 预测；但人们发现**一次预测多个 token（multi-token prediction）**
  对提升模型有帮助。
- **优化器**：以前普遍用 AdamW，现在 **Muon** 用得越来越多，
  尤其是一些最新的开放模型（比如 Kimi K2）。此外还有 SOAP 等。
- **初始化尺度**：听起来也很无聊，但**对大模型的训练稳定性影响巨大**。（如 Xavier 初始化、muP）
- **学习率调度**：余弦调度、WSD 等。
- **正则化**：dropout、权重衰减。
- **批大小**：涉及「临界批大小」的概念。
- **MoE 专属**：负载均衡（比如无辅助损失的做法）。

看到这个列表你可能想：这不就是一堆超参数吗，挨个试试就行了。
但实际上，**是否用有原则的方式仔细设定这些超参数，
决定了一次训练是直接炸掉变成废物，还是达到最先进水平。**

#### Assignment 1 的内容

- 实现 BPE 分词器
- 实现 Transformer、交叉熵损失、AdamW 优化器、完整训练循环
- 做**资源核算**，搞清楚你的 FLOPs 都花到哪去了
- 在 TinyStories 和 OpenWebText 上训练模型
- **排行榜**：在一张 B200 上给定 45 分钟，把 OpenWebText 的困惑度压到最低
  （熟悉 NanoGPT speedrun 的话，形式很像那个）

<Callout type="note" title="补充：困惑度（perplexity）是什么">
这个词后面会反复出现，Percy 在课上是直接用的，这里补一句定义。

语言模型对每个位置输出「下一个 token 是什么」的概率分布。
如果它给真实出现的那个 token 的概率越高，说明它预测得越好。
把整个测试集上每个位置的这个概率取对数、求平均、再取指数的倒数，就得到困惑度。

一个直观的读法：**困惑度约等于「模型在每一步平均在多少个候选之间犹豫」。**
困惑度 1 表示每一步都完全确定；困惑度 50 表示大致相当于在 50 个候选里猜。
**所以困惑度越低越好。**
</Callout>

做完 Assignment 1，你应该就能从零建出一个语言模型了。

<Callout type="insight" title="贯穿全课的三角权衡">
分词、建模、训练看起来是三块独立的东西，但 Percy 说，其实所有事情都是在平衡下面三点：

1. **表达力（expressivity）**：模型要能表示数据里复杂的依赖关系。
2. **稳定性（stability）**：要让参数范数和梯度范数待在一个**既不炸掉也不消失**的区间里。
   课上会花很多篇幅讲这个 —— 训练语言模型很大一部分工作其实就是在处理稳定性。
3. **效率（efficiency）**：在硬件上跑得快，训练和推理都是。

而且这三者会互相拉扯。比如很多架构决策的形式是「把某个东西投影到低维空间，这样能更快」——
接下来的问题立刻就是：**那它效果还一样好吗？** 做这类取舍就是这门课的主线。
</Callout>

### 6.2 系统（Assignment 2）

**目标**：把硬件（GPU 或 TPU）榨到极致。
**组成部分**：kernel、并行、推理。

#### 基础：资源核算

下一讲就开始讲这个。核心是**追踪 FLOPs 花在哪、内存花在哪**。

<Callout type="note" title="补充：FLOPs 是什么">
FLOP = floating point operation，一次浮点运算，也就是一次浮点数的加法或乘法。
FLOPs 是它的复数，指**运算的总次数**（是一个量，不是速度）。

注意区分带斜杠的 **FLOP/s**（每秒多少次），那才是速度。
所以「训练需要 $4.2\times10^{23}$ FLOPs」说的是工作量，
「B200 有 2.25 PFLOP/s」说的是这块硬件每秒能干多少活。
两者相除就是理论上的最短耗时。
</Callout>

会反复出现的一个公式是：训练一个 $N$ 参数的模型、喂 $D$ 个 token，
大约需要 $6ND$ 次浮点运算。比如 700 亿参数的模型训 1 万亿 token，
就是 $6 \times 7\times10^{10} \times 10^{12} = 4.2\times10^{23}$ FLOPs。课上会讲这个 6 是怎么来的。

然后是硬件。关于硬件，最需要记住的一件事是：

<Figure src="/courses/cs336/compute-memory.png" caption="内存和计算单元是分开的：数据必须搬过去、算完再搬回来。" />

**你的内存不在计算单元所在的地方。** 你必须把参数或激活值从内存搬到计算单元，
算完再搬回去。**这个搬运往往才是瓶颈。**

举个具体的数：B200（课上会有机会用）的 bf16 算力是每秒 2.25 千万亿次浮点运算（2.25 PFLOP/s），
内存带宽是每秒 8 TB。这两个数放在一起意味着什么？下一讲会拆开算，
看不同类型的算法各需要多长时间。

还会讲：

- **Roofline 分析**：判断一个计算是被**算力**卡住还是被**内存**卡住。
  Percy 说：**一般来说是被内存卡住。**
- **基准测试和性能分析**（用 nsight 之类的工具）：看实际到底发生了什么。

一台 DGX B200 长这样：8 张 GPU，通过 NVLink 互连。如果你有上千张 GPU，
那就是很多台这样的机器，机器之间用 InfiniBand 或以太网连起来。

<Figure src="/courses/cs336/dgx-b200-topology.png" caption="DGX B200 的系统拓扑。来源：NVIDIA DGX B200 用户指南" />

#### Kernel

**kernel 就是一个在 GPU 上运行的函数。**

当你用普通 PyTorch 时，每一个 PyTorch 原语操作实际上都对应着启动一个内置的 kernel ——
所以**你其实一直在用 kernel，只是没意识到**。

关键在于，对某些类型的计算，你可以自己写 kernel 让 GPU 跑得更快。
主要原则还是那一条：**组织计算，最小化数据移动。**

举个简单例子。假设你要先算 A 再算 B：

- **朴素做法**：从高带宽内存（HBM）读数据 → 算 A → 写回 HBM → 再读回来 → 算 B → 写回 HBM。
  数据来回搬了两趟。
- **融合（fusion）做法**：读一次 → 把 A 和 B 都算完 → 写回一次。

这就是**算子融合**，能省下大量时间。**分块（tiling）**是同一个思路的更复杂版本 ——
FlashAttention 就是典型例子。

除此之外，GPU 本身也变得复杂多了。课上不一定有时间讲全，但至少会让你见识到 GPU 的一些特性 ——
warp 分歧、内存合并访问、bank 冲突、占用率、批量异步内存传输等等 ——
让你体会到要把 GPU 榨干需要考虑哪些东西。写 kernel 会用 **Triton**（也存在 CUDA、CUTLASS、
ThunderKittens 等选择）。

#### 并行

如果你有上千张 GPU 呢？

原则不变，还是最小化数据移动，只不过**GPU 之间搬数据比 GPU 内部搬还要贵得多**。

会讲的内容：

- 用经典的**集合通信操作**（gather、reduce、all-reduce）来理解分布式训练。
- 大局是：模型参数、激活值、梯度、优化器状态这四样东西需要**分片（shard）**到多张 GPU 上；
  同时要把对的数据送到对的节点上去算，算完再写回来。整个是一套编排问题，
  怎么高效地编排就是这一单元的主题。
- 分片有多种方式：切数据、切模型、切模型的不同层、切序列、按专家切 ——
  分别对应**数据并行、张量并行、流水线并行、序列并行、专家并行**。课上会讲各自的取舍。

#### 推理

**目标**：给定一个 prompt 生成 token —— 这是真正使用模型所必需的。

Percy 强调推理的重要性在增长，而且**不只是聊天时用**：
强化学习需要它来做 rollout，test-time compute 需要它，生成合成数据需要它，评测也需要它。
所以推理是语言模型工作中非常关键的一环。

他也坦白说，这部分可能花不到他希望的时间，因为课程已经排得很满了；
课程组讨论过要不要让学生从零写推理，还没定。

推理分两个阶段：

<Figure src="/courses/cs336/prefill-decode.png" caption="推理的两个阶段：prefill 和 decode。" />

- **预填充（prefill）**：把 prompt 里的 token 一次性全部前向传播，建立键值对（KV）。
  这个过程和训练时非常像，**是算力受限（compute-bound）的**。
- **解码（decode）**：一次生成一个 token。这部分**很快就变成内存受限（memory-bound）**，
  这也是推理难做的原因。

加速解码的办法：

- **用更便宜的模型**：剪枝（pruning）、量化（quantization）、蒸馏（distillation）。
- **投机解码（speculative decoding）**：用一个便宜的「草稿」模型往前猜出好几个 token，
  然后用完整模型**并行地**给这些 token 打分，看猜得对不对。猜对了就一次性全部接受，
  比一个一个生成快得多。而且这是**精确解码** —— 结果和不用投机解码是一样的。
- **系统层面的优化**：为推理专门设计的融合 kernel、连续批处理（continuous batching）。

关于连续批处理，Percy 提到推理特有的一个麻烦：如果你在跑一个线上服务，
**请求是在不同时刻随机到达的**，你得想办法把它们组成批次。
而训练时批次是你自己预先定义好的，一切可预测得多。

- 实现分布式数据并行训练
- 实现优化器状态分片
- 对这些实现做基准测试和性能分析

Percy 说细节可能会变，因为助教们有重整系统单元的计划，所以作业可能和去年不太一样，
但覆盖的材料大致相同。

**推荐读物**：Google 的几位作者写的《How to Scale Your Model》。
它把 roofline 分析、Transformer 的数学、以及怎么从概念上思考 LLM 系统讲得很清楚。
唯一的问题是它出自 Google，所以以 TPU 为主，不过高层概念是相通的，而且现在也新增了讲 GPU 的一章。

### 6.3 Scaling laws（Assignment 3）

到这里你已经能训一个语言模型，也能靠优化 kernel 和并行让它跑得快。接下来是**怎么放大**。

设想这么一个情境：**你有 $10^{25}$ FLOPs 的算力（几千万美元），你要训一个什么样的模型？**

这件事很吓人，因为搞砸了就是一大笔钱打水漂。而且你**没法在这个尺度上做常规的超参数搜索**——
你只有一次训练机会。这正是大模型训练特有的、微调或小规模实验里遇不到的核心问题。

<Theorem type="definition" name="Scaling recipe（扩展配方）">
关键的概念转变是：**不要想着「我在训一个模型」，而要想「我在设计一个 scaling recipe」。**

一个 scaling recipe 是**从 FLOPs 预算到一组超参数的映射** —— 本质上就是一个 config 文件：
你给它 $10^{24}$ 或 $10^{25}$，它给你一整套超参数。
</Theorem>

有了 recipe 之后的做法是：

1. 在**较小的尺度**上跑一批实验（比如到 $10^{24}$ FLOPs 为止），得到各个尺度上的损失。
2. **拟合一条 scaling law**，用它预测这个 recipe 在**目标尺度**（比如 $10^{25}$ FLOPs）上的损失。

这个原语能带来两件事：

1. **用小尺度实验来优化面向大尺度的 scaling recipe。**
2. **在真正跑之前就预测出目标尺度上的损失。** Percy 半开玩笑地说，这条的用处是可以拿去融资 ——
   「我跑了小规模实验，我觉得我能训出一个 GPT-5 级别的模型，请给我很多钱。」

<Callout type="warning" title="scaling law 不是自然定律">
另一个常见误解是以为 scaling law 是某种自然规律，会自动成立。**不是的。**

**你必须把它「造」出来**，靠的是精心构造 scaling recipe。而且这个 recipe 必须**能外推**。
具体来说，它得回答：随着规模增加，学习率是保持不变、还是下降？批大小按什么速度增长？
这些都是 scaling recipe 要确定的东西。
</Callout>

为了得到可预测的 scaling law，一个必须考虑的问题是**怎么参数化模型，才能获得
「超参数迁移（hyperparameter transfer）」** —— 也就是让你在小尺度上用的超参数，
要么在大尺度上直接能用，要么是大尺度超参数的一个可预测的函数。（这方面的代表工作是 muP。）

道理很直接：如果每换一个尺度，最优学习率一会儿是 $10^{-5}$、一会儿是 $10^{-4}$，
那你到了大尺度根本没法猜中它。

<Callout type="insight" title="可预测性至少和最优性一样重要">
通常你会想「我要优化效率、我要调超参调到最优」。这当然要做。
但你**同样**需要可预测性 —— 这样到了大尺度才不会被意外打个措手不及。
</Callout>

#### 具体的 scaling law：模型大一点还是数据多一点？

真正会看的 scaling law 是比较经典的那些。问题是：
**给定 FLOPs 预算（$C = 6ND$），你应该训一个更大的模型（$N$ 大），
还是在更多 token 上训（$D$ 大）？**

这就是 Kaplan et al. 的经典计算最优 scaling law 和所谓 **Chinchilla scaling law** 要回答的。
基本做法是 **ISOFLOP 曲线**：

1. 对每一个 FLOPs 预算（比如从 $6\times10^{18}$ 一直到 $3\times10^{21}$），
   在这个预算内扫过不同的模型大小，找出损失最小的那个。
2. 把每个预算下的最优点连起来拟合一条曲线，就能**从 FLOPs 预算预测出最优参数量**。

<Figure src="/courses/cs336/chinchilla-isoflop.png" wide caption="ISOFLOP 曲线：每条曲线是一个固定的 FLOPs 预算，横轴是模型大小，谷底就是该预算下的最优模型。" />

如果你运气好，这些点大致落在一条直线上。如果运气不好，点到处乱飞 ——
那就说明**你不该对自己的外推有任何信心**。

**结论（很粗糙的经验法则）**：$D = 20N$ —— **数据量大约是参数量的 20 倍**。
比如一个 700 亿参数的模型，应该在大约 1.4 万亿 token 上训练。
当然这个数会随数据集和架构变化。

**一个重要的注意事项**：这套计算最优的结论**没有考虑推理成本**。
现在很多模型故意训得远超计算最优的 token 量，就是因为**推理时你想要一个更小的模型**。

#### 一个正在发生的例子：Marin

Percy 提到 Marin 项目在做的一件有意思的事：**预注册（pre-register）结果**。

他们在不同算力预算下拟合了 scaling 图，然后据此对 $10^{22}$ FLOPs 做出预测。
讲这一课时那个模型**正在训**，去 Marin 网站可以跟进，最快当天晚上就能训完 ——
所以周三那节课他会回来汇报，看实际损失和预注册的预测符合得怎么样。

<Figure src="/courses/cs336/scaling-tweet.jpg" caption="Marin 的预注册 scaling 预测。来源：Percy Liang 的推文" />

这件事的意义在于：**对一个从来没训过的大模型，我们能不能提前预测它会有多好。**

#### Assignment 3 的内容

Percy 说这个作业「有意思」（说完自己笑了）。

- 课程定义一个**训练 API**：你给一组超参数，它返回一个损失值。
  实际上课程组事先离线训了一批模型、把结果缓存下来，所以用起来**看着像在真的训练**。
  这样做是因为没有足够算力让每个人真的去训。
- 你在 FLOPs 预算内**提交「训练作业」**，收集数据点。
- 用这些点**拟合 scaling law**，然后**外推**。
- 提交外推出来的超参数和损失预测，课程组评估你落得准不准。
- **排行榜**：给定 FLOPs 预算，损失最小。

用意是复现「你真有 1 亿美元预算、必须极其小心地花掉」的那种高压场景 —— 只不过这里代价很低。

### 6.4 数据（Assignment 4）

现在你会训模型、会让它跑得快、也会放大它了。**还缺什么？——你拿什么数据训？**

这就是数据单元。Percy 说这可能是**最重要的部分之一**，
因为**数据质量基本上决定了你的模型能有多好**。

换个说法：**你想让你的模型做什么？** 数据反映的就是这个。
你想让它会多种语言吗？擅长对话吗？能跑长时间的 agent 编程任务吗？

#### 评测

所以要先谈**评测**，因为评测定义了你希望模型具备哪些能力。

评测是一个相当有深度的话题，**不只是「在几个 benchmark 上跑一下」**。它至少分两类用途：

1. **内部评测**：用来指导模型开发。这里要紧的是**跨规模的平滑性**（还是那句话，我们要可预测）
   和**相对性能** —— 你不太在意绝对值，因为「在某个留出集上困惑度 1.2」本身没什么含义。
2. **外部评测**：用来衡量真实用例上的绝对质量，是报告给客户、审稿人的那种数字。
   这里要紧的是**生态效度（ecological validity）**，也就是测的东西是否真的对应现实使用场景。

Percy 说这两者有时会被混为一谈，但它们服务的目的完全不同。
**困惑度（perplexity）** 就是典型的内部指标 —— 直到今天它仍然是衡量模型内在质量的好办法，
而且不用操心 benchmark 那一套。

至于在什么数据上评测：**最好用互联网上没有的数据**，这样可以避免**污染（contamination）**
（即测试数据在训练时已经被模型见过）。

外部一侧则有更贴近真实使用的**高难度评测**：GPQA、HLE、SWE-Bench、Terminal-Bench 等。

最后一点：语言模型号称是通用的，**那就理应用一组非常多样的评测来测它**。
Percy 建议永远保留多个评测；你可以把它们平均成一个数，
但要清楚**那个平均值掩盖了很多不同的东西**。

#### 数据从哪来

**数据不会从天上掉下来，得主动去收集整理。**

Percy 说在课程和研究里，人们常常是「被给了一个数据集，然后在上面做事」。
但做语言模型、尤其要收集大规模数据集时，你必须自己去找：

- 从互联网爬来的网页
- 书（他说这在今天是有争议的）
- arXiv 论文
- GitHub 代码
- 等等

<Figure src="/courses/cs336/pile-chart.png" caption="The Pile（2021）的数据构成，可以看到语言模型数据集相当多样。来源：The Pile 论文" />

**法律问题**是绕不开的：

- 在受版权保护的数据上训练，能不能主张「合理使用（fair use）」？这是有争议的。
- 有时你可能必须**购买数据许可**（比如 Google 和 Reddit 的授权协议）。
- Percy 举了个具体的两难：**很多 GitHub 代码根本没有许可证**。那你怎么解释？
  是假定它是宽松许可，还是保守地假定它不是？

还有一个现实问题：**原始数据根本就不是文本。** 它是 HTML、是 PDF，代码是一整个目录结构。
这些都需要处理才能变成能拿来训练的文本。

#### 数据处理

处理分几步：

- **转换（transformation）**：把非文本（HTML / PDF）转成文本，抽出正文主体。
- **过滤（filtering）**：只留好的。Common Crawl 里随便一个网页很可能质量极差，
  你多半不想在上面训练。做法通常是训练分类器来判断质量和有害内容。
- **去重（deduplication）**：既省算力，也避免模型死记硬背。技术上用 Bloom filter 或 MinHash。
- **混合（data mixing）**：有多个来源时，每个来源该占多大比例？哪些要加权、哪些要降权？
- **改写 / 合成数据**：拿真实数据让语言模型重写，变成更接近下游任务的形式，
  或者就是更像百科条目的形式。这是当前很活跃的研究方向。

**数据按用途分三类**：

- **预训练数据**：量大且多样。
- **中期训练（mid-training）数据**：高质量的数据，通常放在预训练的**末尾**阶段。
  这里也包括**长上下文数据**，比如大型代码仓库或整本书。
- **后训练（post-training）数据**：监督微调用的，比如对话，
  或者**带工具调用的 agent 轨迹**。

#### Assignment 4 的内容

- 把 Common Crawl 的 HTML 转成文本
- 训练分类器来过滤质量和有害内容
- 用 MinHash 做去重
- **排行榜**：给定 token 预算，困惑度最低

Percy 的评价是：这个作业不见得「不好玩」，但确实是很多人所说的**脏活**。
不过这正是从零构建语言模型的重要一环 —— 得体验完整的过程。

### 6.5 对齐（Assignment 5）

到目前为止，我们都是在**完全监督**下训练模型：预测下一个（或接下来几个）token。

现在模型已经不错了，接下来可以用**弱监督（weak supervision）**进一步改进它。

**为什么用弱监督？** 因为**评判往往比生成容易**。
你不一定总能拿到「这个 prompt 的正确回答是什么」这样的数据，
但你往往能说清楚「什么样算好」。

<Theorem type="proposition" name="基本模板">

1. 让模型**生成**若干个回答。
2. 用**人、验证器（verifier）或语言模型裁判（LM judge）**给这些回答**打分**。
3. **更新模型**，让它更倾向于打分高的回答。

</Theorem>

具体算法：

- **PPO（近端策略优化）**：来自强化学习，InstructGPT 用的就是这一套。
- **DPO（直接偏好优化）**：针对偏好数据，更简单。
- **GRPO（组相对偏好优化）**：去掉了价值函数。

**挑战**：

- **RL 算法不稳定、难调参。** Percy 说在座有些人大概有第一手经验。
  他个人的偏好是**尽可能长时间地待在监督学习的框架里**，实在不得已才上 RL ——
  不过他也说，有些人出于某种原因就是喜欢做 RL。
- **大规模 RL 有大量系统挑战。** 要最大化吞吐量，你得同时有一个**推理服务器**和一个**训练服务器**：
  推理服务器负责生成 rollout；如果 RL 的环境还涉及**代码执行**，那就是一整套编排工作。
- **on-policy 与吞吐量的持续拉扯。** 如果某些 worker 落后了，你就会遇到 off-policy 的问题
  （用旧策略产生的数据去更新新策略）。于是你一直在「保持 on-policy」和「最大化吞吐量」
  之间来回权衡。Percy 形容这是「一团巨大而美妙的混乱」。

  <Callout type="note" title="补充：on-policy 和 off-policy">
  RL 里的「策略（policy）」就是当前这个模型本身 —— 它决定给定输入时生成什么。

  **on-policy** 指：用来更新模型的那批回答，正是**模型当前的样子**生成出来的。
  **off-policy** 指：那批回答是**模型早几步之前的样子**生成的，模型已经变了。

  为什么会变成 off-policy：生成回答（rollout）很慢。等推理服务器把回答生成完，
  训练那边可能已经更新过好几轮参数了。你当然可以每次都等它们同步，
  但那样 GPU 大量时间在空转，吞吐量就掉下去了。
  </Callout>

**Assignment 5** 的内容还在定。去年是实现 DPO 和 GRPO 并在某个数学 benchmark 上跑通，
今年想看看能不能在「贴近真实」这个维度上推得更远。

### 6.6 把五部分串起来：一切都是效率

<Theorem type="proposition" name="贯穿全课的主线">
**资源** = 数据 + 硬件（计算核心、内存、通信带宽）。

**问题** = 给定固定的资源，如何按照某个评测标准建出最好的模型？

效率可以是**数据效率**，也可以是**计算效率**。
</Theorem>

用这个视角回看，五个部分的设计决策都能统一起来：

- **系统**：显然就是计算效率。
- **分词**：直接用原始字节在概念上更优雅，但**以今天的模型架构来说，那样计算效率太低**。
  所以分词很大程度上是在提升计算效率。
- **模型架构**：很多改动的动机就是**减少内存或 FLOPs**（比如共享 KV cache、滑动窗口注意力）。
  而且其中不少是被**推理要更快**这个需求推动的。
- **数据过滤**：不想把算力浪费在一堆冗余的坏数据上做梯度更新。
  就算坏数据不直接伤害模型，在**固定算力预算**下，
  **花在坏数据上的时间就是从好数据那里抢来的**。
- **Scaling law**：本质上就是**在小得多的模型上做有效的超参数调优**。

Percy 补充说：今天我们是**算力受限**的，所以设计决策都反映着「把给定硬件榨干」。
**明天我们可能变成数据受限的**，那时候什么设计决策划算，算法会变。
但**思考效率**这个心态是这门课真正想教给你的东西。

### 7.1 分词器要解决什么问题

从两端说起：

- **一端是原始文本。** 文本是什么？是 **Unicode 字符串**。
- **另一端是语言模型。** 语言模型在 **token 序列**上定义一个概率分布，
  而 token 通常用**整数下标**表示。

比如字符串是 `"Hello, 🌍! 你好!"`，而模型那边看到的是一串整数，形如 `[15496, 11, 995, 0]`。

所以我们需要两个过程：

- **编码（encode）**：把字符串变成 token；
- **解码（decode）**：把 token 变回字符串。

<Theorem type="definition" name="分词器（tokenizer）">
分词器就是一个实现了 `encode` 和 `decode` 两个方法的东西。
关键性质是它必须能**往返（round trip）**：先编码再解码，要能拿回一模一样的原字符串。

```python
class Tokenizer(ABC):
    """分词器的抽象接口。"""
    def encode(self, string: str) -> list[int]:
        raise NotImplementedError

    def decode(self, indices: list[int]) -> str:
        raise NotImplementedError
```
</Theorem>

Percy 强调：**如果你实现的分词器往返回不去，那就是有 bug。**

### 7.2 先看看真实的分词器长什么样

课上推荐了一个交互网站（tiktokenizer.vercel.app）可以随便玩不同的分词器。
Percy 说，看完这些观察你就能体会到**分词器为什么这么讨厌、为什么大家想干掉它**：

<Callout type="warning" title="三个让人别扭的观察">

1. **一个词和它前面的空格会被合成同一个 token。** 所以你看到的很多 token 其实长得像
   「空格 + 单词」，比如 `" world"`。这没什么错，但确实很奇怪。
2. **同一个词出现在开头和出现在句中，是完全不同的 token。**
   比如 `"hello hello"` 里，第一个 `hello` 和第二个 ` hello`
   对应的是**两个毫不相干的整数下标** —— 模型得分别学它们。
3. **数字是每隔几位切一个 token。** 有的分词器切得可预测，有的不可预测。
   有些分词器试图让每一位数字都成为一个 token，但那样 token 数量又会膨胀。
   这里存在取舍。

</Callout>

接着看真实的 GPT-5 分词器（OpenAI 的 tiktoken，编码名 `o200k_base`）：

```python
import tiktoken

tokenizer = tiktoken.get_encoding("o200k_base")
string = "Hello, 🌍! 你好!"

indices = tokenizer.encode(string)              # 变成一串整数
reconstructed_string = tokenizer.decode(indices) # 再变回来
assert string == reconstructed_string            # 必须往返成功
```

<Theorem type="definition" name="压缩率（compression ratio）">
**压缩率 = 字节数 ÷ token 数**，单位是「字节 / token」。

```python
def get_compression_ratio(string: str, indices: list[int]) -> float:
    """给定被切成 indices 的 string，返回平均每个 token 承载多少个 UTF-8 字节。"""
    num_bytes = len(bytes(string, encoding="utf-8"))
    num_tokens = len(indices)
    return num_bytes / num_tokens
```

对上面这个例子：字符串是 20 个字节，切出来 8 个 token，
所以压缩率是 $20 / 8 = 2.5$ 字节/token。
</Theorem>

**压缩率越大越好**，因为压缩率大意味着同一段文本切出来的**序列更短**；
而**注意力的计算量是序列长度的平方**，所以序列短很重要。

那能不能一味地提高压缩率？可以 —— **把词表变大**，token 的可能取值变多，
自然能把更长的片段压成一个 token。但代价是**稀疏性**：
词表里每个元素都被当成一个彼此独立的符号来对待，词表越大，每个符号见到的训练数据就越少。

所以这里也是取舍。今天的分词器，尤其是多语言分词器，
词表大小一般在 **10 万到 20 万**个不同 token 这个量级。

课件里还把 GPT-5 分词器的完整词表导出成了一个文件（每行一个 token）供翻看，
Percy 说为了赶时间课上跳过了 —— 但如果你想直观感受「词表里到底装着什么」，值得自己去翻一遍。

### 7.3 第一种尝试：字符分词

最直接的想法：Unicode 字符串就是一串 Unicode 字符，**那就一个字符一个 token**。

在 Python 里，字符和整数（**码点，code point**）之间用 `ord` 和 `chr` 互转：

```python
ord("a")       # 97
ord("🌍")      # 127757
chr(97)        # "a"
chr(127757)    # "🌍"
```

于是分词器可以这样写：

```python
class CharacterTokenizer(Tokenizer):
    """把字符串表示成 Unicode 码点的序列。"""
    def encode(self, string: str) -> list[int]:
        return list(map(ord, string))

    def decode(self, indices: list[int]) -> str:
        return "".join(map(chr, indices))
```

往返是没问题的。但它有两个问题：

<Callout type="warning" title="字符分词的两个问题">

1. **词表太大。** Unicode 字符大约有 **15 万个**，所以词表大小就是 15 万这个量级。
   Percy 说这个数「倒也不算离谱」，但是——
2. **很多字符极其罕见**（比如 🌍）。这意味着**词表被严重浪费了** ——
   大量下标几乎从来不被用到，却仍然各自占着一份模型参数。

</Callout>

而且**压缩率也不好**：大多数时候你要用相当多的 token 才能表示一段序列。

结论：**字符分词两头不讨好** —— 词表又大，压缩率又低。

### 7.4 第二种尝试：字节分词

换个思路：Unicode 字符串可以表示成**字节序列**，而字节的取值只有 0 到 255。

最常见的 Unicode 编码方式是 **UTF-8**。它的特点是**变长**：

```python
bytes("a", encoding="utf-8")    # b"a"                  —— 1 个字节
bytes("🌍", encoding="utf-8")   # b"\xf0\x9f\x8c\x8d"   —— 4 个字节
```

<Callout type="tip" title="动手看看 UTF-8">

这正是下面「字节分词序列会变长」的原因。
</Callout>

字节分词器：

```python
class ByteTokenizer(Tokenizer):
    """把字符串表示成字节的序列。"""
    def encode(self, string: str) -> list[int]:
        string_bytes = string.encode("utf-8")
        indices = list(map(int, string_bytes))
        return indices

    def decode(self, indices: list[int]) -> str:
        string_bytes = bytes(indices)
        string = string_bytes.decode("utf-8")
        return string
```

**好处**：词表非常小，**一个字节只有 256 种取值，所以词表大小就是 256**。

**坏处**：**压缩率恒等于 1**（每个 token 就是一个字节，定义上必然是 1 字节/token）。
这非常糟糕，意味着**序列会非常长**。

而 Transformer 的上下文长度是有限的（因为注意力是平方复杂度），
所以字节分词在这一点上完全不可接受。

到这里，两种方案都不行：**字符分词是「词表大 + 压缩率低」，字节分词是「词表小 + 压缩率最低」。**

### 7.5 第三种尝试：词分词

这是经典 NLP 里的做法：**把字符串切成词。**

比如用一个正则表达式来切：

```python
import regex

string = "I'll say supercalifragilisticexpialidocious!"
chunks = regex.findall(r"\w+|.", string)
```

这个正则 `\w+|.` 的意思是：**连续的字母数字字符（`\w+`）保持在一起当作一个词，
其余任何单个字符（`.`）各自成为一块**。

切出块之后，再建一个「块 → 整数」的映射表，就得到一个分词器了。

**好处**：**每个 token 都是有意义的** —— 因为词是人发明的，词往往有稳定的语义。
压缩率也相当不错。

**坏处**：

<Callout type="warning" title="词分词的三个问题">

1. **词表大小 = 训练数据里不同块的个数**，这个数可能非常大。
2. **实际上比「大」更糟：词表在原则上是无上界的。**
   因为测试时你会碰到训练时从没见过的词。
3. **未登录词（UNK）问题。** 过去的做法是给没见过的词分配一个特殊的 `UNK` token。
   Percy 说这**很丑**，而且会**搞乱困惑度的计算** ——
   模型可以通过把一切都判成 UNK 来作弊般地降低困惑度。

</Callout>

顺带看一下这个例子的具体情况：`"I'll say supercalifragilisticexpialidocious!"`
里那个超长单词会被整体切成一个 token —— 它压缩率很高，
但这个 token 大概率在真实训练语料里只出现过几次，模型学不到什么。

### 7.6 BPE：字节对编码

这才是实际使用的方案。

**历史**：

- BPE 由 Philip Gage 在 **1994 年**提出，最初是**数据压缩算法**，
  远早于语言模型成为热点。
- **2016 年**（Sennrich et al.）被引入 NLP，用于**神经机器翻译**。在那之前论文用的都是词分词。
- **第一个把 BPE 用于大语言模型的是 GPT-2**（2019）。

<Theorem type="definition" name="BPE 的基本思路">
**在原始文本上「训练」分词器，构造一个贴合这批数据的词表。**

得到的性质是：

- **常见的字节序列被表示成一个 token；**
- **罕见的序列被拆成多个 token。**

注意这同时解决了 UNK 问题：任何东西都能被分词 ——
罕见的东西只是被拆成更小的单元，而不是变成一个 `UNK`。
</Theorem>

**算法梗概**：从每个字节各自作为一个 token 开始，
然后**反复地把出现次数最多的那一对相邻 token 合并起来**。

#### 训练分词器

用 Percy 课上的例子：语料是 `"the cat in the hat"`，做 3 次合并。

```python
def train_bpe(string: str, num_merges: int) -> BPETokenizerParams:
    # 第一步：把字符串变成字节序列，每个字节就是一个初始 token
    indices = list(map(int, string.encode("utf-8")))
    merges: dict[tuple[int, int], int] = {}          # (下标1, 下标2) -> 合并后的新下标
    vocab: dict[int, bytes] = {x: bytes([x]) for x in range(256)}  # 下标 -> 它代表的字节串

    for i in range(num_merges):
        # 1. 数一遍每一对相邻 token 各出现了多少次
        counts = count_adjacent_pairs(indices)

        # 2. 找出出现次数最多的那一对
        pair = max(counts, key=counts.get)

        # 3. 合并这一对：造一个新 token，编号从 256 开始往后排
        new_index = 256 + i
        merges[pair] = new_index                     # 记下这条合并规则
        vocab[new_index] = vocab[pair[0]] + vocab[pair[1]]  # 新 token 代表两段字节接起来
        indices = merge(indices, pair, new_index)    # 把序列里所有这一对都换成新 token

    return BPETokenizerParams(vocab=vocab, merges=merges)


def count_adjacent_pairs(indices: list[int]) -> dict[tuple[int, int], int]:
    """统计 indices 里每一对相邻 token 出现的次数。"""
    counts = defaultdict(int)
    for index1, index2 in zip(indices, indices[1:]):
        counts[(index1, index2)] += 1
    return counts


def merge(indices: list[int], pair: tuple[int, int], new_index: int) -> list[int]:
    """返回 indices，但其中所有 pair 都被替换成 new_index。"""
    new_indices = []
    i = 0
    while i < len(indices):
        # 当前位置和下一个位置正好构成要合并的那一对
        if i + 1 < len(indices) and indices[i] == pair[0] and indices[i + 1] == pair[1]:
            new_indices.append(new_index)
            i += 2      # 两个 token 变一个，跳过两位
        else:
            new_indices.append(indices[i])
            i += 1
    return new_indices
```

跟着 `"the cat in the hat"` 走一遍 Percy 在课上单步执行的过程：

1. 先变成字节序列。
2. **统计相邻对**：发现 `(116, 104)` 出现了 **2 次** ——
   116 是 `t`，104 是 `h`，也就是 `"the"` 和 `"hat"` 里各有一个 `th`。
3. **找出现次数最多的那一对**：就是 `(116, 104)`。
   Percy 提到这里其实有几个并列的，直接取第一个就行。
4. **合并**：造一个新 token，编号 **256**，让它代表 `th` 这两个字节，加进词表。
   从此以后每看到 `th` 就用 256 表示。
5. **替换**：遍历序列，把每一处 `116, 104` 都换成 `256`。这个例子里有两处被替换掉了。
6. **迭代**：下一轮会找到 `(256, 101)`（也就是 `th` + `e` = `the`），合并成 **257**；
   再来一轮得到 **258**。

<Callout type="insight" title="每合并一次会发生什么">
**序列在变短，词表在变大。** 这两件事是同时发生的，正是压缩率和词表大小之间那个取舍的具体体现。

这个玩具例子跑完 3 次合并后，压缩率是 **1.5** 字节/token。
</Callout>

中间那张表是**当前的相邻对频次**，红色那一行就是这一步会被合并的那一对；

#### 用训练好的分词器编码新文本

现在有了 `merges`（合并表），怎么切一段新文本？

**概念上：把学到的那些合并，按学到的顺序，一条一条套用到新字符串上。**

```python
class BPETokenizer(Tokenizer):
    """给定一组 merges 和词表的 BPE 分词器。"""
    def __init__(self, params: BPETokenizerParams):
        self.params = params

    def encode(self, string: str) -> list[int]:
        indices = list(map(int, string.encode("utf-8")))
        # 注意：这个实现非常慢
        for pair, new_index in self.params.merges.items():
            indices = merge(indices, pair, new_index)
        return indices

    def decode(self, indices: list[int]) -> str:
        bytes_list = list(map(self.params.vocab.get, indices))
        string = b"".join(bytes_list).decode("utf-8")
        return string
```

用刚才在 `"the cat in the hat"` 上训出来的分词器去编码 `"the quick brown fox"`，
就得到一个 token 序列；解码回来能拿到原字符串。

（上面那个演示的「用它来编码」标签页做的就是这件事，可以单步看每条合并规则被套用的效果。）

#### Assignment 1 里要在这个基础上做的改进

Percy 说：**上面这个实现是能用的，是一个完整的 BPE 实现，但它极其慢。**
Assignment 1 要求你把它做快，具体方向：

1. **`encode` 目前遍历了所有的合并规则**，这非常慢 ——
   合并规则的数量基本上就是**词表大小减去 256**（比如词表 10 万就是约 10 万条）。
   你应该**只遍历那些真正相关的合并**，为此需要建一些索引结构。
2. **检测并保留特殊 token**（比如 `<|endoftext|>`）。概念上不深，但对做一个现代分词器很重要。
3. **预分词（pre-tokenization）**：为了简单，上面是把整个字符串一次性拿去分词。
   实际做法是**先把文本切成块，再对每一块分别应用分词器**，这样快得多。
   （GPT-2 的分词器就用了一个专门的正则来做这一步。）
4. **尽可能做快。** Percy 说，做到某个程度你会发现 Python 本身就是不快的；
   如果你想用自己喜欢的语言（Rust、C 之类）来实现，尽管去做。

### 7.7 分词这一单元的小结

<Theorem type="proposition" name="小结">

1. **分词器在字符串和 token（整数下标）之间做转换。**
2. **字符分词、字节分词、词分词各有各的严重缺陷**，都远远称不上好。
3. **BPE 是一个有效的启发式方法，而且是数据驱动的** —— 词表是从语料里学出来的。
4. **分词目前仍然是独立的一步。** 也许有一天能做到直接从字节端到端。

</Theorem>

Percy 重申他希望明年不用再教这个，但今年还是得教。不过他补充了一点很重要的判断：
**即使将来分词器被干掉了，取代它的方案也必须满足下面两条性质。**

<Callout type="insight" title="任何替代方案都必须满足的两条">

1. **模型（比如 Transformer）必须在序列的某种「块」或者说抽象上操作，而不是在最原始的单元上。**

   这一点在文本上还不太明显，**换成视频或 DNA 序列就非常明显了** ——
   单个字节或单个碱基的**信噪比极低**，你必须先做某种抽象，
   把它抬升到一个能在上面做建模的层次。

2. **这些块必须是可变的**，也就是要有**自适应计算**：
   **不是所有字节都该被同等对待。** 做不到这一点，方案就是次优的。

</Callout>
