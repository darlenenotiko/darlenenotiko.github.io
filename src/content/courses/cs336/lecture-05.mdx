## 一、为什么做语言模型的人必须懂 GPU

### 1.1 算力是这个领域的货币

从第一讲和 scaling law 的工作里我们已经知道：**算力多了，语言模型就更好**，
而且好得相当可预测。所以这几年整个领域的游戏就是「怎么把更多算力投进训练」，
最近一年多则变成了「怎么把更多算力投进推理」。

能推动进展的东西有三类，而且它们是可以叠加的：

- **更快的硬件**；
- **更好的利用率**（同样的硬件，跑出更多有效算力）；
- **更好的并行**（更多芯片一起干活）。

Tatsu 强调，就算你不是做系统的 —— 这门课里大部分人都不是 —— 你在想架构、
或者只是想搞清楚某件事为什么会这样的时候，**你也需要知道你的模型在硬件上是怎么执行的**。
回到 Percy 第一讲的话：scaling 很大一部分就是**把资源用好**，
而不理解系统，你永远不可能把资源用好。

### 1.2 时钟频率那条路已经走到头了

如果这是 1990 年代，我们讨论的会是**时钟频率**：
让 CPU 串行地执行指令，然后想办法让指令执行得越来越快。

这套办法叫 **Dennard 缩放（Dennard scaling）** —— 晶体管越做越小，时钟频率就越来越高。
它在挺长一段时间里都成立，大约到 **2000 年代**开始失效。

<Callout type="note" title="补充：Dennard 缩放和摩尔定律不是一回事">
两个名字经常被混着说，但讲的是不同的事：

- **摩尔定律**说的是**单位面积上的晶体管数量**大约每两年翻一番。
  这条到今天还在以某种速度继续。
- **Dennard 缩放**说的是：晶体管变小的同时，单位面积的功耗保持不变，
  于是**时钟频率可以跟着提上去**。这条在 2000 年代前后失效了。

Tatsu 特意点了一句这中间的区别：**晶体管数量确实还在涨，但晶体管变小并不意味着它跑得更快**。
时钟频率上不去有很实在的物理原因（主要是功耗密度），
所以「把时钟调快、把指令执行得更快」这条老路走不通了。
</Callout>

那在一个「不能再靠频率变快」的世界里，怎么继续给语言模型喂算力？
答案就是 GPU，以及更一般的**并行 scaling**：不再靠串行地变快，
而是**横向扩展** —— 让更多东西同时执行你的指令。

### 1.3 十年 1000 倍

<Figure src="/courses/cs336/l05-parallel-scaling.png" wide caption="单芯片推理性能十年涨了 1000 倍以上，以及这 1000 倍分别来自哪里。来源：Bill Dally 的 HotChips 主题演讲，转引自 CS336 第 5 讲课件" />

这张图 Tatsu 认为一张就概括了并行 scaling 的全貌：早期的 K20X、M40 时代，
每秒能做的运算量按今天的标准说得上「体面但可怜」；到 P100、V100 前后曲线突然起飞，
**逐年拿到超指数的增长**。

右边把这 1000 倍拆开了，值得记住每一项的量级：

- **数字表示**（FP32 → FP16 → Int8，中间还有 TF32、BF16）：约 **16 倍**；
- **复杂指令**（DP4、HMMA、IMMA 这些一条指令干很多活的指令）：约 **12.5 倍**；
- **工艺**（28nm → 16nm → 7nm → 5nm）：约 **2.5 倍**；
- **稀疏性**：约 **2 倍**。

注意这里最大的一块是**数字表示**，不是工艺 —— 这一点后面讲低精度时会反复回来。
把这些乘起来再加上模型效率的改进，就是十年 1000 倍以上。

结论只有一句：**没有 GPU 的 scaling，就没有语言模型的 scaling。**

图里还标出了几个关键节点：**2017 年 V100 时代开始有 tensor core**（张量核心），
再往后是**结构化稀疏**和 **FP8** 这些更低的数字格式在继续推高峰值算力。

## 二、GPU 和 CPU 的分工完全不同

<Figure src="/courses/cs336/l05-cpu-vs-gpu.png" wide caption="左：CPU 有一个大的控制单元、大缓存和少量 ALU；GPU 反过来 —— 一大片小小的 ALU，控制和缓存都很小。右：同样四个任务，CPU 一个接一个很快做完（低延迟），GPU 每个都拖很久但重叠着做（高吞吐），白色是在等数据。来源：NVIDIA 开发者博客，转引自 CS336 第 5 讲课件" />

先把设计哲学讲清楚：

- **CPU 为「少量线程、每个都很快」而设计。** 它要处理复杂的分支、复杂的条件判断、
  复杂的控制流。所以 CPU 有**很大的控制单元**和**少量的 ALU**（算术逻辑单元）。
  它优化的是**延迟**：从拿到一条指令到完成它的时间要非常短。
- **GPU 为「大量线程、总吞吐量最大」而设计。** 单个任务派下去可能要很久才完成 ——
  可能算一会儿、停下、去干别的、再回来 —— 但**所有任务加起来的吞吐量非常大**。
  支撑这一点的是**成百上千个轻量计算单元**，它们全部并行执行。
  对应的代价是：**对分支的支持少得多**（控制单元和缓存都小）。

<Callout type="note" title="补充：延迟和吞吐量的区别">
这两个词在这一讲里会反复出现，先把它们分清楚：

- **延迟（latency）**：一件事从开始到完成花了多久。
- **吞吐量（throughput）**：单位时间里一共完成了多少件事。

两者可以完全反向。假设有 1000 个互不相干的小任务：

- 一个「低延迟」的处理器每个任务花 1 微秒，串行做完要 1000 微秒。
- 一个「高吞吐」的处理器每个任务要花 10 微秒，但能同时做 1000 个，
  总共只花 10 微秒 —— 单个任务慢了 10 倍，总时间快了 100 倍。

训练语言模型就是后一种情形：我们有海量彼此独立的乘加要做，
**没人关心某一个乘法什么时候完成**，只关心一秒钟能做完多少个。
</Callout>

## 三、GPU 的解剖

Tatsu 说他不希望你把 GPU 想成 `nvidia-smi` 打印出来的一段文字，
而是希望你能**在脑子里看见这块硬件长什么样**。

### 3.1 执行单元：SM 和 SP

<Figure src="/courses/cs336/l05-sm-anatomy.png" wide caption="左：一个 SM 的内部结构，可以看到寄存器堆、FP32/INT32 单元和 TENSOR CORE。右：完整的 GA100 芯片，那些绿色小块每一个都是一个 SM。来源：CS336 第 5 讲课件" />

GPU 的**基本单位是 SM（streaming multiprocessor，流式多处理器）**。
一个 SM 大致相当于一个「核」：它是一个独立的计算单元，
内部有一堆用来加速计算的子部件，也能访问几种不同的内存。

SM 内部再往下分是 **SP（streaming processor，流式处理器）**，
它们负责并行地执行**线程（thread）**。

**GPU 上有非常多个 SM**，每一个都能被独立编程、独立执行不同的任务（block），
并且都能访问外部的全局内存。

<Callout type="note" title="补充：SM、SP、CUDA 核、tensor core 分别是什么">
这几个词层级关系是这样的：

- **SM**：调度和资源分配的单位。一个 block 一定跑在**一个** SM 上，
  一个 SM 有自己的共享内存和 L1 缓存。**这是你写 kernel 时真正需要在意的粒度。**
- **SP / CUDA 核**：SM 内部真正做一次浮点加法或乘法的电路。
  NVIDIA 市场材料里说的「CUDA 核数」就是 SP 的总数。
- **Tensor core**：SM 内部**专门做小矩阵乘法**的电路，和 SP 是两套东西。
  上面那张 SM 图里，右边那两个写着 TENSOR CORE 的大绿块就是它。

一组具体数字：**A100 有 108 个 SM、6912 个 CUDA 核**（108 × 64）、**432 个 tensor core**（108 × 4）。
Tatsu 课上说「A100 有 128 个 SM」，那是完整 GA100 芯片的 SM 数（课件那张图的标注就是
「GA100 Full GPU with 128 SMs」）—— 但实际卖给你的 A100 只启用了 **108** 个。
后面算波量化的时候用的是 108，这个数很关键。
</Callout>

### 3.2 内存层级：越靠近 SM 越快

Tatsu 说，这一讲会反复回到的**不是算力，而是内存**。现代硬件和语言模型优化，
基本上是被内存定义的。所以要搞清楚有哪几种内存、它们物理上在哪。

<Figure src="/courses/cs336/l05-memory-anatomy.png" wide caption="左上是 A100 上实测的各级内存访问延迟（单位是时钟周期）；右边是 GA100 的芯片照片，绿色是 SM，中间蓝色的大块是 L2 缓存，四周是 HBM 的接口。来源：CS336 第 5 讲课件" />

按从快到慢排：

| 内存 | 位置 | A100 实测延迟 |
| --- | --- | --- |
| 寄存器（register） | SM 内部，每个线程私有 | 最快 |
| 共享内存 / L1 缓存 | SM 内部 | 共享内存 23 / 19 周期（读 / 写），L1 缓存 33 周期 |
| L2 缓存 | 芯片上，所有 SM 共用 | 200 周期 |
| 全局内存（global memory / HBM） | 芯片外面那几块内存颗粒 | 290 周期 |

几个要记住的点：

- **全局内存比 L1 慢将近 9 倍**（290 / 33）。这是速度上的巨大差别。
- **你平时说的「显存」就是全局内存。** 当一块 H200 告诉你它有 141 GB 内存时，
  说的是全局内存，不是共享内存或者 L1 缓存。
- **物理位置决定速度。** 全局内存是芯片旁边那些独立的内存颗粒；
  L2 在芯片本体上；L1 在每个 SM 里面 —— 所以离计算单元越近，延迟越低。
- **SRAM（共享内存 / 缓存用的存储）比 DRAM（全局内存）贵大约 100 倍，但快大约 8 倍。**

<Callout type="note" title="补充：HBM、DRAM、SRAM 是什么">
- **DRAM（dynamic RAM）**：用一个电容存一位数据。电容会漏电，所以要周期性刷新。
  密度高、便宜，但慢。
- **SRAM（static RAM）**：用六个晶体管存一位数据。不用刷新，快得多，
  但**同样容量占的面积大得多**，而且**只要通着电就一直在耗电**。
- **HBM（high bandwidth memory，高带宽内存）**：把多层 DRAM 芯片堆起来、
  用极宽的总线接到 GPU 旁边。它本质还是 DRAM，只是通过「总线特别宽」拿到高带宽。
  H100 的 3.35 TB/s、B200 的 8 TB/s 说的都是 HBM 带宽。

所以这一讲里说「HBM」「全局内存」「DRAM」时，指的是同一个东西。
</Callout>

那既然共享内存这么好，为什么不干脆整块芯片都用共享内存？Tatsu 给了三条理由：

1. **贵**，贵好几百倍；
2. **耗电**：SRAM 要持续供电才能保住数据，能耗很高；
3. **必须物理上靠近**：信号传播本身就是个难题，做不了很大。

所以实用的加速器都是**分层的内存结构**，你必须尊重这个层级才能跑快。

<Callout type="insight" title="一个反例：全 SRAM 的芯片">
确实有人做过「整块芯片都是 SRAM」的设计 —— Groq 就是这样。
对**推理**这类特定负载它非常有用（Tatsu 说，如果你造的是这种加速器，
那你干脆把整个矩阵塞进去就完了，全程都很快）。

但代价是**极其昂贵**。所以对绝大多数加速器来说，你还是得面对这个内存层级。
</Callout>

<Callout type="note" title="课堂提问：共享内存和 L1 缓存到底差在哪">
这个问题被问了两三次，Tatsu 的回答是：

- **L1 缓存是硬件自己管的**。它自动保存最近访问过的数据，**你没法控制它**。
- **共享内存是可编程的**。你在 kernel 里显式地把东西放进去、拿出来。

物理上它们通常是同一块 SRAM，只是划分成两部分用途不同。

还有同学问：L2 比 L1 慢，只是因为物理距离吗？Tatsu 说物理距离是很大一部分原因 ——
它们**都是 SRAM**（这正是全局内存慢那么多的原因之一，因为全局内存是 DRAM），
但物理距离和中间的互连线路决定了 L2 更慢。

他补充了一句：**这不是天然如此的。** TPU 上对应 L2 的那一级就快得多，
因为他们在硅片上做了不同的设计取舍 —— 这也是 TPU 的卖点之一。
</Callout>

### 3.3 执行模型：线程、块、warp

硬件的图景有了，现在看软件和编程模型。GPU 的执行模型里有**三个角色**：

<Theorem type="definition" name="线程、块、warp">

- **线程（thread）**：真正干活的东西，并行运行。GPU 的线程遵循 **SIMT 模型**
  （single instruction, multiple threads，单指令多线程）：
  **所有线程执行的是完全相同的指令**，只是输入不同 ——
  所以它们能做不同的有用工作，但指令必须一致。
- **块（block）**：一组线程。block 之所以是个有意义的概念，
  是因为**一个 block 保证跑在同一个 SM 上**。而 SM 有共享内存 ——
  所以 block 是「能共用一块快内存」的那个单位。
- **Warp（线程束）**：**32 个连续编号的线程**总是一起执行。
  这是 GPU 内部的**调度单位**。

</Theorem>

<Figure src="/courses/cs336/l05-execution-model.png" wide caption="一个 CUDA 程序被切成 block，每个 block 分配到一个 SM 上；block 内部再切成每 32 个线程一组的 warp；SM 里的 warp 调度器决定下一个周期让哪个 warp 执行。来源：CS336 第 5 讲课件" />

Warp 的存在是为了**降低调度器的开销** —— 与其一个线程一个线程地决定谁跑，
不如 32 个一起决定。

<Callout type="note" title="课堂提问：「所有线程执行相同指令」，是 block 内还是 warp 内？">
有同学问了这个关键问题。Tatsu 的回答是：**是 warp 内。**

同学接着确认：所以调度器决定下一条指令发给哪个 warp？
Tatsu 说对 —— **调度器决定下一个执行的是哪个 warp、在哪个 SM 上**。

还有同学问：一个 block 里有多少个 warp？Tatsu 说他记不住，
而且**这个数和硬件有关**。（block 的大小是你在启动 kernel 时自己指定的，
再按 32 切成 warp。）
</Callout>

### 3.4 内存模型：谁能读到什么

设备端的代码（也就是跑在 GPU 上的代码）能访问这几类内存：

- **寄存器**：每个线程私有，最快最局部。典型用途是存**数组的起止地址**这类东西。
- **本地内存（local memory）**：每个线程私有，但在 SM 的范围内。
- **共享内存（shared memory）**：**block 内共享**。
  如果你有东西需要跨线程传递、或者会被多个线程反复用到，就放这里。
- **全局内存（global memory）**：所有 block 都能读写，但你得付等待数据从远处过来的延迟。
- **常量内存（constant memory）**：只读。Tatsu 说他在实际编程里没怎么见人用。
- **主机内存（host memory）**：机器上的 CPU 内存。如果你的数据大到 GPU 装不下，
  可以往这边卸载。

<Callout type="warning" title="这一节唯一必须记住的事">
**一旦你走出共享内存，速度就会掉下来。**

所以这一讲从头到尾的主线就是：**把线程组织好，尽量减少访问全局内存的次数。**
</Callout>

## 四、旁支：TPU 是什么样的

Tatsu 花了两页讲 TPU。原因有两个：TPU 本身在变得更常见；
更重要的是，**TPU 是 GPU 的另一条演化路线**，对比着看能看出哪些设计是必然的、哪些是可选的。

<Figure src="/courses/cs336/l05-gpu-tpu-map.png" wide caption="上：TPU 的核心结构（标量单元、向量单元 VPU、矩阵乘法单元 MXU、片上内存和 HBM）。中：GPU 概念到 TPU 概念的对照表。下：H100 和 TPU v5p 的具体数量对比。来源：《How to Scale Your Model》，转引自 CS336 第 5 讲课件" />

**高层看，TPU 和 GPU 非常像。** Tatsu 把这叫做**收敛演化**：
如果你想造一个高能效的机器学习加速器，你大概率会走到同一个地方 ——
**轻量的控制单元 + 又快又大的矩阵乘法单元 + 快内存**。

TPU 的结构是：

- **标量单元（scalar unit）**：有点像 CPU，负责把指令派发给 VPU 和 MXU；
- **向量单元（VPU）**：做逐元素的操作（比如激活函数），也负责把数据装进 MXU；
- **矩阵乘法单元（MXU）**：做矩阵乘法，是芯片 FLOP/s 的来源；
- **内存**也是分层的：慢的叫 **HBM**，快的片上内存叫 **SMEM / VMEM**。

对照表里每个 GPU 概念都能找到 TPU 的对应物，而且对应得相当精确。
比如 GPU 的 tensor core（矩阵乘法单元）几乎精确对应 TPU 的 MXU ——
**细究下去它们底层用的是同一种电路**，都叫**脉动阵列（systolic array）**：
数据从一边流进去、一边流出来，在流动过程中完成矩阵乘。

<Callout type="warning" title="一个非常容易搞混的命名冲突">
Tatsu 特意停下来强调了这件事：

- **TPU 把它的「流式多处理器」叫做 tensor core。**
- **GPU 把它的「矩阵乘法单元」叫做 tensor core。**

两个东西名字一模一样，只能靠上下文区分。
**说 TPU 的 tensor core，指的是一个处理器；说 GPU 的 tensor core，指的是一个矩阵乘法器。**

（另外有同学问，GPU 和 TPU 上典型的脉动阵列尺寸分别是多大。Tatsu 说这个他记不住，得回去查。）
</Callout>

真正的差别在**数量和尺寸**：

| | H100（GPU） | TPU v5p |
| --- | --- | --- |
| SM / TPU tensor core | **132** | **2** |
| Warp 调度器 / VPU | 528 | 8 |
| 矩阵乘法单元（GPU tensor core / MXU） | **528** | **8** |
| 片上快内存（SMEM / VMEM） | 32 MB | 128 MB |
| 寄存器 | 32 MB | 256 kB |

也就是说：**GPU 靠「小而多」的矩阵乘法单元，TPU 靠「大而少」的矩阵乘法单元**，
但两者的矩阵乘峰值算力是可比的。

**这个取舍的后果是灵活性。** 小而多意味着你能把它们编程成各种形状去干各种事；
而在 TPU 上你**被锁死在大矩阵乘法上**。

<Callout type="insight" title="一个具体后果：批大小扫不到 64 以下">
Tatsu 讲了他最近一篇论文里的真事：他们做批大小扫描，扫到 **64 就停了**。

为什么？因为 **TPU 的 tensor core 拒绝接受维度小于 64 的输入**。
硬件就是要求你喂大矩阵乘法进去。
</Callout>

至于 TPU 和 GPU 最大的差别，Tatsu 说其实**不在单颗芯片上，而在互连网络**——
因为芯片本身就是为了乘矩阵而生的，长得都差不多。
网络那部分留到讲并行的那几课。另外 TPU **没有 warp**，只有 block，
这在矩阵乘和非矩阵乘之间有不同的取舍。

**结论**：因为两者共享「内存层级 + 矩阵乘法单元」这个骨架，
**这一讲教的核心概念在 GPU 和 TPU 上是通用的**。

## 五、GPU 模型的长处，以及矩阵乘法的特权地位

GPU 之所以极其成功，Tatsu 归纳了三条：

1. **容易扩展**：想要更多吞吐量，就往上加 SM ——
   只要内存带宽跟得上，吞吐量就一直涨。
2. **意外地好写**：因为是 SIMT。就算你去写 PTX 这种接近机器码的东西也是可行的，
   因为**你不是在给每一个线程单独编程**，你说的是「这是我的指令、这是我的输入，
   把它跑在所有输入上」。这和函数式编程里的 map 是同一种感觉。
3. **线程是轻量的**：可以随时被停下、随时被启动。
   调度器发现某个 warp 卡住了（比如在等内存），可以立刻换另一个 warp 上来。
   **换入换出非常快**，这正是即使有任务停滞、总吞吐量仍然很高的原因。

### 从「拿 shader 硬凑矩阵乘」到 tensor core

早期的 NVIDIA GPU 只有**可编程着色器（programmable shader）**。
研究者发现，商品级图形硬件的这种大规模并行对科学计算非常有效，
于是**用 shader 硬凑出矩阵乘法** —— 甚至发现**换不同的渲染设置能让矩阵乘更快**。
Tatsu 说这是很酷的黑客活。

但从 **V100 开始，NVIDIA 直接给了你一块专门做矩阵乘法的硬件**：tensor core。
从那一刻起，**矩阵乘法成了机器学习里唯一被特殊优待的操作**。

<Figure src="/courses/cs336/l05-matmul-speedup.png" caption="矩阵乘法与非矩阵乘法浮点运算的峰值算力对比（注意纵轴是对数刻度）。P100 之前两条线是重合的；V100 引入 tensor core 之后彻底分开，到 H100 已经差了一个数量级以上。来源：CS336 第 5 讲课件" />

<Theorem type="proposition" name="矩阵乘法快 10 倍以上">
在现代 GPU 上，**矩阵乘法的吞吐量比其它任何浮点运算高 10 倍以上**。

这直接决定了架构设计：**任何一个想靠算力 scale 的机器学习架构，
里面一定得有矩阵乘法** —— 因为那是唯一能真正把算力吃满的路子。
</Theorem>

<Callout type="note" title="课堂提问：V100 之前就没有矩阵乘法单元吗？">
对，**tensor core 是 V100 才加的**。在那之前你得手动编程，
虽然靠那么多 SM 和 ALU 也能有不错的吞吐量，
但**一块专门的矩阵乘法电路彻底改变了局面**。
</Callout>

## 六、内存墙：算力比内存长得快

这是硬件部分的最后一块，也是后面所有优化的根源。

<Figure src="/courses/cs336/l05-memory-wall.png" wide caption="峰值算力（灰）、DRAM 带宽（绿）、互连带宽（蓝）二十年来的归一化增长。三条线的斜率完全不同。来源：AI and Memory Wall，转引自 CS336 第 5 讲课件" />

三条线，三个速度：

- **峰值算力**：20 年 **60000 倍**（每两年 3.0 倍）；
- **DRAM 带宽**：20 年 **100 倍**（每两年 1.6 倍）；
- **互连带宽**：20 年 **30 倍**（每两年 1.4 倍）。

这意味着什么？

**在图的左端**，也就是 GPU 编程的早期，你其实不太需要操心内存 ——
因为那时候算力也没比内存传输快多少。
**越往右走，算力和内存之间的差距就越大**。

所以：**这一讲讲的绝大部分优化都是内存优化。**
越往未来走，内存瓶颈和通信瓶颈只会越来越突出，
把硬件吃满需要付出的功夫也只会越来越多。

<Callout type="note" title="课堂提问：如果这个差距继续拉大，未来的芯片会怎么设计？">
Tatsu 说这个问题逼他讲了他本来不想讲的东西：**推理这边的硬件格局更疯狂。**

- 大约从去年开始，人们做一件叫 **prefill / decode 分离（disaggregation）** 的事：
  **预填充**阶段是重矩阵乘法的，放在一种芯片上；
  **解码**阶段更受内存带宽限制，放在另一种芯片上。
- 有的模型走得更远。Tatsu 提到 **StepFun 3**（一个中国的开源模型）
  甚至试着**把不同的层分到不同的加速器上** —— 注意力去一种加速器、
  MLP 去另一种加速器。

他的结论是：**这个差距越大，任何能把宝贵的快内存用好的花招就越值钱。**
这一点在推理上体现得特别明显，因为**推理比训练更受内存限制**。
</Callout>

<Theorem type="proposition" name="第一部分小结：GPU 的三句话心智模型">

1. **GPU 是大规模并行的** —— 同一条指令同时作用在很多个执行单元上。
2. **算力（尤其是矩阵乘法）增长得比内存快得多** —— 所以内存才是关键。
3. **正因为内存关键，一切都必须尊重内存层级** ——
   能放进共享内存的就别放全局内存。

</Theorem>

## 七、屋顶线模型：先搞清楚你被什么卡住

Tatsu 给第二部分定了一个目标：**看懂下面这张图。**

<Figure src="/courses/cs336/l05-matmul-mystery.png" wide caption="两个 N×N 方阵相乘，横轴是 N，纵轴是实测吞吐量（TF/s）。粉色是算术强度带来的上升趋势，橙色是分块与对齐造成的分层，绿色圈出的是周期性的塌陷。来源：Horace He，thonking.ai，转引自 CS336 第 5 讲课件" />

这就是一个再普通不过的方阵矩阵乘。你可能会想：矩阵越大活越多，吞吐量当然越高。
**但图里全是奇怪的花纹** —— 某些尺寸下矩阵乘会比邻近尺寸慢得多。为什么？
而且，怎么才能让自己待在这条曲线的**上沿**而不是下沿？

### 7.1 屋顶线

<Figure src="/courses/cs336/l05-roofline.png" caption="屋顶线模型：横轴是操作强度（FLOP/字节），纵轴是能达到的吞吐量，都是对数刻度。斜线段是内存受限区，水平段是算力受限区。来源：CS336 第 5 讲课件" />

上面那张怪图，形状上有点像经典的**屋顶线模型（roofline model）**：
一段斜线，然后一段水平线。

- **斜线段 = 内存受限（memory bound）**：不管给你多少算力，吞吐量都上不去，
  因为数据喂不过来。
- **水平段 = 算力受限（compute bound）**：每读一个字节能配上足够多的运算，
  计算单元被完全喂饱了。**过了这个点，再增加每单位工作量也没用**，
  因为矩阵乘法单元已经跑满。

<Theorem type="definition" name="算术强度与机器平衡点">

**算术强度（arithmetic intensity，也叫操作强度）**：

$$
I \;=\; \frac{\text{这段计算做了多少次浮点运算}}{\text{它在慢内存和计算单元之间搬了多少字节}}
\qquad(\text{单位：FLOP/字节})
$$

**机器平衡点（ridge point）** 是硬件自身的一个常数：

$$
I^{*} \;=\; \frac{\text{峰值算力（FLOP/s）}}{\text{内存带宽（字节/s）}}
$$

判据非常简单：

- 若 $I < I^{*}$ → **内存受限**，实际吞吐量约为 $I \times \text{带宽}$；
- 若 $I \ge I^{*}$ → **算力受限**，实际吞吐量约为峰值算力。

</Theorem>

代进具体数字看一眼这个门槛有多高：

- **A100**（BF16 峰值 312 TFLOP/s，带宽 2.039 TB/s）：$I^{*} \approx 153$ FLOP/字节；
- **H100**（BF16 峰值 989.5 TFLOP/s，带宽 3.35 TB/s）：$I^{*} \approx 295$ FLOP/字节；
- **B200**（BF16 峰值 2250 TFLOP/s，带宽 8 TB/s）：$I^{*} \approx 281$ FLOP/字节。

**每从内存里搬一个字节，你得配上将近 300 次浮点运算，才刚刚够把 H100 喂饱。**
这个门槛高得离谱 —— 绝大多数逐元素操作离它差了三个数量级。

<Callout type="warning" title="这一节唯一的问题">
**怎么才能不被内存卡住？**

答案就是把算术强度顶上去：**每读一个字节，多做一点运算。**
接下来六个技巧，除了第一个之外，全都是在做这件事。
</Callout>

## 八、六个让 GPU 跑快的技巧

Tatsu 列的六个是：

1. **控制分歧**（这个和内存无关）
2. **低精度计算**
3. **算子融合**
4. **重计算**
5. **内存合并访问**
6. **分块（tiling）**

第一个的味道和其余五个不一样，其余五个都在讲同一件事：**尽量少搬内存。**

### 8.1 控制分歧：为什么不要写 if

<Figure src="/courses/cs336/l05-control-divergence.png" wide caption="上：SIMT 模型，一条指令解码后广播给一整个 warp 的所有线程。下：一个 if-else 在 GPU 上的实际执行时间线 —— 两个分支被串行地走了一遍，另一半线程在各自的时段里空转。来源：CS336 第 5 讲课件" />

在 GPU 上，一个 warp 里的每个线程执行的是**同一条指令**。
那如果你在 GPU 代码里写了一个 `if` 会发生什么？

在 CPU 上很清楚：选一个分支，执行它，然后继续。
**在 GPU 上完全不是这样，而且有点反直觉**：

> 每个线程都会把 `if` 和 `else` **两边都走一遍**。
> 如果你不属于当前这个分支，你就把这次计算**掩掉（mask out）**，坐在那里空转。

所以执行时间线长这样：一部分线程走下面的分支、先执行，
上面那批等着；然后上面那批执行，下面那批等着。
**为了执行一个 `if`，你实际上把两条路都走了**，
而且总有一半左右的线程在干等。

这就叫**控制分歧（control divergence）**。它有好的一面 —— 简化了编程模型；
坏的一面是**你的 GPU 会出现大段什么都不干的空隙**。

<Callout type="tip" title="实践中怎么绕开">
这就是为什么 GPU 代码里**你很少看到 if**。

Tatsu 举的例子是 ReLU：你会发现 GPU 代码里往往是**乘以 0 或者乘一个掩码**，
而不是写「如果 x 小于 0 就把它置零」这样的条件更新。原因是：
**乘法是所有线程一起走完的一条指令**，而一个 `if` 可能要花两个时钟周期
（或者说走两趟操作）。
</Callout>

### 8.2 低精度：位数少了，要搬的字节就少了

这是 Tatsu 认为**最重要**的一条，也是 NVIDIA 在硬件上投入最多的方向。
回头看那张十年 1000 倍的图：**相当大一部分增长来自数字表示的变化** ——
FP32 → BF16 → Int8，一路把精度往下砍。

道理很直白：**位数少一半，要搬的字节就少一半**，内存瓶颈立刻缓解一截。

#### 一个能手算的例子：逐元素 ReLU

课上算的这个例子非常小，但把事情钉死了。对一个长度为 $n$ 的向量做
$x \leftarrow \max(0, x)$：

**FP32 的情况**：

- 内存访问：读 1 个元素、写 1 个元素，每个 4 字节 → 每次操作 **8 字节**；
- 运算：1 次比较，算 **1 次浮点运算**；
- 所以是 **8 字节 / FLOP**。

**FP16 的情况**：

- 每个元素 2 字节 → 每次操作 **4 字节**；
- 运算数不变；
- 所以是 **4 字节 / FLOP**。

<Callout type="warning" title="课件上这个数写反了，Tatsu 当场纠正过">
幻灯片上写的是「Intensity: 8 bytes / FLOP」。Tatsu 讲到这里自己停下来说：
**「抱歉，这是算术强度的倒数。」**

算术强度的定义是 **FLOP ÷ 字节**，所以正确的数是：

- FP32：$1/8 = 0.125$ FLOP/字节；
- FP16 / BF16：$1/4 = 0.25$ FLOP/字节；
- FP8：$1/2 = 0.5$ FLOP/字节。

拿去和 H100 的机器平衡点 295 FLOP/字节比一下 —— **差了大约 1000 倍**。
逐元素操作在任何硬件、任何精度下都是**彻底的内存受限**。
把精度减半只是让它跑快一倍，改变不了它的性质。这一点后面讲融合时还会回来。
</Callout>

#### 低精度在矩阵乘法里是怎么工作的

<Figure src="/courses/cs336/l05-tensorcore-mixed.png" wide caption="左：tensor core 的数据流 —— 16 位输入相乘得到全精度乘积，用 FP32 累加器求和，最后输出 FP32。右：哪些操作能用 16 位、哪些需要更高精度或更大范围。来源：NVIDIA 混合精度教程，转引自 CS336 第 5 讲课件" />

真实的做法比「全部换成低精度」复杂得多：

- **输入下转（downcast）成低精度**再相乘；
- **乘积用全精度保存**；
- **累加（把矩阵乘里的部分和加起来）用 FP32 做**；
- 结果可能以 FP32 输出。

Tatsu 说，低精度之所以是**黑魔法**，不是因为「把精度降下来」这件事本身 ——
那谁都会。难的是**搞清楚哪个操作必须用哪种数字格式**：

- **可以用 16 位存储的**：矩阵乘法，以及大多数逐点操作（relu、tanh、加、减、乘）；
- **需要更高精度（FP32）的**：**把小数值加进大的和里**容易产生舍入误差，
  所以各种**归约操作**（求和、softmax、归一化）要留在 FP32；
- **需要更大范围（FP32 或 BF16）的**：输出远大于输入的逐点操作（exp、log、pow），
  以及损失函数。

他说，低精度训练走到今天这个状态**花了好几年**，
而且这些工作大多是**缓慢的、增量的、经验性的**：
一点点摸清哪些操作能下转、怎么下转、怎么在尽量多用低精度的同时保住训练稳定性。

#### FP8：不再有唯一的标准格式

到了 16 位，大家基本都用 **BF16**。但到了 **FP8**，
**不存在一个所有人都用的标准格式**了，出现了两种：

- **E4M3**：4 位指数 + 3 位尾数。尾数多一点、范围小一点；
- **E5M2**：5 位指数 + 2 位尾数。范围大一点、精度粗一点。

它们用在不同的地方。位数一少，「一种格式打天下」就不成立了。

#### MXFP8：多个缩放因子，以及转置忽然不再免费

<Figure src="/courses/cs336/l05-mxfp8.png" wide caption="左：FP16 / BF16 / FP8 E4M3 / FP8 E5M2 的位分布对比。右上：普通 FP8 一整个矩阵共用一个 FP32 缩放因子，MXFP8 则每 32 个元素配一个 E8M0 缩放因子。右下：MXFP8 在前向和反向里的数据流。来源：CS336 第 5 讲课件" />

先说**为什么需要缩放因子**：E4M3 只有 4 位指数，
**动态范围极窄**（最大只能表示到 448），数值稍微大一点或小一点就会上溢 / 下溢。
所以要配一个**缩放因子**把整个张量拉回可表示的区间。

传统 FP8 训练的做法是：整个矩阵的激活值都是 8 位，
外面配**一个 FP32 的缩放因子**。

但人们发现这样不够好：**同一个矩阵内部的数量级可能差别很大** ——
序列的某一部分激活值可能比另一部分大得多。
于是就有了 **MXFP8** 这类**多缩放因子**格式：

- **每 32 个元素配一个缩放因子**，每个缩放因子只管自己那一小块子矩阵；
- **缩放因子本身也是 8 位的**，格式是 **E8M0** —— 只有指数位，没有尾数，
  所以它能表示的全是 2 的幂；
- 因为缩放因子变多了，元素本身可以选**尾数更多**的 **E4M3**。

<Callout type="insight" title="停下来想一想：这个设计有什么问题？">
Tatsu 在课上让大家先别看幻灯片第三条，自己想一分钟。

问题出在**转置**上。

原来转置一个矩阵是件小事：转就转了，不用重新量化，什么都不用做。
现在不行了 —— **转置后的矩阵，它的分组模式和原来的完全不一样。**
「内存里连续的 32 个元素」在转置之后变成了「跨行分布的 32 个元素」，
根本不是同一组。所以你**可能得把整个矩阵按新的 1/32 分组重新量化一遍**。

那实际怎么解决的？Tatsu 说他觉得这件事「既疯狂又很酷」：
**训练时每个量化矩阵直接存两份** —— 原矩阵一份、转置一份。
什么时候要转置，转置版本就在那儿等着。
</Callout>

现实里的 MXFP8 训练还有两个细节：

- **不是所有权重都转成 MXFP8**，只量化那些**被判断为安全**的层。
  这又是一个只能靠试错摸出来的东西。
- **转置也是单独量化的**（对应上面说的两份拷贝）。

那收益有多大？**矩阵乘法那部分大约能省 20% 到 30%，矩阵越大可能更多。**
注意**不是 2 倍加速** —— 因为量化和反量化本身有不小的开销，这些优化不是白来的。

<Callout type="note" title="课堂提问：为什么第一层和最后一层难量化？">
有同学提到读到过「第一层和最后一层难量化」。Tatsu 说确实如此。

他对第一层没有很好的直觉；但**最后一层**难量化的原因他认为是：
**它是一阶因素，直接驱动了很大一部分损失**。
所以如果你硬要量化最后一层，往往会同时得到**训练不稳定**和**损失明显变差**。
</Callout>

#### FP4：一共只有 16 个数

<Figure src="/courses/cs336/l05-mxfp4-values.png" wide caption="4 位浮点（E2M1）能表示的全部数值 —— 一共 16 个：0、±0.5、±1、±1.5、±2、±3、±4、±6。来源：CS336 第 5 讲课件" />

Tatsu 说他把这页放上来「几乎是因为它太荒谬了」：
**一整个数字格式的所有取值都能在一页幻灯片上列完。**
范围就是 $-6$ 到 $6$，一共 16 个数。

结构和 MXFP8 一样：分块 + 缩放因子。课上给的参数是
**每 16 个元素配一个缩放因子，缩放因子本身是 E4M3 格式的 FP8**。

Tatsu 说：**已经有论文用 FP4 训过模型**，但他没听说有人在真实场景里
用 FP4 训过认真的大模型。不过他认为**下一代模型很可能就是 FP4 的**。

<Callout type="note" title="补充：MXFP4 和 NVFP4 的命名">
课件上把它标成 MXFP4，但描述的参数（**16 个元素一组、E4M3 的缩放因子**）
其实是 NVIDIA 在 Blackwell 上的那个变体，业界通常叫 **NVFP4**。

开放计算项目（OCP）标准里的 **MXFP4** 是**32 个元素一组、缩放因子用 E8M0**。
两者的元素格式都是 E2M1（也就是上图那 16 个值），差别在分组粒度和缩放因子的精度：
NVFP4 分组更细、缩放因子精度更高，代价是缩放因子的额外开销是前者的两倍。
</Callout>

<Callout type="note" title="课堂提问四则：关于量化的实践">
课上关于低精度问了不少问题，一并记在这里：

**「这只对矩阵有用吗？」** 不是，**你可以量化任何东西** ——
比如 ReLU 之后的激活值也可以量化，量化了就能减少内存搬运。
但**没那么大必要**：矩阵乘法量化之后吞吐量提升非常大，
其它操作的开销往往不值回票价。所以你看 MXFP8 训练的那张图，
**被量化成 FP8 的全是矩阵乘法**。

**「这是硬件限制吗？」** 不是限制，但**硬件里确实为这些格式内置了加速电路**。

**「你把它说成是省内存带宽，那算力上有收益吗？」** 有，而且是**线性的** ——
乘更少位数的数就是更快。但因为你必须做量化和反量化，**这份收益被稀释了**。

**「缩放因子需要训练吗？」** 缩放因子怎么定是**依赖具体实现的选择**，
不同的库有不同的量化 / 反量化方式。「训练」这个词说重了：
可能是扫一遍取最大最小值，也可能是看历史滑动统计量。
**推理时**你可能会更精细地去拟合它们（那时确实有梯度），**训练时一般不会**。

**「结构化稀疏（structured sparsity）呢？」** 这方面做过很多工作，
Tatsu 提到 Chris Ré 在各种结构化矩阵上做了不少。
但他认为**算力收益和表示能力损失基本互相冲掉了** ——
不是个坏主意，**但经验上没做成**。

**「现在的做法是什么？先全精度训再蒸馏成量化模型？」**
组合拳：一般是**先训一个更大的模型再往下量化**；
现在也有各种**量化 scaling law** 的论文在刻画这里的取舍。
最先进的做法是**量化感知训练（QAT）+ 模型规模选择 + 各种训练后量化（PTQ）**混用。
Tatsu 补了一句：**这块还远远没被完全搞清楚**，
连他聊过的工业界团队都还在说自己在做「量化的科学」。
</Callout>

<Callout type="tip" title="动手：数字格式、算术强度与屋顶线判定">
下面这个演示把这一节的三件事放在一起：

1. **上半**是六种格式的位分布和可表示范围。切一遍你会看到 BF16 的取舍 ——
   它的指数位和 FP32 一样多（所以范围一样大），尾数砍到 7 位。

### 8.3 算子融合：读一次，全算完，写一次

融合的想法很简单，简单到 Tatsu 说「你会惊讶于这件事居然经常没被做」。

考虑一个很普通的计算：$\sin^2(x) + \cos^2(x)$。PyTorch 的计算图长这样：
从 $x$ 出发，做 sin、做 cos，两边各平方，再相加，得到输出。**一共 5 个逐点算子。**

**如果什么都不做**，这 5 个算子每一个都会**从全局内存读一遍、写回一遍**——
也就是启动 **5 个 CUDA kernel**，付 5 个来回的内存开销。

**融合（fusion）的做法**是：写一段 GPU 代码，**从全局内存读一次**，
在 SM 内部把整条链算完，**再写回一次**。一个 kernel 搞定所有操作。

<Figure src="/courses/cs336/l05-fusion-graph.png" wide caption="左：融合前，5 个逐点算子各自是一个节点，每个都要读写一次全局内存。右：TorchInductor 融合之后，它们被压成了一个 CUDA kernel。来源：CS336 第 5 讲课件" />

**简单的融合编译器可以自动做。** `torch.compile`（PyTorch 的编译器）
和 `jax.jit`（JAX 的编译器，和 JAX 的集成更深）都会把这类基础操作
融合成一次 CUDA 调用。更高级的融合有时仍然需要人工介入。

<Callout type="note" title="补充：一段可跑的融合基准（课件上没有）">
这段代码是为了帮助理解补的。它在真实 GPU 上量一下融合到底值多少。

```python
import torch

# 一个足够大的张量，保证我们量的是内存带宽而不是 kernel 启动开销
# 2**27 个 bf16 元素 = 1.34 亿个元素 = 256 MB
x = torch.randn(2**27, device="cuda", dtype=torch.bfloat16)

def f(x):
    # 5 个逐点算子：sin、平方、cos、平方、相加
    return torch.sin(x) ** 2 + torch.cos(x) ** 2

f_fused = torch.compile(f)          # 让编译器把这 5 个算子融成一个 kernel

def bench(fn, n=50):
    for _ in range(5):              # 预热：第一次调用要编译 / 要分配显存，不能算进去
        fn(x)
    torch.cuda.synchronize()        # GPU 是异步的，计时前必须等它把活干完
    start = torch.cuda.Event(enable_timing=True)
    end = torch.cuda.Event(enable_timing=True)
    start.record()
    for _ in range(n):
        fn(x)
    end.record()
    torch.cuda.synchronize()
    return start.elapsed_time(end) / n   # 返回单次平均毫秒数

t_naive = bench(f)
t_fused = bench(f_fused)
print(f"未融合 {t_naive:.3f} ms，融合后 {t_fused:.3f} ms，加速 {t_naive / t_fused:.2f}x")

# 把耗时换算成有效带宽，看看离硬件峰值还有多远
bytes_naive = 2 * 5 * x.numel() * 2   # 5 个算子，每个 1 读 1 写，bf16 每元素 2 字节
bytes_fused = 2 * 1 * x.numel() * 2   # 融合后全程只读一次写一次
print(f"未融合有效带宽 {bytes_naive / (t_naive * 1e-3) / 1e12:.2f} TB/s")
print(f"融合后有效带宽 {bytes_fused / (t_fused * 1e-3) / 1e12:.2f} TB/s")
```

两点值得留意：

- 用 `torch.cuda.Event` 而不是 `time.time()`，因为 CUDA 调用是**异步**的 ——
  Python 那行代码返回时 GPU 可能还没开始算。
- 算出来的「有效带宽」应该接近这块卡的 HBM 带宽标称值。
  如果差得很远，说明你量到的是别的东西（比如 kernel 启动开销）。
</Callout>

### 8.4 重计算：把激活扔掉，反向时再算一遍

反向传播我们都知道：前向沿着计算图往上走，**把激活值存下来**；
反向时用存下来的激活值算梯度，一路往回推。

但现在我们不是从「怎么算梯度」的数学视角看这件事，而是从**系统视角**看：
**怎么把内存搬运降到最低？**

举个最简单的例子：$x \to \text{sigmoid} \to \text{sigmoid} \to \text{sigmoid} \to \text{out}$。
每个 sigmoid 都会在计算图里留下一个需要保存的激活值（记作 $s_2$、$s_1$）。

**存激活的方案，数一下访存次数**：

- 前向：读 $x$（1 次读），写 $s_2$、$s_1$、$\text{out}$（3 次写）；
- 反向：读 $s_2$、$s_1$、$d_{\text{out}}$（3 次读），写 $d_x$（1 次写）。

**一共 8 次访存**，而算术强度低得可怜 —— 因为我们只是在堆 sigmoid，
每次访存对应的浮点运算少得可以忽略。

<Figure src="/courses/cs336/l05-recompute-before.png" caption="存激活的方案：前向 1 次读 + 3 次写，反向 3 次读 + 1 次写，一共 8 次。来源：PyTorch dev-discuss，转引自 CS336 第 5 讲课件" />

**重计算的方案**：把中间激活值**全部扔掉**。

- 前向：读 $x$（1 次读），写 $\text{out}$（1 次写）—— 输出必须写，那是图的输出；
- 反向：读 $x$ 和 $d_{\text{out}}$（2 次读），**在反向过程中现场把 $x$ 重新前向一遍**，
  边算边用，最后写 $d_x$（1 次写）。

**一共 5 次访存。** 从 8 降到 5，也就是 **5/8**。

<Figure src="/courses/cs336/l05-recompute-after.png" caption="重计算的方案：前向只留输出（1 读 1 写），反向从 x 现场重新前向（2 读 1 写），一共 5 次。来源：PyTorch dev-discuss，转引自 CS336 第 5 讲课件" />

<Callout type="insight" title="为什么这笔交易划算">
正常情况下你绝不会想「重新算一遍我已经算过的东西」。

但设想一个**算力极其充裕、内存极其昂贵**的世界：你的计算单元根本没用满，
而任何一次读写都要付出很大的代价。**那这笔交易就是划算的。**
而这正是现在 GPU 所处的位置 —— 回想内存墙那张图。

Tatsu 说：平时我们只关心「把梯度算出来」，**一旦有了系统的心态，
你就会开始考虑重计算。**
</Callout>

<Callout type="tip" title="动手：把融合和重计算的字节账算出来">
下面这个演示把上面两节的账摆开：

- **上半**：拖动算子个数，看未融合和融合后的搬运量差多少倍，
  以及在 3.35 TB/s 的带宽下分别要花多少毫秒。**加速比恰好等于算子个数** ——
  因为融合把 $2k$ 次访存压成了 2 次。
- **下半**：拖动 sigmoid 层数 $L$。存激活的访存次数是 $2L+2$，
  重计算永远是 **5**（跟层数无关）。$L=3$ 时正好是课上的 8 对 5；
  层数越多，重计算越划算；$L=1$ 时反而是存激活更省 —— 这个技巧要堆得够深才有意义。
</Callout>

### 8.5 内存合并访问：DRAM 是按「突发」读的

<Figure src="/courses/cs336/l05-dram-burst.png" wide caption="上：地址空间被划分成一个个「突发段」，访问其中任何一个位置，同段内其它位置都会被一并送出。下左：DRAM 存储单元的物理排布 —— 行地址译码器选中一行之后，整行被复制到感应放大器上。来源：CS336 第 5 讲课件" />

这一条 Tatsu 自己都笑称是「DRAM 冷知识」，但它出人意料地重要。

全局内存是 **DRAM**，而 DRAM 因为物理结构的关系，**是按「突发（burst）」读的**。
意思是：当你访问第一个位置时，**你并不是只拿到那一个值** ——
同一个突发段里的其它值会**顺带白送给你**。

物理原因是：DRAM 的存储单元排成一个网格，
读数据时先要**激活某一行**（把它复制到感应放大器上），而这个「加电压选中一行」的动作
**相当耗时**。一旦选中了，把这一行里的多个元素读出来就便宜得多。

**实际数字**：一个突发段通常是 **128 字节甚至更多**。
也就是说**一次读操作可能返回 128 字节的数据**，只要它们落在同一个连续块里。

<Theorem type="definition" name="合并访问（coalescing）">
如果**一个 warp 里的所有线程**这一步访问的地址**落在同一个突发段里**，
就说这次内存访问是**合并的（coalesced）**。

合并的时候，整个 warp 的这次取数**只发出一个 DRAM 请求**，
而且突发段捎回来的数据**每一个字节都被用上了**。
</Theorem>

<Callout type="note" title="课堂提问：为什么是这种形状？">
有同学问 DRAM 的读为什么会有那种网格 + 行 / 列的形状。Tatsu 的解释是：

存储单元排成一个网格，**决定读哪些单元的放大器是按列的层级工作的**，
而**给它加电压这个动作本身相当花时间**。
一旦选中了某一列（或者行，取决于图怎么画），
**读出这一列里的多个元素就便宜得多**。

所以，如果你想读左上角那个红格子，你会**顺带免费拿到一整行**。
但你不会免费拿到别的行。
</Callout>

#### 用在矩阵上：行主序意味着什么

<Figure src="/courses/cs336/l05-coalescing-matmul.png" wide caption="(A) 每个线程沿着自己那一行往前走 —— 不合并。(B) 线程一起沿列方向推进 —— 合并。右边是逐次取数的地址落点，注意 (B) 每一步读的是内存里连续的一段。来源：CS336 第 5 讲课件" />

助记法是：**对行主序矩阵，沿着主轴（行）方向移动的线程是不合并的。**

具体想一遍。假设有一个 4×4 的行主序矩阵，内存里的线性顺序是：
第 0 行的 4 个元素、第 1 行的 4 个元素……

现在假设**每个线程负责一行，每个时钟周期往右移一格**。
第一步，线程们分别读 $(0,0)$、$(1,0)$、$(2,0)$、$(3,0)$ —— **这是一整列**。
在行主序里这四个地址彼此相距一整行的长度，**每一个都落在不同的突发段里**。
结果是：**你明明只想要一列，却把整个矩阵都读了一遍。**

反过来，如果把访问方向转 90 度，让线程们**一起读同一行的相邻元素**，
那这一步取的就是内存里连续的一段 —— **每个线程的读都在同一次内存事务里完成**。

**所以矩阵按行存还是按列存、访问方向怎么安排，是值得想一想的事情。**

<Callout type="tip" title="动手：把 DRAM 请求次数数出来">
下面这个演示是一个 8×8 的行主序矩阵，格子里写的是它在内存里的线性地址。
8 个线程一起取数，每次点「下一步」推进一个时钟周期。

- 选「**线程各走一行**」：每一步 8 个线程的地址相距 8，
  落在 **8 个不同的突发段**里 —— 每个突发段只用上 1 个元素，其余全是白读的。
  走完 8 步一共发 **64 次**请求。
- 选「**线程共走一行**」：每一步读的是连续的 8 个地址，
  只落在 **2 个突发段**里（突发段设为 4 个元素时）。
  走完 8 步一共 **16 次**请求 —— 正好等于「整个矩阵 ÷ 突发段大小」这个理论下限。

**4 倍的差距，计算量一模一样。** 把突发段改成 8 个元素，差距会拉到 8 倍。
</Callout>

### 8.6 分块：最重要的那一条

<Figure src="/courses/cs336/l05-tiling-phases.png" wide caption="分块矩阵乘法：把 M 和 N 切成 2×2 的子矩阵，一次装一对子块进共享内存，算出对 P 的部分和，再换下一对。来源：CS336 第 5 讲课件" />

Tatsu 把这一条留到最后，因为**它对性能的影响可能是最大的**。

**分块（tiling）的核心想法**：把要读的东西切成小块，
**每一块装进共享内存**；装进去之后，**在这块上反复读、尽可能多地算**，
算完了再换下一块。

先看**不分块**会发生什么。做一个 $N \times N$ 的矩阵乘法，
**同一个元素会被重复读很多次** —— 准确地说，**每个输入元素会被从全局内存读 $N$ 次**。
（比如 $M_{0,0}$ 会在计算输出第 0 行的每一个元素时都被用到。）

**分块之后**，取分块尺寸为 $T$：

<Theorem type="proposition" name="分块的账">

- **不分块**：每个输入元素被从**全局内存**读 $N$ 次。
- **分块（块大小 $T$）**：每个输入元素被从**全局内存**读 $N/T$ 次，
  然后在每个块内部从**共享内存**读 $T$ 次。

也就是**全局内存访问减少了 $T$ 倍**。

极端情况可以检查一下：$T = N$ 时，每个元素从全局内存读 **1 次**，
在共享内存里访问 $N$ 次 —— 正是我们想要的。

</Theorem>

如果你的共享内存能装下很大的块，**就能大幅削减矩阵乘法的全局内存访问**。
而且顺带还有个好处：**分块之后的取数是可以合并的**（每次装一块正好是连续的一段）。

<Callout type="note" title="课堂提问三则：关于分块">
**「你这么做是因为放不下吧？」** 对，就是这样。
如果你造的是那种全 SRAM 的加速器（Groq 之类），
那你直接把整个矩阵扔进去，全程都快 —— **但极其昂贵**。

**「分块和并行是不是同一回事？」** 有同学问这种切分会不会也发生在加速器之间。
Tatsu 说：**是的，各种并行看起来都有点像分块。**
如果数据在一个轴上、权重在另一个轴上，沿其中一个轴切开就是**数据并行**，
另一种切法是**张量并行**。这些会在并行那几讲展开。

**「PyTorch 会自动做这些吗？」** 会。
**你调用的任何矩阵乘法 kernel 底层都做了分块。**
但如果你在做某种非常规的操作，你可能就得自己考虑分块 ——
你要让自己的读能在局部被反复复用。不过 Tatsu 提醒：
**PyTorch 也不是完美的**，接下来就是它不完美的地方。
</Callout>

#### 麻烦一：分块量化

<Figure src="/courses/cs336/l05-tile-quant.png" wide caption="(a) 矩阵尺寸 256 能被 128 的分块整除，正好 4 块。(b) 尺寸变成 257，就多出两个几乎是空的细长块 —— 启动了 6 个线程块，其中两个的活基本是白干的。来源：NVIDIA 深度学习性能文档，转引自 CS336 第 5 讲课件" />

**分块尺寸未必能整除矩阵尺寸。**

假设分块是 128×128：

- 矩阵是 **256×256**：完美，切成 4 块，皆大欢喜。
- 矩阵变成 **257×257**：突然多出**两条极瘦的块**，里面几乎什么都没有 ——
  但它们照样要占一个 SM、照样要走完整个流程。

所以**分块尺寸是需要优化的东西**。影响它的因素有三个：

- **能不能合并取数**；
- **共享内存有多大**；
- **矩阵维度的整除性**。

<Callout type="tip" title="PyTorch 里的 max-autotune 到底在干什么">
如果你用过 `torch.compile(mode="max-autotune")`，你会发现打开这个开关之后
**PyTorch 会跑十五分钟的基准测试**。

它在干的事就是：**在你的矩阵上把各种分块尺寸都试一遍**，看哪个真的更快。
Tatsu 说，认真优化这些东西**能拿到不小的收益**。
</Callout>

#### 麻烦二：内存对齐

<Figure src="/courses/cs336/l05-alignment.png" wide caption="上：内存是按突发段来的。左下：分块边界和突发段边界对齐时，四次读就能把整个子矩阵读进来。右下：矩阵尺寸偏移一格，同一个分块就横跨了两个突发段，读同一块数据要发两倍的请求。来源：Horace He，thonking.ai，转引自 CS336 第 5 讲课件" />

第二个麻烦更微妙，和上一节的合并访问连着。

设想一个理想情形：**突发段的宽度正好等于分块的宽度**。
那么读四次，你就把整个子矩阵读进来了 —— 每次读都是完美合并的。

现在**把矩阵尺寸挪一格**。第一行被顶偏了，那个蓝色的块跑到了别的位置上，
**再也没有办法用一次合并读把一个分块读出来** ——
所有东西都参差错位了。**读一个分块至少要发两个分块量的请求。**

解决办法是**填充（padding）**：人为地把矩阵补大一点，
让**突发段的边界和分块的边界对齐**。

<Callout type="insight" title="Karpathy 那个著名的例子">
这就解释了一件看起来神神秘秘的事。Andrej Karpathy 在做 nanoGPT speedrun 时说：

> 我把词表大小从 **50257 改成 50304**，拿到了 **25% 的加速**。

「靠填充来提速」听起来很怪，但看过上面那张对齐图就说得通了：
**填充会让行和列挪位置**，挪对了取数就非常高效，挪错了就不是。

顺带算一下这两个数：50257 是个质数级别的尺寸，什么都不整除；
而 $50304 = 128 \times 393$ —— **它能被 128 整除**，只多加了 47 个（永远不会被用到的）词表项。
</Callout>

<Callout type="note" title="课堂提问：有没有推荐的分块尺寸？">
Tatsu 说：**你不该去想分块尺寸**，那属于更底层的系统问题，
除非你在自己写 kernel。**你该想的是你的矩阵尺寸。**

同学追问：那是不是要让矩阵尺寸能被分块尺寸整除？Tatsu 说对，
**一般用 2 的幂，理想情况下也能被 32 整除。**

还有同学问：那我用更大的数（比如 64）是不是更好？
Tatsu 说**不会** —— 这纯粹是个**整除性质**：
只要它能整除你的突发段大小，就够了。
（确实存在别的现象会让更大的因子有额外好处，但**这里不是**。）
</Callout>

## 九、破案：那张怪图到底怎么回事

现在我们手上的零件够了，可以回去解释那张方阵矩阵乘的吞吐量图了。它由三层效应叠成。

### 9.1 第一层：算术强度（整体上升趋势）

这层最容易：**矩阵越大，每读一个字节能配的运算越多**。
方阵矩阵乘的 FLOPs 是 $2N^3$，理想情况下要搬的元素是 $3N^2$ 个，
所以算术强度 $\propto N$，**随 N 线性增长**。这就是曲线整体往右上方走的原因。

### 9.2 第二层：分块与对齐（那几条平行的带）

<Figure src="/courses/cs336/l05-tiling-divisibility.png" wide caption="同一张图，但按矩阵尺寸能被 K 整除来着色：蓝色只能被 1 整除（连偶数都不是），橙色能被 2 整除，绿色 8，红色 16，紫色 32。可以看到 16 和 32 这两组表现一样好。来源：Horace He，thonking.ai，转引自 CS336 第 5 讲课件" />

把图里每个点按**它的尺寸能被多少整除**来着色，图案立刻清楚了：

- **蓝色**（只能被 1 整除，连偶数都不是）和**橙色**（能被 2 整除）：吞吐量明显最差；
- **绿色**（8）、**红色**（16）、**紫色**（32）：越往上越好，
  而且 **16 和 32 表现一样好**，没有进一步的收益。

Tatsu 特别强调：**这不是因为 2 的幂有什么魔力。**
原因是 **16 和 32 已经足够大，正好能填满一个突发窗口** ——
到了这个程度，你就能把整个分块用合并读取进来。**对齐的影响非常大。**

### 9.3 第三层：波量化（那些周期性的塌陷）

<Figure src="/courses/cs336/l05-wave-quantization.png" wide caption="放大看那段周期性塌陷，以及它的成因：1792 边长切出 98 个分块，1793 切出 120 个，而 A100 只有 108 个 SM。来源：CS336 第 5 讲课件" />

最后是最怪的那个：**为什么加一个维度就能让吞吐量断崖式下跌？**
（Percy 管这个叫「GPU 冷知识」，Tatsu 说 Percy 大概讨厌这个话题，但他很喜欢。）

现象出现在 **1792 到 1793** 之间。用 **256 × 128** 的分块算一下：

$$
\frac{1792}{256} \times \frac{1792}{128} \;=\; 7 \times 14 \;=\; 98 \quad\text{个分块}
$$

尺寸加到 1793，两个方向都要向上取整，于是：

$$
\left\lceil \frac{1793}{256} \right\rceil \times \left\lceil \frac{1793}{128} \right\rceil
\;=\; 8 \times 15 \;=\; 120 \quad\text{个分块}
$$

98 变成 120 是变多了，**但这点差别解释不了那么大的跌幅**。真正的原因是：

<Theorem type="proposition" name="波量化（wave quantization）">
**A100 有 108 个 SM。**

- **98 个分块**：一次全部塞进 108 个 SM，**跑一波就完事**。
- **120 个分块**：塞不下。前 108 个跑完之后，**还剩 12 个**，
  必须再跑一整波 —— 而这一波里**有 96 个 SM 完全闲着**。

于是有效利用率从 $98/108 = 90.7\%$ 掉到 $120/216 = 55.6\%$。
**你只是把矩阵边长加了 1，就要多等整整一波的时间。**
</Theorem>

<Callout type="tip" title="动手：把那张怪图的第三层复现出来">
下面这个演示按上面的公式算分块数、波数和 SM 占用率，
并把**占用率随 N 的变化**画成曲线 —— 那些锯齿就是波量化。

几件值得试的事：

- 点「跳到 1792」「跳到 1793」，看这两个数据点的差距。
- 把 SM 数从 A100 的 108 换成 H100 的 132，你会看到**锯齿的位置整体移动了** ——
  在一块卡上跑得飞快的矩阵尺寸，换一块卡未必还快。
- 把分块尺寸换成 128×128 或 64×64，锯齿的频率会变密（因为分块变多了，
  波数上升得更快，但每一波的浪费相对更小）。
- 注意占用率曲线是**锯齿状**的，不是单调的 —— **矩阵变大不一定跑得更快**，
  这正是那张怪图的形状来源。
</Callout>

另外有同学问到**晶圆级引擎（wafer scale engine）**和 GPU 的对比。
Tatsu 说这不是他熟悉的领域，但他上一课后和人聊过：
晶圆级的**编译难度极大** —— 会出现波干扰之类的效应，各种可怕的事情。

<Theorem type="proposition" name="第二部分小结：这六招其实是三件事">

**一、减少访存次数**

- **合并访问**：让一个 warp 的取数落在同一个突发段里。
- **算子融合**：读一次，把整条链算完，写一次。

**二、把内存搬到共享内存里反复用**

- **分块**：把子矩阵装进共享内存，在里面尽可能多算。

**三、拿别的资源去换内存**

- **量化**：拿精度换内存（和算力）。
- **重计算**：拿算力换内存。

（**控制分歧**是唯一一个不属于内存故事的，它是 SIMT 执行模型的直接后果。）

</Theorem>

## 十、收官：把 FlashAttention 拆开

Tatsu 说这一段是「胜利巡场」。**FlashAttention 是这几年注意力上最大的改进之一，
而它完全是系统层面的工作** —— 从 PyTorch 那个朴素实现，变成一个**极其巧妙的融合 kernel**。

原论文里那张图显示的是**服务这些模型的延迟大幅下降**；而正如第四讲提过的，
FlashAttention 也是一种**内存高效**的注意力算法，所以你能把注意力做得越来越大。
**收益的很大一部分来自「从 HBM（也就是全局内存）搬运的数据量变少了」。**

论文里说自己就做了两件事：**分块（tiling）和重计算（recomputation）**，
从而让访存次数变成次二次的。**这两个词我们现在都懂了。**

### 10.1 注意力有哪几步

注意力的计算是：**三个矩阵乘法（K、Q、V），中间夹一个 softmax。**

具体是 $Q K^{\top}$ 一个矩阵乘，softmax，然后再和 $V$ 做一个矩阵乘。

### 10.2 分块第一步：KQV 的矩阵乘

<Figure src="/courses/cs336/l05-flash-tiling.png" wide caption="FlashAttention 论文的图 1。左边是内存层级（SRAM 19 TB/s / 20 MB，HBM 1.5 TB/s / 40 GB，CPU DRAM 12.8 GB/s / 大于 1 TB）；右边是外层循环搬 K、V 的块，内层循环搬 Q 的块，中间的计算在 SRAM 里做完。来源：Dao et al.，转引自 CS336 第 5 讲课件" />

Tatsu 说：**论文的图 1 说白了就是分块矩阵乘法**，我们已经见过了。
把 K 和 Q 切开，一块一块地乘，算完写出去。**这里没有任何新东西。**

注意左边那个内存层级的具体数字：**SRAM 有 19 TB/s 但只有 20 MB，
HBM 有 1.5 TB/s 和 40 GB** —— 这正是我们一路在讲的层级。
FlashAttention 的收益基本上就来自**从 HBM 搬运的数据量变少了**。

**麻烦在 softmax。** softmax 是一个**全局操作**，它把所有位置绑在了一起 ——
它把不同的分块**连起来了**。所以不能天真地按块算。

### 10.3 分块第二步：在线 softmax

<Figure src="/courses/cs336/l05-online-softmax.png" wide caption="左：普通的「安全 softmax」，要扫三遍（先求最大值，再求归一化分母，再逐元素归一化）。右：在线 softmax，把前两遍合并成一遍，边走边修正最大值。来源：Milakov & Gimelshein, 2018，转引自 CS336 第 5 讲课件" />

**这是整个 FlashAttention 的关键一招。** Tatsu 说，一旦你看到这一步，剩下的都很简单。

**普通的 softmax**（所谓「安全 softmax」）要扫三遍数据：

$$
y_i \;=\; \frac{e^{\,x_i - \max_k x_k}}{\sum_{j} e^{\,x_j - \max_k x_k}}
$$

减去最大值是为了**数值稳定**（否则 $e^{x_i}$ 会溢出）。算法上：

1. 第一遍：$m_k \leftarrow \max(m_{k-1},\, x_k)$，扫完得到全局最大值 $m_V$；
2. 第二遍：$d_j \leftarrow d_{j-1} + e^{\,x_j - m_V}$，扫完得到分母 $d_V$；
3. 第三遍：$y_i \leftarrow e^{\,x_i - m_V} / d_V$。

**问题在于第二遍用到了第一遍的结果 $m_V$** —— 你必须先看完所有数据才能开始累加。
这就是「softmax 是全局操作」的具体含义。

**在线 softmax** 把前两遍合并成一遍。核心是：
**边走边更新最大值，同时把已经累好的分母按新的最大值修正一下。**

<Theorem type="definition" name="在线 softmax（Milakov & Gimelshein, 2018）">

初始化 $m_0 = -\infty$，$d_0 = 0$。对 $j = 1, \dots, V$：

$$
m_j \;\leftarrow\; \max(m_{j-1},\, x_j)
$$

$$
d_j \;\leftarrow\; d_{j-1} \times e^{\,m_{j-1} - m_j} \;+\; e^{\,x_j - m_j}
$$

扫完之后，对每个 $i$：

$$
y_i \;\leftarrow\; \frac{e^{\,x_i - m_V}}{d_V}
$$

关键就是 $d_{j-1} \times e^{\,m_{j-1} - m_j}$ 这一项：
它把「按旧的最大值 $m_{j-1}$ 算出来的分母」**换算成「按新的最大值 $m_j$ 算的分母」**。
因为若 $m_j = m_{j-1}$（最大值没变），这个因子就是 1，什么都不做；
若最大值变大了，它就把之前所有项统一乘上一个修正系数。

这构成一个**伸缩求和（telescoping sum）**：不管你分几段来算，
只要每段结束时都做一次这样的修正，结果和一次算完完全一样。

</Theorem>

**为什么这件事对分块至关重要**：因为它是**在线的**，可以一块一块地推进。
**算某一块的 softmax 不需要看到其它块**，
你算完一块就把部分结果（当前的最大值 $m$ 和分母累加器 $d$）留着，
接着算下一块。**这两个要留的东西都只是标量，几乎不占内存。**

### 10.4 拼起来：FlashAttention 的前向

<Figure src="/courses/cs336/l05-flash-forward.png" wide caption="FlashAttention 前向的数据流。蓝色实线框的东西存在 HBM 里，橙色虚线框的东西在 SRAM 里算、不落回 HBM。注意最右边的「rescaling to correct denominator」—— 那就是在线 softmax 的修正项。来源：Dao, 2023，转引自 CS336 第 5 讲课件" />

现在把图看懂就很容易了。图里同时出现了这一讲的三个技巧：

1. **分块地算内积** $S^{(i)} = Q (K^{(i)})^{\top}$ —— 这是分块矩阵乘；
2. **融合指数运算** $A^{(i)} = \exp(S^{(i)})$ ——
   $S$ 算出来之后**不写回 HBM**，直接在 SRAM 里取指数；
3. **分块地算 softmax**：一边推进一边维护 $\ell^{(1)} = \sum_i \exp(S^{(1)})_i$，
   $\ell^{(2)} = \ell^{(1)} + \sum_i \exp(S^{(2)})_i$ —— 就是上面的伸缩求和；
4. 最后每一块和 $V$ 相乘（还是分块矩阵乘），
   **在最末尾统一除以正确的分母**。

**反向传播这一讲没有展开**，但结论是：注意力的激活值是 $n^2$ 大小的，
**存不起**。所以 FlashAttention 干脆**全部扔掉，在反向时一块一块重算**——
这就是论文里说的重计算那一半。

<Callout type="note" title="补充：一段可跑的在线 softmax（课件上没有）">
上面的算法看起来抽象，但十几行就能写完，而且可以直接和 PyTorch 对拍。
这段代码是为了帮助理解补的。

```python
import torch

def online_softmax(x: torch.Tensor, tile: int = 7) -> torch.Tensor:
    """按 tile 大小分块，一块一块地算 softmax。故意用 7 这种不整除的块大小，
    验证算法对任意分块都成立。x 的形状：(n,)。"""
    n = x.numel()
    m = torch.tensor(float("-inf"), dtype=x.dtype)   # 到目前为止见过的最大值
    d = torch.tensor(0.0, dtype=x.dtype)             # 到目前为止累好的分母

    # 第一遍：分块扫过去，同时维护 m 和 d
    for s in range(0, n, tile):
        blk = x[s : s + tile]
        m_new = torch.maximum(m, blk.max())          # 新的最大值
        # 关键的一步：把旧分母按新最大值换算，再加上这一块的贡献
        d = d * torch.exp(m - m_new) + torch.exp(blk - m_new).sum()
        m = m_new

    # 第二遍：有了全局的 m 和 d，逐元素归一化（真实 kernel 里这一步和上面融合在一起）
    return torch.exp(x - m) / d

x = torch.randn(1000) * 30          # 乘 30 是为了制造大数值，测试数值稳定性
ours = online_softmax(x)
ref = torch.softmax(x, dim=0)
print("最大绝对误差:", (ours - ref).abs().max().item())   # 应该在 1e-8 量级
assert torch.allclose(ours, ref, atol=1e-6)
```

注意第一次循环时 $m = -\infty$、$d = 0$，
所以 `d * torch.exp(m - m_new)` 是 $0 \times e^{-\infty} = 0 \times 0 = 0$，
不会出问题。

真正的 FlashAttention kernel 和这段代码的区别在于：它不是在一维向量上做，
而是在注意力矩阵的每一行上做；而且**归一化那一步和矩阵乘 $V$ 融合在了一起**，
所以中间那个 $n \times n$ 的矩阵**从来没有在 HBM 里出现过**。
</Callout>

## 十一、整讲小结

<Theorem type="proposition" name="这一讲留下的三件事">

1. **理解硬件是必要的。** GPU 撑起了整个 scaling，
   而且要理解「我们为什么做某些决策」，必须一路下探到底层细节。
   Tatsu 说得很直白：他**不希望你成为那种照着模仿、把矩阵尺寸凑成 32 的倍数
   却不知道为什么的人** —— 他要你真的明白为什么。

2. **当前的算力增长方式，逼着我们围绕「矩阵乘法 + 数据移动」思考。**
   矩阵乘法是唯一算术密度足够高的核心操作；
   而因为算力和内存之间的差距在拉大，**数据移动必须被仔细对待**。

3. **对硬件敏感的架构工作是有巨大价值的。**
   FlashAttention 就是例子。把硬件想清楚再去做架构，
   对未来系统的性能是决定性的。

</Theorem>
