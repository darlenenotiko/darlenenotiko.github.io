## 一、推理出现在哪里，为什么它越来越重要

### 1.1 推理无处不在

模型训完之后你不会就坐在那儿不动。如果你是研究者，你在论文里放一张图说「我的模型是这样的」；
但对其他所有人来说，你是真的要**用**这个模型。用起来是什么样子？

- **实际使用**：跟 AI 助手 / 聊天机器人对话、代码补全、agent（现在非常热门）、批量数据处理。
- **模型评测**：尤其是那些**需要生成**才能评的评测（比如指令遵循）。
- **强化学习**：RL 训练的内部就要用推理 —— 采样一批 rollout，给它们打分，再据此更新权重。

所以推理散落在语言模型工作的每一个角落。

### 1.2 训练是一次性成本，推理是重复成本

效率是这门课的主线，而在推理这里，效率比别处更要紧，原因很直接：

**训练是一次性成本。** 它可能非常贵、确实非常贵，但跑完就跑完了。
**推理是重复成本。** 你每天都要重新付一遍。

一组具体的数：

- OpenAI 被估计**每天产出约 8.6 万亿个 token**。
- 作为参照，今年早些时候发布的 **DeepSeek v4 是在 32 万亿个 token 上训练的**。

把这两个数放在一起：**不到四天**，OpenAI 在推理侧产出的 token 数就超过了 DeepSeek v4
整个预训练所用的 token 数。当然前沿模型可能训在更多 token 上，但方向是清楚的 ——
**推理侧生成的 token 总量会高于训练侧**。

### 1.3 agent 让这件事变得更极端

Percy 说，过去一年推理的重要性又涨了一档，原因是使用形态变了：

- **过去的聊天机器人视角**：你输入一个 prompt，拿回一个 response，然后**人去读它**。
- **今天的 agent 视角**：一个查询进来，agent 开始做一堆事情 —— 思考、推理、调用工具、自我检查 ——
  最后才产出一段给人读的输出。

关键差别在于：**agent 产出的绝大部分 token 根本不是给人读的**，它们是中间轨迹。
所以应该这样理解：

<Theorem type="proposition" name="生成的 token 数 = 花掉的算力">
在 agent 场景下，生成多少 token 就等于花掉多少算力，而且**没有上限** ——
问题越有野心，需要的算力和 token 就越多。
</Theorem>

这条推论很重要：在聊天场景里，推理快到某个程度就**够快了**，因为人读不了那么快；
但在 agent 场景里，**推理再快都有价值**，你从榨干推理里能拿到的收益没有天花板。

### 1.4 谁在做推理

商业侧：

- **服务闭源模型的提供商**（OpenAI、Anthropic、Google 等）—— 他们必须为自己的模型做推理服务。
- **服务开放权重模型的提供商**（Together、Fireworks、Baseten、DeepInfra、Groq、Cerebras 等）。

开源侧有几个常用的包，各有各的定位：

- **vLLM**：来自伯克利，**首创了 PagedAttention**（本讲最后会讲）。最流行，是个好的默认选择。
- **SGLang**：也来自伯克利，**首创了 RadixAttention**，对 **agentic 工作负载**特别好，不过还没有 vLLM 那么流行。
- **TensorRT-LLM**：来自 NVIDIA，**针对 GPU 做了极致优化**，非常快，但适用面更窄。
- **llama.cpp**：纯 C++，**支持 CPU 推理**，可以在本地跑。

### 1.5 「快」到底指什么：三个指标

推理是个巨大的东西，让它快 10%、快一倍都是大事。但「快」有三个不同的含义，
它们适用于不同场景，而且**彼此之间是有取舍的**：

<Theorem type="definition" name="推理的三个速度指标">

1. **首 token 时间（time-to-first-token, TTFT）**：用户提交查询后，到**第一个 token 出现**之前
   要等多久。适用于**交互式应用** —— 这段时间里屏幕上什么都没有，等得越久体验越差。
   一旦 token 开始出现，反而不需要那么快，因为你也读不了那么快。
2. **延迟（latency，秒/token）**：站在**单个用户**的角度，token 冒出来的速度有多快。
   也就是流式输出的手感。同样适用于交互式应用。
3. **吞吐（throughput，token/秒）**：站在**很多个查询**的角度，单位时间总共产出多少 token。
   适用于**批量处理** —— 你有一个 PB 的数据要用语言模型过一遍，你只关心这个作业什么时候跑完，
   某一条查询是早一点还是晚一点无所谓。

</Theorem>

延迟和吞吐显然高度相关（吞吐大致是延迟的倒数），很多优化会让两者同时变好。
但后面会看到，**在批大小这个维度上它们是直接对立的**。

### 1.6 一个高层判断：推理和训练是完全不同的负载

如果这一讲只带走一句话，Percy 说应该是这句：

<Callout type="insight" title="推理和训练的根本差别">
**训练时你一次性看到全部 token。** 监督微调或预训练时，整个序列是已知的，
序列长度在 Transformer 里就只是张量的一个维度 —— 注意力和 MLP 都是大矩阵乘法，
**可以沿序列维度并行**，所有 token 一次算完。

**推理时你做不到这一点。** 由于自回归的本性，你必须**一个 token 一个 token 顺序生成**，
**没法沿序列维度并行**。

这就是推理为什么是一个和训练截然不同的负载。而后果是：**很难维持高算术强度、
很难把算力吃满。**
</Callout>

## 二、复习：记号与算术强度

这一节基本是复习，主要目的是统一记号。

### 2.1 张量维度的记法

课件用一套类似 einops 的记法：**用一个符号同时表示某个维度和它的长度**。

- $B$：批维度，也就是**序列条数**；
- $T$：序列维度，也就是 **token 数**；
- $D$：模型维度；
- $H$：注意力头维度。

写一个矩阵乘法时，用颜色区分三类维度：

- **收缩维（contracting）**：在**两个操作数里都出现**，并且**从结果里消失**。
  例如 $BT\underline{D} \times \underline{D}H \to BTH$ 里的 $D$。
- **普通维（regular）**：只在**一个操作数**里出现，并且**保留在结果里**。上式里的 $B$、$T$、$H$。
- **批维（batching）**：在**两个操作数里都出现**，但**保留在结果里**、不被消掉。
  例如 $\underline{B}\underline{D} \times \underline{B}\underline{D} \to B$ 里的 $B$。

第一个式子的含义是：对每一个序列、每一个 token 位置都有一个向量，把它乘以一个矩阵。
第二个式子的含义是：做一批点积。**记住第二种形式**，后面会看到它正是注意力在解码时的样子，
而它的算术强度极差。

### 2.2 Transformer 块的完整形状

<Figure src="/courses/cs336/l10-transformer-diagram.png" wide caption="一个 Transformer 块的完整张量流：左边是电路图，右边是符号表。每个圆点是一次张量收缩，红色标注的是被收缩掉的维度，蓝色是批维度。来源：Google Scaling Book，转引自 CS336 第 10 讲课件" />

Percy 说这张图看起来像电路图，细节很多，但他认为这大概是**对 Transformer 最干脆的定义** ——
市面上大量的 Transformer 讲解其实很难让人看清到底发生了什么，而这张图**直接告诉你每个张量的形状**，
于是你可以推理出依赖关系。

一个 Transformer 块做的事：拿到某一层的激活 $X$，先过注意力，再过 MLP。

- 注意力这边：把 $X$ 分别用查询矩阵、键矩阵、值矩阵投影。
  查询的形状是 $B \times T \times N \times H$（批 × 序列 × 头数 × 头维度）；
  而键和值那边不是 $N$ 而是 $K$ —— 一个**可能更小的键值头数**。这一点是后面 GQA 那一节的伏笔。
- 注意力操作本身有**批维度 $B$ 出现在两个操作数里**，收缩掉的是头维度。
  **这个 $B$ 同时出现在两边，正是推理慢的根源**，第三节会展开。
- MLP 这边很直接：门控矩阵、上投影、下投影，几个矩阵乘法而已。

几个贯穿全讲的约定：

- $F = 4D$：MLP 把 $D$ 维的模型维度**上投影成 4 倍**。以后看到 $F$ 就想成 $4D$。
- $D = NH$：模型维度被**切分到 $N$ 个头**上。
- $N = KG$：在分组查询注意力里，$N$ 个查询头被分成 $K$ 组，每组有 $G$ 个查询头。
- $S$ 和 $T$ 都表示序列维度，但含义不同：**$S$ 是输入（条件）token 数，$T$ 是输出（要生成）token 数**。
  训练时两者相等（输入即输出）；推理生成时 $T = 1$，$S$ 是已有的上下文。

<Callout type="note" title="补充：K 和 G 到底谁是谁">
课上有同学问过这个记号，Percy 当场也绕了一下，这里明确一次（图右边的符号表是对的）：

- **$K$ = 键值头的个数 = 组数**。
- **$G$ = 每个键值头对应几个查询头** $= N / K$。

也就是说，$K$ 个键值头，每个被 $G$ 个查询头共用，$K \times G = N$。
多头注意力是 $K = N$（每个查询头都有自己的键值），多查询注意力是 $K = 1$（所有查询头共用一份）。
</Callout>

### 2.3 算术强度：热身

这个概念第 2 讲讲过，这里热身一遍。假设要做一个矩阵乘法：
$X$ 的形状是 $B \times D$，$W$ 的形状是 $D \times F$。直觉上 $B$ 是批大小，
$D$ 是隐藏维度 / 模型维度，$F$ 是 MLP 里的上投影维度。

推理时一律用 bf16，也就是**一个数占 2 个字节**。逐步核算：

```python
flops = 0              # 浮点运算次数
bytes_transferred = 0  # 在 HBM 和计算单元之间搬运的字节数

# 1. 从 HBM 读 X（B × D）
bytes_transferred += 2*B*D
# 2. 从 HBM 读 W（D × F）
bytes_transferred += 2*D*F
# 3. 计算 Y = X @ W
flops += 2*B*D*F       # 每个输出元素要 D 次乘加，一次乘加算 2 次浮点运算
# 4. 把 Y（B × F）写回 HBM
bytes_transferred += 2*B*F

assert flops == 2*B*D*F
assert bytes_transferred == 2*B*D + 2*D*F + 2*B*F
```

注意这里的结构：**读写量都是「元素个数」这个量级的项（二次的），
而矩阵乘法的运算量是三次的。** 这个「次数差一阶」正是能拿到高算术强度的原因。

<Theorem type="definition" name="算术强度（arithmetic intensity）">
$$
\text{算术强度} \;=\; \frac{\text{浮点运算次数}}{\text{搬运的字节数}}
$$

含义是「每搬 1 个字节，能顺带做多少次运算」。**越高越好。**
</Theorem>

代入上面的结果：

$$
\text{强度} \;=\; \frac{2BDF}{2BD + 2DF + 2BF}
$$

如果假设**批维度 $B$ 远小于 $D$ 和 $F$**（把 $D$ 和 $F$ 都写成 $cB$ 再让 $c$ 趋于无穷），
这个式子就化简成：

$$
\text{强度} \;\to\; B
$$

也就是**算术强度就等于批大小**。

<Callout type="note" title="补充：这和第 2 讲的结论是什么关系">
第 2 讲算算术强度时用的是**方阵**：两个 $n \times n$ 矩阵相乘，运算量是 $n^3$ 量级、
搬运量是 $n^2$ 量级，所以强度是 $n$ 量级。

这里只是把它推广到**非方阵**：三个维度 $B$、$D$、$F$ 各自不同，
而**最短的那一边（这里是 $B$）决定了强度**。结论的形状是一样的。
</Callout>

### 2.4 加速器强度：多少才算够

硬件那边也有一个「强度」：查规格表拿到**每秒浮点运算次数**，再拿到**内存带宽**，两者相除。

$$
\text{H100 的加速器强度} = \frac{989 \times 10^{12}\ \text{FLOP/s}}{3.35 \times 10^{12}\ \text{字节/s}} \approx 295
$$

意思是：**H100 每从显存搬 1 个字节，就有能力做 295 次浮点运算。**
如果你的计算达不到这个比例，搬运就成了瓶颈，算力在那儿闲着。

<Theorem type="proposition" name="判据">
- 计算强度**大于**加速器强度 → **算力受限（compute-bound）**，好。
- 计算强度**小于**加速器强度 → **内存受限（memory-bound）**，坏。

对上面那个矩阵乘法，结论是：**批大小超过 295 才算力受限。**
</Theorem>

**极端情况：$B = 1$。** 这时矩阵乘法退化成矩阵-向量乘法：

- 算术强度 = 1。
- 严重内存受限：你把整个 $D \times F$ 矩阵读进来，却只做了 $2DF$ 次浮点运算。

Percy 说：**这基本上就是推理时会遇到的情况。** 推理拿不到那种胖矩阵，
拿到的是非常「瘦」的矩阵或张量。

## 三、推理的算术强度：拆开看 prefill 和 decode

### 3.1 最朴素的做法，以及它有多糟

先看不做任何优化会怎样。

<Figure src="/courses/cs336/l10-naive-inference.webp" wide caption="朴素采样：每生成一个 token，就把整条历史重新喂一遍 Transformer。绿色是已有的 token，蓝色是新采样出来的。来源：Google Scaling Book，转引自 CS336 第 10 讲课件" />

流程是：prompt `never gonna give you` 送进 Transformer，Transformer 算出键、值和各层激活，
最后在词表上给出 logits，采样得到一个 token `up,`；把这个 token **拼回 prompt**，再来一遍；
得到 `never`，再拼回去，再来一遍……

如果你手上有一个「输入一个序列、输出下一个 token 分布」的黑箱（Transformer 正是这样一个黑箱），
你确实可以就这么反复调用它。这是能用的，但**非常糟糕**：

- 一次前向传播是 $O(T^2)$（因为注意力是平方的）；
- 每生成一个 token 都要做一次完整的前向；
- 所以**生成 $T$ 个 token 一共要 $O(T^3)$ 次浮点运算**。

### 3.2 观察：前缀的工作可以复用

关键观察是：生成 `never` 那一轮和生成 `up,` 那一轮，前面 `never gonna give you` 这几个 token
的键和值**算出来是一模一样的**。

为什么？因为这是**因果（causal）Transformer** —— 一个位置只能看到它前面的位置。
所以你往序列末尾追加 token，**前面那些位置的激活不会变**。
（如果是双向模型就不成立了：加一个 token，所有东西都会变。）

于是有了最直接的做法：**把键和值缓存在 HBM 里**，在相邻两次生成之间复用。

<Figure src="/courses/cs336/l10-cached-inference.webp" wide caption="带 KV cache 的采样。红框是 prefill（算力受限），蓝框是 generate（内存带宽受限）。每生成一个 token，只往 KV cache 里追加一小段新的键值。来源：Google Scaling Book，转引自 CS336 第 10 讲课件" />

于是推理被切成两个阶段：

<Theorem type="definition" name="推理的两个阶段">

1. **预填充（prefill）**：给定 prompt，把它编码成向量，**填满 KV cache**。
   因为整个 prompt 是已知的，这一步**可以像训练一样沿序列并行**。
2. **生成 / 解码（generation / decode）**：一个一个地生成回复 token，**必须顺序进行**。
   好处是不用再为已经看过的 token 重算 KV。

</Theorem>

KV cache 的形式化定义：**对每一条序列（共 $B$ 条）、每一个 token（共 $S$ 个）、每一层（共 $L$ 层）、
每一个键值头（共 $K$ 个），存一个 $H$ 维向量；而且键和值各存一份。**

<Callout type="note" title="补充：KV cache 到底有多大（一步步算给你看）">
拿这一讲后面要用的 Llama 2 13B 做例子：$K = 40$ 个键值头，$H = 128$ 维头维度，
$L = 40$ 层，bf16（每个数 2 字节）。

**一个 token、一层**要存多少字节？

$$
\underbrace{2}_{\text{键}+\text{值}} \times \underbrace{40 \times 128}_{K \cdot H} \times \underbrace{2}_{\text{bf16 字节数}} \;=\; 20\,480\ \text{字节} \approx 20\ \text{KB}
$$

**一个 token、全部 40 层**：$20\,480 \times 40 = 819\,200$ 字节 $\approx 0.82$ MB。

也就是说 —— **每多生成一个 token，显存里就多压上大约 0.8 MB。**

**一条 1024 token 的序列**：$0.82\ \text{MB} \times 1024 \approx 838.9$ MB。

**64 条并发序列**：$838.9\ \text{MB} \times 64 \approx 53.7$ GB。

对照一下：Llama 2 13B 的参数本身在 bf16 下是 26.0 GB。
**64 条 1024 长度的序列，KV cache 就已经是参数的两倍多了。**
这就是为什么本讲后半段几乎所有招数都在打 KV cache 的主意。
</Callout>

### 3.3 MLP 层的核算

现在正式算 FLOPs 和内存 IO。记住：**$S$ 是我们条件所依赖的 token 数，$T$ 是我们要生成的 token 数。**
先抽象地算，之后再分别代入 prefill（$T = S$）和生成（$T = 1$）。

全程**只数矩阵乘法** —— 其他操作的浮点运算次数少得多，而且往往可以融合进矩阵乘法的 kernel 里。

```python
flops = 0
bytes_transferred = 0

# 1. 从 HBM 读激活 X（B × T × D）
bytes_transferred += 2*B*T*D
# 2. 从 HBM 读三个权重矩阵：Wup(D×F)、Wgate(D×F)、Wdown(F×D)
bytes_transferred += 3 * 2*D*F
# 3. 上投影 U = X @ Wup
flops += 2*B*T*D*F
# 4. 把 U（B × T × F）写回 HBM
bytes_transferred += 2*B*T*F
# 5. 门控 G = X @ Wgate
flops += 2*B*T*D*F
# 6. 把 G（B × T × F）写回 HBM
bytes_transferred += 2*B*T*F
# 7. 下投影 Y = (GeLU(G) * U) @ Wdown
flops += 2*B*T*D*F
# 8. 把 Y（B × T × D）写回 HBM
bytes_transferred += 2*B*T*D

assert flops == 6*B*T*D*F
assert bytes_transferred == 4*B*T*D + 4*B*T*F + 6*D*F
```

于是算术强度是

$$
\frac{6BTDF}{4BTD + 4BTF + 6DF}
$$

再次假设 **$B \cdot T$ 远小于 $D$ 和 $F$**，化简得到：

$$
\text{MLP 强度} \;\to\; B \cdot T
$$

这和 2.3 节的结果完全同构 —— MLP 说到底就是几个大矩阵乘法，只是多了几个维度。
而且这个结果也符合直觉：**在 MLP 里，批维度和序列维度上的一切都是彼此独立的**，
它们不发生交互（注意力就不是这样了）。

代入两个阶段：

1. **prefill（$T = S$）**：强度是 $B \cdot S$。**很容易做到算力受限** —— 批大一点、序列长一点就行。
2. **生成（$T = 1$）**：强度退化成 $B$。这里有两个麻烦：
   - 一次只生成一个 token，$T$ 被钉死在 1；
   - $B$ 是**并发请求数**。如果你在做批量处理，这个数你自己说了算；
     但如果你在服务一个聊天机器人，它就是**当前有多少并发用户**，
     可高可低、随时间变化、**不可预测**。（第九节的连续批处理就是在处理这件事。）

总体来说 MLP 这边**不算太糟**：只要批够大就还行。

### 3.4 注意力层的核算

注意力这边用 FlashAttention 的方式来算（也就是不把 $S \times T$ 的注意力矩阵写回显存）。
再强调一次记号：**$S$ 是已经生成 / 已经在上下文里的 token 数，$T$ 是这一步要产出 logits 的 token 数。**

```python
flops = 0
bytes_transferred = 0

# 1. 从 HBM 读 Q（B × T × D）、K（B × S × D）、V（B × S × D）
bytes_transferred += 2*B*T*D + 2*B*S*D + 2*B*S*D
# 2. 算注意力分数 A = Q @ Kᵀ
flops += 2*B*S*T*D
# 3. 算输出 Y = softmax(A) @ V（softmax 本身的浮点运算量可以忽略）
flops += 2*B*S*T*D
# 4. 把 Y（B × T × D）写回 HBM
bytes_transferred += 2*B*T*D

assert flops == 4*B*S*T*D
assert bytes_transferred == 4*B*S*D + 4*B*T*D
```

这次不需要任何近似，直接约分就能得到一个**精确的**表达式：

$$
\text{注意力强度} \;=\; \frac{4BSTD}{4BSD + 4BTD} \;=\; \frac{S \cdot T}{S + T}
$$

代入两个阶段：

1. **prefill（$T = S$）**：

   $$
   \frac{S \cdot S}{S + S} = \frac{S}{2}
   $$

   **这是好消息** —— 只要序列够长，注意力在 prefill 阶段也能保持高强度。
   比如 $S = 1024$ 时强度是 512，已经超过 H100 的 295 了。

2. **生成（$T = 1$）**：

   $$
   \frac{S \cdot 1}{S + 1} = \frac{S}{S+1} \;<\; 1
   $$

   **这是坏消息，而且是致命的。** 强度小于 1，而我们需要它是 295 量级才能把 H100 喂饱。
   **这就是整个推理的瓶颈所在。**

### 3.5 为什么加大批大小救不了注意力

请注意上面那个精确式子 $S \cdot T / (S+T)$ 里**根本没有 $B$**。也就是说：

> **在生成阶段，把批大小调大对注意力的算术强度毫无帮助。**

这和 MLP 完全不同（MLP 的强度是 $B \cdot T$，调大 $B$ 直接线性受益）。为什么？

<Theorem type="proposition" name="根本原因">
- **在 MLP 层，每一条序列打的是同一份权重。** $W_{\text{up}}$、$W_{\text{gate}}$、$W_{\text{down}}$
  **不依赖 $B$**。所以你把权重从显存读进来一次，就能给所有 $B$ 条序列用 —— 成本被摊薄了，
  这正是高算术强度的来源。
- **在注意力层，每一条序列有自己的一份 KV cache。** $Q$、$K$、$V$ **全都依赖 $B$**。
  多来一条序列，就要多搬一整份它自己的 KV cache，**什么都摊不掉**。

</Theorem>

用 2.1 节那个记法来看就更清楚了：注意力里 $B$ 是**批维度**（在两个操作数里都出现、还留在结果里），
形如 $\underline{B}\underline{D} \times \underline{B}\underline{D} \to B$。这**本质上就是在做一堆点积**，
而点积的算术强度是出了名的差 —— 每条序列各做各的，做再多份也不会让强度变高。

### 3.6 小结

<Theorem type="proposition" name="四个格子">

| | MLP | 注意力 |
| --- | --- | --- |
| **prefill** | $B \cdot S$ —— 很好 | $S / 2$ —— 没那么好但可用 |
| **生成** | $B$ —— 可用，但要靠并发请求 | 小于 1 —— **无法改善** |

**prefill 是算力受限的，生成是内存受限的。**

而右下角那一格是根本性的瓶颈：**只要你还在用 Transformer，这一格就没法改善。**
</Theorem>

Percy 说：以后再听到有人讲「推理是内存受限的」，你就知道这句话具体指的是什么了。

<Callout type="tip" title="动手：把这四个格子拖出来看">

$6BTDF / (4BTD + 4BTF + 6DF)$，茜红是注意力的精确强度 $S \cdot T / (S+T)$，
黄色横线是加速器强度（H100 是 295，B200 是 281）。形状取 Llama 2 13B（$D = 5120$，$F = 13824$）。

## 四、吞吐与延迟：把数字真的算出来

上一节证明了推理是内存受限的。这一节算一下**单个请求在理论上能达到的最好延迟和吞吐**。

假设：**计算和通信可以完美重叠，忽略一切开销。** 在这个假设下事情变得很简单 ——
**要算一件事花多久，只要看它要搬多少内存。**

Percy 说这在某种意义上是好事（简单），在另一种意义上很让人恼火（你的加速器就杵在那儿什么也不干）。

### 4.1 四个统计量的公式

```python
# ── 参数量 ──
# 2*V*D      ：输入嵌入 + 输出嵌入（各 V×D）
# D*F*3*L    ：每层 MLP 的三个矩阵（up / gate / down），共 L 层
# 2*D*N*H*L  ：每层的查询投影和输出投影（各 D×N×H）
# 2*D*K*H*L  ：每层的键投影和值投影（各 D×K×H）—— 注意这里是 K 不是 N
num_params = 2*V*D + D*F*3*L + (2*D*N*H + 2*D*K*H)*L

# 参数占多少字节：bf16 每个数 2 字节（训练时要乘更大的倍数，因为还有梯度和优化器状态）
parameter_size = 2 * num_params

# 每条序列的 KV cache：S 个 token × K 个头 × H 维 × L 层
#   × 2（键一份、值一份）× 2（bf16 字节数）
kv_cache_size_per_seq = S * (K*H) * L * 2 * 2

# 总显存占用 = B 条序列的 KV cache + 参数
memory = B * kv_cache_size_per_seq + parameter_size

# 延迟由内存 IO 决定：每生成一个 token，要把全部参数和全部 KV cache 读一遍
latency = memory / memory_bandwidth

# 吞吐是延迟的倒数，但因为我们是 B 条序列并行生成，所以还要乘 B
throughput = B / latency
```

要理解的两件事：

- **延迟公式为什么长这样**：生成一个 token 时，你必须把**所有参数**读一遍（每层每个矩阵都要用到），
  还要把**所有 KV cache**读一遍（注意力要和历史里每个位置比对）。因为是内存受限的，
  搬这些字节的时间就是这一步的时间。
- **吞吐为什么是 $B / \text{延迟}$**：这一步搬完之后，$B$ 条序列**各自**都往前推进了一个 token。
  所以一次「延迟」换来了 $B$ 个 token。

Llama 2 13B 的配置：

```python
def llama2_13b_config():
    return {
        S: 1024,   # 序列长度
        D: 5120,   # 模型维度
        F: 13824,  # 前馈（MLP）维度
        N: 40,     # 查询头数
        K: 40,     # 键值头数 —— 这里没有用 GQA，所以 K = N
        H: 128,    # 头维度
        L: 40,     # 层数
        V: 32000,  # 词表大小
        memory_bandwidth: 3.35e12,  # H100 的显存带宽（字节/秒）
    }
```

代进去：

- $\text{num\_params} \approx 1.30 \times 10^{10}$，也就是 **130 亿**。
  这是个好的自洽检查 —— 这个模型对外宣称就是 130 亿参数。
- 参数占用 $= 2 \times 1.30\times10^{10} \approx 26.0$ GB。
- 每条序列的 KV cache $= 1024 \times (40 \times 128) \times 40 \times 2 \times 2 \approx 838.9$ MB。
- 显存 $\approx 838.9\ \text{MB} \times B + 26.0\ \text{GB}$，是一个 **$B$ 的线性函数**。
- 延迟 = 上式除以 $3.35 \times 10^{12}$，形状完全一样。
- 吞吐 = $B$ 除以延迟。

所以只要记住这两句：

<Theorem type="proposition" name="记住这两个形状">
**延迟是 $B$ 的线性函数**：斜率是每条序列的 KV cache 大小，截距是参数大小。

**吞吐正比于 $\dfrac{B}{B + \text{常数}}$**：随 $B$ 上升，但会**趋于饱和**。
</Theorem>

### 4.2 代三个批大小进去

| 批大小 $B$ | 总显存 | 延迟（秒/token） | 吞吐（token/秒） |
| --- | --- | --- | --- |
| 1 | 26.9 GB | 0.0080 | 125 |
| 64 | 79.7 GB | 0.0238 | 2 690 |
| 256 | 240.8 GB —— **装不下** | 0.0719 | 3 562 |

逐行读：

- **$B = 1$**：延迟 0.008 秒/token，吞吐约 125 token/秒。
- **$B = 64$**：**延迟变差了三倍**（0.008 → 0.0238），但**吞吐涨了 21 倍**（125 → 2690）。
- **$B = 256$**：延迟继续变差（0.0719），吞吐继续涨但只涨到 3562 —— 相比 $B = 64$ 只多了 32%，
  而 $B$ 却翻了 4 倍。**收益已经明显递减。** 更要命的是，
  **显存 240.8 GB 远超 H100 的 80 GB，根本装不下。**

这里出现了一个很有意思的现象：我们直觉上想「让它变快」，但**「快」在这里有两个含义，
而且它们的方向完全相反**。你调批大小这一个旋钮，就是在这两者之间选边站。

<Callout type="insight" title="增大批大小到底做了两件什么事">
1. **延迟变差**：因为 KV cache 的大小是 $O(B)$ 的，每一步要读写的 KV cache 更多了。
   而且它是批处理的 —— 一条查询要等这一批里所有序列都算完这一步，才能拿到自己的那个 token。
2. **吞吐变好**：因为**参数是所有序列共享的**。26 GB 的参数只读一遍，
   却推进了 $B$ 条序列各一个 token —— 读参数的成本被摊薄了。

<Theorem type="proposition" name="延迟与吞吐的取舍">
- **小批**：延迟好，吞吐差。
- **大批**：吞吐好，延迟差。
</Theorem>

而且吞吐有一个**硬上限**：当 $B$ 大到 KV cache 完全压过参数时，
吞吐趋近于 $\text{带宽} / \text{每条序列的 KV cache} = 3.35\times10^{12} / 8.389\times10^{8} \approx 3993$ token/秒。
你永远碰不到这个渐近线，因为在那之前显存就先满了。换成 B200（192 GB）能把批大小再推高一些，
但同样会撞到某个上限。
</Callout>

### 4.3 两句补充：并行与 TTFT

**并行**这一讲不展开（想深入可以看 Scaling Book 的推理章），但两个层次值得知道：

- **最简单的并行**：**启动 $M$ 份模型副本**。延迟不变，吞吐乘以 $M$。
- **更难的并行**：把**模型和 KV cache 分片**到多个设备上。

**TTFT（首 token 时间）**：本质上就是 **prefill 花的时间** —— prefill 做完就可以开始生成了。
由此得到一条很实用的策略：

<Theorem type="proposition" name="两个阶段用不同的批大小">
- **prefill 阶段用较小的批** → 更快的 TTFT。
- **生成阶段用较大的批** → 更好的吞吐。
</Theorem>

<Callout type="tip" title="动手：把批大小拖一遍">
下面这个演示就是 4.1 节那段代码，参数写死成 Llama 2 13B。
靛蓝是延迟（左轴），茜红是吞吐（右轴），灰色阴影区是**显存装不下**的批大小。

几件值得动手试的事：

- 把 $B$ 从 1 拖到 1024，对着表核对 4.2 节那三行数字（$B = 1 / 64 / 256$）。
- 注意 $B = 32$ 附近：**KV cache 第一次超过参数本身**（$32 \times 838.9\ \text{MB} = 26.8$ GB
  对 26.0 GB）。从这里往右，延迟就开始明显恶化了。
- 把序列长度 $S$ 从 1024 调到 4096，看装不下的区域怎么往左扑过来 ——
  KV cache 是正比于 $S$ 的。
- 「KV 头数 K」那几个按钮先别动，第五节讲 GQA 时会回来用它。

</Callout>

## 五、有损捷径之一：把 KV cache 变小

现在我们有了一套思考推理效率的框架。接下来的问题是：**怎么让它更快？**
办法非常多样，从改模型架构一直到系统层面的优化都有 —— 推理其实是一个横跨很多层的话题。

第一个、也是现在回头看最显然的一个方向：

> **内存是推理的瓶颈，而 KV cache 占了大量内存**（批一大甚至超过参数本身）。
> **那就把 KV cache 做小。**

但要小心：这些做法都是**有损的**，必须检查**准确率有没有掉太多**。

### 5.1 分组查询注意力（GQA）

<Figure src="/courses/cs336/l10-gmqa.png" wide caption="三种注意力的键值共享方式。左：多头注意力，每个查询头有自己的键和值。中：分组查询注意力，一组查询头共用一份键值。右：多查询注意力，所有查询头共用同一份键值。来源：Ainslie et al. 2023（GQA 论文），转引自 CS336 第 10 讲课件" />

想法很简单：**查询头照旧有 $N$ 个，但键值头只有 $K$ 个**，每个键值头被 $N/K$ 个查询头共用。

- **多头注意力（MHA）**：$K = N$，完全不压缩。
- **多查询注意力（MQA）**：$K = 1$，压到极致。Percy 说**基本没人用**，因为效果确实差。
- **分组查询注意力（GQA）**：$K$ 取中间某个值，希望在准确率和速度之间找到平衡。

<Figure src="/courses/cs336/l10-gqa-speed.png" caption="每个样本的推理耗时随 GQA 组数的变化。粉色虚线是 MHA（约 2.5 秒），橙色虚线是 MQA（约 0.5 秒）。蓝色是 GQA：组数从 1 到 8 几乎贴着 MQA，16 和 32 开始爬升，到 64（也就是退化成 MHA）就冲到顶。来源：Ainslie et al. 2023，转引自 CS336 第 10 讲课件" />

**为什么 GQA 能同时改善延迟和吞吐？** 因为它把 KV cache 缩小了 $N/K$ 倍。
而**减少内存占用就等于提速** —— 我们是内存受限的。

回到 Llama 2 13B，把上一节的公式重跑一遍（H100，序列长 1024）：

| 配置 | 参数量 | 每条序列 KV cache | 总显存 | 延迟（秒/token） | 吞吐（token/秒） |
| --- | --- | --- | --- | --- | --- |
| $K = 40$（MHA），$B = 64$ | 13.0 B | 838.9 MB | 79.7 GB | 0.0238 | 2 690 |
| $K = 8$（GQA 1:5），$B = 64$ | 11.3 B | 167.8 MB | 33.4 GB | 0.0100 | 6 417 |
| $K = 8$（GQA 1:5），$B = 256$ | 11.3 B | 167.8 MB | 65.6 GB | 0.0196 | 13 068 |

读这张表要注意三件事：

1. **第一行 → 第二行：延迟和吞吐同时变好。** 延迟从 0.0238 降到 0.0100，吞吐从 2690 涨到 6417。
   所以**延迟和吞吐并不是永远对立的** —— 只要你真的**减少了内存**，两者都受益。
   **真正制造对立的是批大小这一个维度。**
2. **参数量几乎没动**（13.0 B → 11.3 B），**动的全是 KV cache**（838.9 MB → 167.8 MB，正好 5 倍）。
3. **第二行 → 第三行**：既然省出了显存，那就把批大小推上去。$B = 256$ 时 MHA 版本装不下（240.8 GB），
   GQA 版本只要 65.6 GB，**装得下**；代价是延迟回落到 0.0196，收获是吞吐再翻一倍到 13 068。

<Callout type="insight" title="这些旋钮是一起转的">
上面这三行演示了一个很典型的操作模式：**先压缩 KV cache，省出来的显存再拿去加大批大小**，
从而在延迟和吞吐之间重新做一次取舍。单独看任何一个旋钮都会漏掉这个组合玩法。

（第四节那个演示顶部的「KV 头数 K」按钮就是干这个的：选 $K = 8$，
你会看到装不下的灰色区域一下子往右退了一大截。）
</Callout>

**最后必须做的一步：检查准确率。**

<Figure src="/courses/cs336/l10-gqa-accuracy.png" wide caption="GQA 论文的准确率对照。MHA-XXL 推理耗时 1.51 秒、平均分 47.2；GQA-8-XXL 只要 0.28 秒、平均分 47.1 —— 快 5 倍多而几乎不掉分。MQA-XXL 更快（0.24 秒）但平均分掉到 46.6。来源：Ainslie et al. 2023，转引自 CS336 第 10 讲课件" />

<Callout type="warning" title="准确率评测要打折看">
Percy 特别提醒：这类「准确率没掉」的结论**永远要打个折扣**。
这是**在某一个特定模型上**测出来的；后面 DeepSeek 那篇论文就会给出相反的证据 ——
GQA **确实会掉分**。

他的原话大意是：**除了纯数学推导出来的东西，其余一切都要留一分怀疑。**
</Callout>

### 5.2 多头潜在注意力（MLA）

<Figure src="/courses/cs336/l10-mla-schema.png" wide caption="四种注意力的对比。斜纹填充的方块表示「推理时要缓存的东西」。MHA 缓存 N 份键值，GQA 缓存 K 份，MQA 缓存 1 份，而 MLA 只缓存一个压缩后的潜向量，用到时再投影出键和值。来源：DeepSeek-V2 论文，转引自 CS336 第 10 讲课件" />

DeepSeek 提出的思路和 GQA 完全不同：**键值头的数量不减，减的是每个 token 存下来的维度。**

回忆一下正常注意力怎么算键和值：拿到激活 $h$，乘一个矩阵得到 $K = W_K h$，
再乘另一个矩阵得到 $V = W_V h$。这两个东西的维度都是 $N \cdot H$，也就是模型维度那个量级，**很大**。

MLA 的做法是：

$$
c = W_c h \quad (\text{压缩到 } C \text{ 维}), \qquad K = W_K\, c, \quad V = W_V\, c
$$

**只把 $c$ 存进 KV cache**（$C$ 维，小得多），需要键和值的时候**现场投影出来**。

具体到 DeepSeek-V2：**把 $N \cdot H = 16384$ 维压到 $C = 512$ 维** —— 这是相当激进的压缩。

<Callout type="note" title="补充：MLA 和 RoPE 不兼容，怎么办">
有一个别扭的地方：**MLA 和 RoPE（旋转位置编码）不兼容**。
RoPE 是直接作用在键和值上的 —— 它按 token 在序列里的位置对键值向量做旋转。
可是 MLA 缓存的是压缩后的 $c$，键值是事后才投影出来的，位置信息没法这样加进去。

DeepSeek 的解决办法是**额外留出 64 个维度专门处理 RoPE**，
所以实际缓存的是 $512 + 64 = 576$ 维。相比 16384 仍然是巨大的压缩。
</Callout>

延迟和吞吐的改善就是前面那套数学的直接推论 —— **KV cache 越小越快**，
在撞到某个点之前几乎是线性的。

那准确率呢？两张表：

<Figure src="/courses/cs336/l10-mla-accuracy.png" wide caption="DeepSeek-V2 论文表 8：同为 70 亿参数的稠密模型，MHA 在难 benchmark 上明显强于 GQA 和 MQA（MMLU 45.2 对 41.2 对 37.9）。这和 GQA 论文的结论是有张力的。来源：DeepSeek-AI 2024，转引自 CS336 第 10 讲课件" />

第一个结论：**MHA 比 GQA 好**（虽然更贵）。注意这**和 5.1 节 GQA 论文的结论相冲突** ——
这正是上面那个「打折看」提醒的实例。

<Figure src="/courses/cs336/l10-mla-accuracy2.png" wide caption="DeepSeek-V2 论文表 9：MoE 模型上 MLA 与 MHA 的对比。大 MoE 那一组，每 token 的 KV cache 从 860.2K 个元素降到 34.6K（约 25 分之一），而 MMLU 从 57.5 涨到 59.0。来源：DeepSeek-AI 2024，转引自 CS336 第 10 讲课件" />

第二个结论：**MLA 甚至比 MHA 还略好一点**（就算保守地说，也是「差不多」），
**而且便宜得多** —— 大 MoE 那一组 KV cache 小了约 25 倍。

<Callout type="note" title="补充：课堂问答 —— 为什么不干脆把模型维度调小">
有同学问：把 KV cache 压缩，和干脆把模型维度 $D$ 调小，这两者相比如何？

Percy 说这些消融实验里没有直接对比，但他的猜测是：**直接减小模型维度会更差**，
因为那是**不加区分地把所有东西一起缩小**。

而这一整类工作的诀窍在于：**找到模型里那些「可以被挤压」的地方**。
哪里可以挤，先验上说不清楚，**只能靠大量实验试出来**。
</Callout>

### 5.3 跨层注意力（CLA）

<Figure src="/courses/cs336/l10-cla-diagram.png" caption="左：传统 Transformer，每一层都有自己的键值投影。右：跨层注意力，只有部分层计算键值，其余层直接复用下面那一层的 KV cache（红线）。来源：Brandon et al. 2024，转引自 CS336 第 10 讲课件" />

又一个压缩 KV cache 的思路，而且它挑的是一个全新的维度：

> **GQA 是在「头」这个维度上共享键值，CLA 是在「层」这个维度上共享键值。**

具体做法：只让**一部分层**真的计算键和值，其余的层**直接用前一层的 KV cache**。
如果两层共用一份，KV cache 就减半。

<Figure src="/courses/cs336/l10-cla-results.png" caption="10 亿参数模型上的帕累托前沿：横轴是每 token 的 KV cache 字节数（对数轴），纵轴是验证困惑度（越低越好）。红点是加了 CLA 的模型，蓝点是没加的。在同样的 KV cache 预算下，红点普遍比蓝点低。来源：Brandon et al. 2024，转引自 CS336 第 10 讲课件" />

怎么读这张图：对任何一种方法，你都可以通过调 $K$ 和头维度来**扫出一条「KV cache 大小 vs 困惑度」的曲线**。
论文的结论是，**加上 CLA 之后这条帕累托前沿整体向左下移动了** ——
同样的 KV cache 预算下困惑度更低，或者同样的困惑度下 KV cache 更小。

### 5.4 局部（滑动窗口）注意力

<Figure src="/courses/cs336/l10-longformer-attention.png" wide caption="四种注意力模式。(a) 完整的平方注意力；(b) 滑动窗口 —— 每个 token 只看最近的 k 个；(c) 空洞滑动窗口 —— 窗口里跳着取；(d) 全局 + 滑动窗口 —— 少数固定位置能被所有位置看到，其余走局部窗口。来源：Longformer（Beltagy et al. 2020），转引自 CS336 第 10 讲课件" />

这是一个相当老、也相当自然的想法（Sparse Transformer 2019、Longformer 2020、Mistral 7B 2023）：

> **完整注意力矩阵是 $n^2$ 的。不如生成每个 token 时只看最近的 $k$ 个 token。**

这样做的收益非常大：

<Theorem type="proposition" name="滑动窗口的两个性质">
1. **KV cache 与序列长度无关。** 它只取决于窗口大小 $k$、批大小、层数等，
   **和 $S$ 脱钩了** —— 这对长上下文是极大的好消息。
2. **有效上下文随层数线性增长。** 虽然每一层只看 $k$ 个 token，但信息可以逐层往回传播：
   第 1 层看到 $k$ 个，第 2 层通过它们又间接看到更远的 $k$ 个……
   所以**实际有效的上下文长度比标称的窗口长得多**。
</Theorem>

还可以玩花样：层与层之间不必都用密集窗口，可以**空洞式地拉开**（图 c）；
也可以**全局 + 滑动窗口**混用（图 d）—— 让一小撮固定位置能被所有位置看到，其余走局部窗口。

**问题是：这仍然会掉准确率。** 它减少了表达力，**没有免费的午餐**
（Percy 的说法是「至少这顿午餐是收费的」）。

**解决办法：把局部注意力和全局注意力交错起来。** 现在的**混合（hybrid）模型**通常
一部分层用完整注意力、另一部分层用局部注意力，在准确率和速度之间来回找平衡点。

<Callout type="note" title="补充：课堂问答 —— 线性注意力 vs 滑动窗口">
有同学问：线性注意力（linear attention）和滑动窗口注意力该怎么选？

**线性注意力是什么**：不存 KV cache，而是把**整段历史压缩成一个固定大小的表示**。
最朴素的版本就是**把所有的键值外积直接加起来变成一个矩阵/向量** ——
显然它和序列长度无关。更讲究的版本有 **Mamba** 和 **Gated DeltaNet**，
它们试图在压缩的同时**尽量少遗忘**。

**怎么比较**：两者都被用来替代完整注意力，都有人跑出好结果。而且它们**捕捉的东西不一样**：

- 关心**局部的高分辨率细节** → 滑动窗口更好。
- 只需要**对过去的大致概括** → 线性注意力更好。

所以也可以三者混用：完整注意力 + 滑动窗口 + 线性注意力。

**长上下文场景下呢？** 追问的同学想知道线性注意力是不是长上下文的更好选择。
Percy 的回答是：**没有免费的午餐。** 如果你在做「大海捞针」这类任务，
而你把整段历史压进了一个很小的状态里，**信息就是丢了，你捞不回来**。

但他补充了一个判断：**Mamba / DeltaNet 这类比滑动窗口「更有潜力」。**
理由是 Mamba 在做递归时可以只看最近的状态，**所以它至少能表示滑动窗口能表示的东西**；
而滑动窗口一旦定下来，**就到头了，没有别的余地了**。
</Callout>

### 5.5 DeepSeek v4 的注意力

<Figure src="/courses/cs336/l10-deepseek-v4-attention.png" wide caption="DeepSeek 的稀疏注意力结构。左下：token 级压缩器把 KV token 压成更少的条目；中间：top-k 选择器只保留最相关的一部分；左上：另外还并上一段滑动窗口的 KV；右边虚线框：lightning indexer 用轻量的查询和键算出 index score，负责快速决定该留哪些。来源：DeepSeek，转引自 CS336 第 10 讲课件" />

DeepSeek 在注意力机制上持续折腾。继 MLA 之后，v4 为了支持 **100 万 token 的上下文**又加了几层：

- **压缩稀疏注意力（CSA）**：**把每 $m$ 个 token 压缩成 1 个**。
- **DeepSeek 稀疏注意力（DSA）**：在压缩后的条目里**只选 top-$k$**。
  选法是：另外算一套**轻量级的查询和键**，用它们做一次小规模注意力得到 index score，
  据此判断该保留哪些 token。Percy 形容这是「一种闪电般快速地判断该留哪些 token 的办法」。
- **高度压缩注意力（HCA）**：压得更狠。

### 5.6 本节小结

<Theorem type="proposition" name="压缩 KV cache 这条线">
**目标**：因为推理是内存受限的，而 KV cache 是内存大头，所以把 KV cache 做小 ——
**同时不能伤准确率**。

**手段**：

- **降低 KV cache 的维度**：GQA（在头上共享）、MLA（压缩成潜向量）、CLA（在层上共享）。
- **截断 KV cache**：局部 / 滑动窗口注意力，通常只用在一部分层上。
- **换掉 KV cache**：线性注意力 / 状态空间模型（Mamba 2、Gated DeltaNet）。
- **换掉自回归本身**：**扩散模型**是一种非自回归的生成方式，可能快得多。

</Theorem>

## 六、有损捷径之二：量化

第五节动的是架构，这一节动的是**数字本身的表示**，属于系统那一侧。

**核心想法：降低数的精度。** 内存变少 → 延迟和吞吐都变好（因为我们是内存受限的）。
当然，还是要担心准确率。

### 6.1 量化的机制

量化就是把一个浮点数映射成一个小整数，再在需要时映射回来。需要两个参数：
**缩放系数（scale）**和**零点（zero point）**。

```python
x = 5.2342
scale = 0.1        # 每一格代表 0.1
zero_point = 4     # 整数 4 对应实数 0

x_quant  = round(x / scale) + zero_point   # 量化：round(52.342) + 4 = 56
x_approx = (x_quant - zero_point) * scale  # 反量化：(56 - 4) × 0.1 = 5.2
```

原来的 5.2342 变成了 5.2 —— **误差 0.0342，这就是量化的代价**。
误差的大小由 `scale` 决定：`scale` 越小越精确，但**能表示的范围也越窄**
（因为整数的位数是固定的，比如 int8 只有 $-128$ 到 $127$ 共 256 个格子）。
所以**怎么给每一层 / 每一个张量挑 scale 和 zero point，就是量化算法的主要工作。**

### 6.2 有哪些格式

<Figure src="/courses/cs336/l10-fp-formats.png" caption="几种数值格式的位分布：深绿是符号位，中绿是指数位，浅绿是尾数位。位数越少，能表示的数越少、越粗糙。来源：Baseten 博客，转引自 CS336 第 10 讲课件" />

- **fp32（4 字节）**：训练时的参数和优化器状态需要它。
- **bf16（2 字节）**：**推理的默认选择**，这一讲前面所有的核算都是按 bf16 算的。
- **fp8（1 字节）**：H100 上的 e4m3 格式表示范围是 $[-240, 240]$。**如果你胆子够大，可以直接拿它训练。**
- **int8（1 字节）**：范围 $[-128, 127]$。**比 fp8 更不准，但更便宜，只用于推理。**
- **int4（0.5 字节）**：范围 $[-8, 7]$。更便宜，也更不准。

从 bf16 降到 int4，KV cache 和参数都**小 4 倍**。按第四节的公式，这直接意味着延迟约 4 倍的改善。

### 6.3 什么时候做量化：QAT 与 PTQ

**做法一：量化感知训练（quantization-aware training, QAT）。**
如果你担心量化会把模型搞坏，那就**训练时就把量化考虑进去**：
在前向传播里做一次「量化再反量化」，**模拟出量化误差**，让梯度看到它。

- **优点**：权重是**朝着「将来会被量化」这个前提训出来的**，效果自然更好。
- **缺点**：**需要昂贵的大规模训练**。

**做法二：训练后量化（post-training quantization, PTQ）。**
这是实践中更常见的做法 —— 先正常训练，训完之后再量化，**便宜得多**。

- **朴素版**：拿一小批样本数据跑一遍，**为每一层或每一个张量确定 scale 和 zero point**，
  然后各自独立地量化。这个做法**一般效果不够好**。
- **GPTQ**：**利用 Hessian（二阶导）信息**，逐层量化，并且**把已经产生的量化误差记录下来、
  传播到还没被量化的权重上去**，让后面的权重**主动补偿前面的误差**。

### 6.4 激活感知量化（AWQ）

<Figure src="/courses/cs336/l10-awq-schema.png" wide caption="AWQ 的三种做法对比。(a) 直接四舍五入量化到 int3，困惑度 43.2 —— 坏掉了。(b) 根据激活挑出 1% 的重要权重，保持 fp16，困惑度回到 13.0，但混精度硬件效率差。(c) 量化前先按激活幅度缩放权重，全部 int3，困惑度同样是 13.0。来源：Lin et al. 2023，转引自 CS336 第 10 讲课件" />

AWQ 的出发点是一个经验观察：

> **有些激活通道（activation channel）的数值特别大，而与这些通道相乘的权重更重要。**
> 那就**给这些权重分配更多精度**。

具体做法：根据激活的统计量挑出 **0.1% 到 1% 的权重**保持高精度，其余的量化到低精度。
效果是 **fp16 → int3，内存降到四分之一，速度提升 3.2 倍**。

<Callout type="note" title="补充：量化对准确率的影响到底有多大">
初学者最容易糊的一点是「量化到底会不会把模型搞坏」。上面这张 AWQ 图给了非常直白的答案，
三个数字放在一起看：

- **直接四舍五入量化到 int3**：困惑度 **43.2**。这个模型基本上废了 ——
  回忆第 1 讲，困惑度可以粗略理解成「每一步平均在多少个候选之间犹豫」，
  从十几跳到四十多是灾难性的退化。
- **只把 1% 的权重留在 fp16**：困惑度回到 **13.0**。
- **量化前先按激活幅度缩放，然后全部 int3**：困惑度同样是 **13.0**，
  而且**不需要混精度**，硬件跑起来更规整。

所以结论不是「量化有害」也不是「量化无害」，而是：
**同样是 3 比特，做法不同，结果可以是废模型和好模型的区别。**
</Callout>

## 七、有损捷径之三：剪枝与蒸馏

**核心想法：直接把一个昂贵模型的一部分撕下来扔掉，然后再把它修好。**
听起来很粗暴，但确实管用。

<Figure src="/courses/cs336/l10-pruning-kd-loop.png" wide caption="NVIDIA 的剪枝-蒸馏循环：1. 拿一个训好的大模型；2. 在校准集上估计每个嵌入维度 / 注意力头 / 隐藏通道 / 层的重要性；3. 按重要性排序；4. 裁掉不重要的部分；5. 用原模型蒸馏修复裁剪后的模型。整个过程可以迭代。来源：Muralidharan et al. 2024（NVIDIA），转引自 CS336 第 10 讲课件" />

算法三步：

1. **在一个很小的校准数据集（1024 个样本）上，判断哪些「层 / 注意力头 / 隐藏维度」是重要的。**
2. **把不重要的部分删掉**，得到一个更小的模型。此时这个模型**并不好用**。
3. **把原模型蒸馏进这个剪枝后的模型**，也就是拿你关心的数据或任务**再训一会儿，把它「治好」**。

<Figure src="/courses/cs336/l10-pruning-kd.png" caption="结果：横轴是训练这个模型所花的 token 数（万亿），纵轴是 MMLU 分数。绿色箭头是剪枝路径 —— 从 Nemotron-4 15B 剪成 Minitron 8B，分数从 65.1 掉到 63.8，而所花的 token 数从 8 万亿降到约 0.1 万亿。相比同等大小的 Nemotron-3 8B，是「便宜 40 倍、好 9%」。来源：Muralidharan et al. 2024，转引自 CS336 第 10 讲课件" />

具体结果：**150 亿参数的模型剪成 80 亿参数**，准确率没掉多少，
而**用掉的训练量比从头训一个 80 亿模型少得多**。

从某种角度看，这是一条「训练侧」的路子：你还是要训，只是**用一个好模型的零件来初始化**，
而不是从随机初始化开始。

<Callout type="note" title="补充：课堂问答 —— 怎么判断哪些部分重要">
有同学问：怎么区分重要的层和不重要的层？

**基本做法**：准备一个**校准集**，把输入喂进模型，**观察激活的幅度**。
有些单元的激活接近零（所谓「死单元」），那就是可以删的；激活大的就留下。

**追问一**：激活大凭什么就重要？万一它一直都很大呢？

Percy 说这是个**经验观察** —— 实际情况就是有些通道的激活会比其他通道大很多。
如果这条不成立，这些技术就不会奏效；但它恰好成立，那就利用它。
（他补充说，这大概是这些模型训练出来的方式导致的。）

**追问二**：假设某个神经元在所有样本上都恒定输出 100，这算重要吗？

Percy 的回答：如果确实是恒定的，那你**不能直接删掉它**（删了后面的东西全乱套）。
但你可以**同时看方差**：**均值高、方差低**的单元，说明它只是在提供一个常数偏移 ——
那就有办法**把它折进偏置项里**，然后把这个单元本身删掉。
</Callout>

### 7.1 把三条有损路线合起来看

<Theorem type="proposition" name="减少推理复杂度而不伤准确率的两个配方">
所有这些手段其实都在做同一件事：**减少参数量或 KV cache**。做法上分两条路：

**从零配方（from scratch）**

1. 定义一个更快的模型架构（GQA / MLA / CLA / 局部注意力 / 更低精度……）。
2. 训练这个更快的模型。

**蒸馏配方（distillation）**

1. 定义一个更快的模型架构。
2. **用原模型的权重来初始化它**（尽管两者架构不一样，硬拼出一个「弗兰肯斯坦」）。
3. **用蒸馏把这个更快的模型修好。**

</Theorem>

## 八、无损捷径：投机采样

到目前为止的所有招数都是**有损的** —— 把 KV cache 压狠了、精度降低了、模型剪掉了一块，
都有可能伤到准确率。这一节的方法很优雅，因为它是**无损的**：
**输出的分布和不做任何优化时一模一样。**

### 8.1 出发点：检查比生成快

再回忆一次两个阶段：

- **prefill**：给一段序列，**并行地**编码所有 token。**算力受限**，很快。
  而且很关键的一点是 —— **它顺带给出了每个位置上的概率分布。**
- **生成**：一次一个 token。**内存受限**，慢。

把这两句放在一起就得到一个不对称性：

<Theorem type="proposition" name="核心不对称性">
**检查（checking）比生成（generation）快。**

给你一串已经写好的 token，让模型说「这串写得好不好」——
这只要一次并行的前向传播；比让模型一个一个把它们生成出来快得多。
</Theorem>

### 8.2 做法

**用一个便宜的草稿模型（draft model）$p$ 猜出接下来的几个 token**（比如 4 个），
**再用我们真正关心的目标模型（target model）$q$ 并行地审阅这几个 token，逐个决定接受还是不接受。**

这个设计里两边的成本是**刻意配平**的：

- **草稿模型小而便宜**，所以尽管它也是内存受限、也要一个一个生成，**代价不大**。
- **目标模型大而昂贵**，但我们让它**并行处理一批 token**（而不是一个），
  所以它这一次前向的算术强度高得多，**代价同样不大**。

Google 的演示视频里能直接看出效果：用大模型逐 token 生成非常慢；
换成投机解码后，画面上是**小模型先刷出一串 token，大模型再一次性审阅** ——
于是 token 是**一簇一簇（burst）地冒出来**的。

### 8.3 算法与接受规则

<Figure src="/courses/cs336/l10-speculative-sampling-algorithm.png" wide caption="投机采样算法。上半部分：用草稿模型 p 自回归地采样 K 个 token，再用目标模型 q 一次并行算出 K+1 组 logits。下半部分：逐个判断，采样 r ~ U[0,1]，若 r 小于 min(1, q/p) 则接受，否则从残差分布 (q − p)₊ 中采样一个 token 并跳出循环。若 K 个全部被接受，则额外从 q 采样一个 token。来源：Chen et al. 2023，转引自 CS336 第 10 讲课件" />

一轮的完整流程：

1. **草稿阶段**：用草稿模型 $p$ **自回归地**采样 $K$ 个 token $\tilde{x}_1, \dots, \tilde{x}_K$。
2. **审阅阶段**：用目标模型 $q$ **一次并行前向**，算出 $K+1$ 组 logits ——
   分别是「在原始上下文之后」「在原始上下文 + $\tilde{x}_1$ 之后」……
   「在原始上下文 + $\tilde{x}_1 \dots \tilde{x}_K$ 之后」的分布。
3. **判定阶段**：从 $t = 1$ 到 $K$ 逐个判断。采样 $r \sim U[0,1]$：
   - 若 $r < \min\left(1, \dfrac{q(\tilde{x}_t \mid \cdot)}{p(\tilde{x}_t \mid \cdot)}\right)$ → **接受**这个 token，继续看下一个。
   - 否则 → **拒绝**，从**残差分布** $(q - p)_+$ 中采样一个 token 顶上，**然后跳出循环**
     （后面的草稿 token 全部作废，因为上下文已经和草稿假设的不一样了）。
4. **奖励 token**：如果 $K$ 个草稿 token **全部被接受**，就额外从 $q$ 采样一个 token。
   所以最好情况下一轮能推进 $K+1$ 个 token。

<Callout type="note" title="补充：接受和拒绝规则，走一遍具体数字">
这条规则是这一节最容易糊的地方，用一个具体例子走一遍。

设 $K = 4$，草稿模型猜出四个 token：`mat` / `,` / `and` / `looked`。
目标模型一次前向就给出了每个位置上它自己的概率。逐个判定：

| 位置 | 草稿猜的 token | 草稿概率 $p$ | 目标概率 $q$ | 接受概率 $\min(1, q/p)$ | 掷出的 $r$ | 结果 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | `mat` | 0.30 | 0.60 | $\min(1, 2.0) = 1$ | 0.91 | **接受** |
| 2 | `,` | 0.50 | 0.40 | $\min(1, 0.8) = 0.8$ | 0.35 | **接受** |
| 3 | `and` | 0.40 | 0.05 | $\min(1, 0.125) = 0.125$ | 0.70 | **拒绝** |
| 4 | `looked` | — | — | — | — | 作废 |

读这张表：

- **位置 1**：目标模型比草稿模型**更**看好 `mat`（0.60 对 0.30），比值大于 1，
  **无条件接受** —— 草稿低估了它，接受不会有任何偏差问题。
- **位置 2**：目标模型比草稿模型**略微**不看好 `,`（0.40 对 0.50），
  于是**以 80% 的概率接受**。这次 $r = 0.35$，小于 0.8，接受。
- **位置 3**：目标模型**强烈**不认可 `and`（0.05 对 0.40），只有 12.5% 的概率接受。
  这次 $r = 0.70$，**拒绝**。此时**不是丢掉这一步**，而是从**残差分布**
  $(q - p)_+$（即逐个 token 取 $\max(q - p, 0)$ 再归一化）中采一个 token 放在这个位置。
- **位置 4** 及以后：全部作废。因为位置 3 的 token 已经变了，草稿在位置 4 的猜测所基于的上下文不再成立。

**这一轮的收获**：3 个 token（2 个接受 + 1 个从残差补上）。
**这一轮的代价**：草稿模型 4 次小前向 + 目标模型 **1 次**并行前向。
如果不用投机解码，同样 3 个 token 需要目标模型做 **3 次**串行前向。

**为什么一定至少能推进 1 个 token**：即使位置 1 就被拒绝，你也会从残差分布里采到一个 token。
这正是它和普通拒绝采样的区别 —— 普通拒绝采样被拒时**什么都拿不到、要重新循环**，
而这里做了修改，**保证每一轮至少产出一个 token**。
</Callout>

### 8.4 为什么结果是精确的

这套流程是**以 $p$ 为提议分布、$q$ 为目标分布的（修改版）拒绝采样**。
关键性质是：**采出来的 token 严格服从目标模型 $q$ 的分布。**

课件给了一个两元素词表的例子，直接把这件事验算出来。设词表只有 $\{A, B\}$：

- 目标模型的概率是 $[q(A), q(B)]$，草稿模型的概率是 $[p(A), p(B)]$。
- 假设 $p(A) > q(A)$（**草稿模型高估了 A**）。
- 那么必然 $p(B) < q(B)$（**草稿模型低估了 B**），因为两边都要归一化到 1。
- 残差分布 $\max(q - p, 0)$ 在 $A$ 上是 0，在 $B$ 上是正的 ——
  归一化之后就是 $[0, 1]$，也就是**残差分布必定给出 B**。

现在算最终采到每个 token 的概率：

$$
P[\text{采到 } A] \;=\; \underbrace{p(A)}_{\text{草稿猜 }A} \times \underbrace{\frac{q(A)}{p(A)}}_{\text{接受概率}} \;+\; \underbrace{p(B) \times 1 \times 0}_{\text{草稿猜 }B\text{，被接受，但那不是 }A} \;=\; q(A)
$$

$$
P[\text{采到 } B] \;=\; \underbrace{p(B) \times 1}_{\substack{\text{草稿猜 }B\text{，因为 }q(B) > p(B)\\ \text{所以无条件接受}}} \;+\; \underbrace{p(A) \times \left(1 - \frac{q(A)}{p(A)}\right) \times 1}_{\substack{\text{草稿猜 }A\text{ 但被拒，}\\ \text{残差分布必定给出 }B}} \;=\; q(B)
$$

第二个式子展开就是 $p(B) + p(A) - q(A) = 1 - q(A) = q(B)$。

**两个概率都精确等于目标模型的概率。** 所以投机解码不是近似 ——
**它就是从目标模型里采样，一点偏差都没有。**

**这正是「无损」应有的样子** —— 变的只有速度。

那 $K$（一次猜几个 token）该取多少？这里有一个明确的甜点：

<Figure src="/courses/cs336/l10-speculative-sampling-stats.png" wide caption="左：生成 128 个 token 的平均耗时随草稿 token 数 K 的变化 —— XSum（橙）在 K = 3 附近见底后回升，HumanEval（蓝）下降得更久但很快趋平。中：接受率随 K 单调下降 —— HumanEval 从 K=1 时的 0.95 降到 K=7 时的 0.68，XSum 降得更快，到 0.45。右：一轮循环的总耗时随 K 线性上升，从 14.1 毫秒涨到 28.4 毫秒。来源：Chen et al. 2023，转引自 CS336 第 10 讲课件" />

两头都不好：

- **$K$ 太小**：目标模型那一次前向只审阅很少的 token，**没有充分利用它的并行能力**。
- **$K$ 太大**：**被拒绝的概率越来越高**（中间那张图），猜得越远越不可能对，
  而且每一轮循环本身的耗时是线性上涨的（右边那张图）。

**甜点大约在 $K = 3$ 到 4。**

### 8.6 草稿模型怎么来，以及后续工作

实践中的典型配比：

- 目标模型 700 亿参数，草稿模型 80 亿参数；
- 目标模型 80 亿参数，草稿模型 10 亿参数。

**理想情况是草稿模型尽可能接近目标模型** —— 接受率越高，加速越大。
这意味着你应该**蒸馏**出草稿模型。

<Callout type="insight" title="第五到七节的所有招数在这里全部复用">
Percy 指出一个很漂亮的串联：前面几节讲的所有压缩手段（GQA、MLA、量化、剪枝蒸馏）
在这里都能直接用上，而且**多了一条退路**：

**先用各种手段压出一个更小的模型。**

- 如果压出来的模型**你满意** → 直接拿它上线服务。
- 如果**你不满意**（准确率掉太多）→ **那就把它当草稿模型**，
  用原来那个大模型去审阅、修正它。

于是**一个「压坏了」的模型也不算白压。**
</Callout>

**改进草稿模型的后续工作**：

<Figure src="/courses/cs336/l10-medusa-eagle.png" wide caption="几种草稿机制的对比。左上：标准投机采样，用一个小 LLM 逐个生成草稿。左下：Medusa，在目标模型的特征上挂多个头，一次并行产出多个草稿 token。右下：EAGLE，草稿模型直接吃目标模型的高层特征 f，再自回归地产出下一个特征和 token。来源：EAGLE 论文，转引自 CS336 第 10 讲课件" />

- **Medusa**：让草稿模型**并行地生成多个 token**（挂多个预测头），而不是逐个自回归。
- **EAGLE**：让草稿模型**直接使用目标模型的高层特征**，这样它对目标模型的预测更准。

### 8.7 本节小结

<Theorem type="proposition" name="投机采样">
1. **输出是从目标模型精确采样的**（数学保证，不是近似）。
2. **它利用的是「检查比生成快」这个不对称性。**
3. **草稿模型这一侧还有大量创新空间**（这部分涉及训练）。
</Theorem>

## 九、处理动态负载

前面所有的分析都假设你有一个整整齐齐的批次。但真实场景是：**你在跑一个线上服务，
用户随时进来跟你的模型聊天。** 这时候批处理变得很麻烦，原因有三个：

<Theorem type="definition" name="线上流量为什么难批处理">
1. **请求在不同时刻到达。** 你要是等凑够一批再开跑，**先到的请求就白等了**。
2. **序列之间有共享前缀。** 比如同一个系统提示词（system prompt），
   或者对同一个 prompt 采样多个回答。
3. **序列长度各不相同。** 用 padding 补齐是很浪费的。

这和训练时那种「一整块 批大小 × 序列长度 的稠密 token 方阵」相差极远。
</Theorem>

### 9.1 连续批处理（continuous batching）

这个想法来自 **Orca** 系统（OSDI 2022 的论文），出现得相当早。

<Figure src="/courses/cs336/l10-static-batching.png" wide caption="静态批处理的浪费。黄色是 prompt token，蓝色是生成的 token，红色 END 是序列结束。四条序列在不同时刻结束（T4、T5、T6、T8），但整个批次必须等最慢的那条走完，先结束的序列之后的格子全是空转。来源：Anyscale 博客，转引自 CS336 第 10 讲课件" />

**问题**：训练时你拿到的是一整块稠密的 token 方阵；推理时请求**在不同时刻到达、在不同时刻结束**，
你拿到的是一个**参差不齐（ragged）的数组**。

**解法：迭代级调度（iteration-level scheduling）。**

- **一步一步地解码**：每一步给批次里所有序列各生成一个 token。
- **序列结束就把它踢出批次**，不用等别人。
- **新请求到达就直接塞进批次**，不用等当前这批生成完。

之所以叫「连续」批处理，就是因为**批次的成员是动态更新的** ——
旧的完成后被驱逐，新的随时加入，中间不停机。

**但还有一个问题**：批处理要求所有序列有相同的形状（张量的每一片维度要一致），
可这里每条请求的长度都不一样，怎么办？

**解法：选择性批处理（selective batching）。**

假设批次里有三条序列，长度分别是 3、9、5（形状是 $[3, H]$、$[9, H]$、$[5, H]$）：

- **注意力部分**：**每条序列单独算**。因为注意力的计算本身依赖于序列长度 ——
  一个是 $3 \times 3$ 的计算，一个是 $9 \times 9$ 的计算，没法有效地拼成一个张量。
- **非注意力部分（MLP 等）**：**把所有序列首尾相接**，拼成一条长度 $3 + 9 + 5 = 17$ 的「超级序列」，
  形状 $[17, H]$，一次算完。

这个拆分之所以划算，正是因为**MLP 部分占了大量的浮点运算**，
而它恰好是**逐 token 独立**的（第 3.3 节说过：MLP 里批维度和序列维度上的一切彼此独立），
所以怎么拼都不影响结果。

### 9.2 分页注意力（PagedAttention）

最后一个想法来自引入 **vLLM** 的那篇论文。vLLM 现在有很多别的功能了，但这是它当时的核心。

问题是：**KV cache 到底怎么在显存里存放？**

**过去的做法**：请求进来 → 为「prompt + 回复」**一次性分配一段连续的显存**，
大小按**最大长度**算（比如 1024 个 token 的位置）。

<Figure src="/courses/cs336/l10-paged-attention-fragmentation.png" wide caption="连续分配导致的两种碎片。请求 A 的 prompt 只占了 7 个槽，加上刚生成的 1 个和预留的 2 个，剩下 2038 个槽永远不会被用到（内部碎片）。请求 A 和 B 之间还夹着一段太小以至于放不下任何东西的空隙（外部碎片）。请求 B 那边也浪费了 507 个槽。来源：Kwon et al. 2023（vLLM 论文），转引自 CS336 第 10 讲课件" />

Percy 说这就是**碎片化（fragmentation）** —— 当年你得给硬盘做磁盘碎片整理，就是这个问题。
它有两种形态：

- **内部碎片（internal fragmentation）**：你必须预留到最大长度，因为**你不知道模型什么时候会停**。
  但实际上可能只生成了几十个 token，**剩下的两千多个槽位就白占着，谁也用不了**。
- **外部碎片（external fragmentation）**：两段已分配区域之间会有空隙，而那个空隙**太小了，
  放不下任何一个新请求**，于是纯粹浪费。

**解法：借操作系统的分页机制。**

Percy 说，做这个的是一群系统出身的人，**他们知道这个问题几十年前就被解决过一次了**，
所以直接把同一套办法搬过来：

> **把一条序列的 KV cache 切成若干个不必连续的「块（block）」。**

<Figure src="/courses/cs336/l10-paged-attention-blocks.png" caption="块大小为 4 的分块方式：Four / score / and / seven 是 Block 0，years / ago / our / fathers 是 Block 1，brought / forth 是还没填满的 Block 2。查询向量 forth 要和这三个块里的所有键值做注意力，但这些块在显存里的物理位置完全不必相邻。来源：Kwon et al. 2023，转引自 CS336 第 10 讲课件" />

块放在哪里无所谓，只要**块内是对齐的**、并且**有一张索引表记录每个逻辑块在物理上的位置**就行。

<Figure src="/courses/cs336/l10-paged-attention-logical.png" wide caption="两个请求的逻辑块到物理块的映射。请求 A 的逻辑块 0/1/2 分别落在物理块 7/1/3，请求 B 的逻辑块 0/1 落在物理块 5/2 —— 两者的物理块在显存里是交错的，靠索引表关联。来源：Kwon et al. 2023，转引自 CS336 第 10 讲课件" />

### 9.3 分块之后的额外好处：共享前缀

一旦 KV cache 是按块管理的，**两个请求就可以共享同一个物理块**。这带来两类非常实际的收益：

<Figure src="/courses/cs336/l10-paged-attention-sharing.png" wide caption="共享前缀的典型场景：两条序列共用同一段少样本示例的 prompt（黄色），只有任务输入（绿色）和输出（蓝色）不同。这一整段共享前缀的 KV cache 只需要算一次、存一份。来源：Kwon et al. 2023，转引自 CS336 第 10 讲课件" />

1. **共享系统提示词。** 如果很多用户走的是同一个系统提示词，
   **它的 KV cache 只需要算一次、存一份**，所有请求共用。
   考虑到系统提示词往往有上千个 token，这省下的是实打实的算力和显存。
2. **同一个 prompt 采样多个回答。** 这在程序合成、RL rollout 等场景里非常常见 ——
   **prompt 部分的 KV cache 完全共享，只有各自生成的回复部分是独立的。**

那两个共享块的序列后来分叉了怎么办？答案还是操作系统那一套：**写时复制（copy-on-write）**。

<Figure src="/courses/cs336/l10-paged-attention-parallel.png" wide caption="写时复制。两个采样共享物理块 1（Four score and seven / years ago our ...）。当采样 A1 在第四个位置写入 fathers、采样 A2 写入 mothers 时，引用计数从 2 降到 1，块被复制一份到物理块 3，两边各写各的。来源：Kwon et al. 2023，转引自 CS336 第 10 讲课件" />

具体过程：对 prompt `Four score and seven years ago our ___` 采样两个回答。

- 两个采样先共享同一批物理块，块上记着**引用计数 2**。
- 如果两边**碰巧采到同一个 token**，那就继续共用，什么都不用做。
- 如果两边**采到不同的 token**（一个 `fathers`、一个 `mothers`），
  就把这个块**复制一份**，引用计数从 2 降到 1，两边各写各的，然后继续往下走。

也就是说：**能共享的前缀尽量共享，只在真正分叉的那个块上付复制的代价。**

### 9.4 vLLM 的其他优化

这些课上没时间展开，但列一下：

- **把「读块」和「注意力计算」融合进一个 kernel**，减少 kernel 启动开销。
- **用最新的 kernel**：FlashAttention、FlashDecoding。
- **用 CUDA graph** 来避免 kernel 启动开销。

<Callout type="insight" title="这一节的共同主题">
连续批处理借的是**操作系统的调度**，分页注意力借的是**操作系统的虚拟内存分页**，
投机采样借的是**处理器的投机执行（speculative execution）**。

推理服务这一层的大量创新，本质上是**把系统领域几十年前就解决过的问题，
重新搬到语言模型的场景里**。
</Callout>

## 十、总结

<Theorem type="proposition" name="这一讲的五条">

1. **推理非常重要**：实际使用、模型评测、强化学习都要用它。
   而且训练是一次性成本，**推理是每天都要付的成本**。
2. **推理和训练是特征完全不同的负载**：训练可以沿序列并行，推理必须顺序生成，
   所以**推理是内存受限的**；线上服务还额外是**动态的**。
3. **压缩技术**：新的注意力架构（GQA / MLA / CLA / 局部注意力 / 线性注意力）、
   量化、剪枝与蒸馏、投机采样。**它们全都在遵循同一条原则 ——
   把 KV cache（或参数）做小，同时别把准确率伤太狠。**
4. **系统领域的老想法在这里大放异彩**：投机执行、分页、写时复制、迭代级调度。
5. **新架构还有巨大的改进空间。**

</Theorem>

最后这一条是 Percy 收尾时特别强调的：

> 说到底，**KV cache 的存在方式、以及注意力被构造出来的方式，
> 从根本上就让 Transformer 成为一个「对推理不友好」的架构。**
> 如果有人能设计出一个**从一开始就为推理而生**的架构 —— 状态空间模型、线性注意力、
> 扩散模型都是候选 —— **那可能会解锁很多东西。**
